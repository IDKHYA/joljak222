# 두 "계절" 개념 혼용 실태 보고서

작성 기준일 2026-07-02. 시스템에는 서로 완전히 독립인 두 개의 "계절" 축이 존재한다. 이 보고서는 두 축이 코드·UI·데이터·논문의 각 층에서 어떻게 혼용되고 있는지를 파일:라인 단위로 추적하고, 실제 동작에 미치는 영향과 v2 처방을 정리한다. `리버스frd.md`와 `의류인식_리버스해부.md`의 자매 문서다.

## 1. 요지

| | 축 A — 퍼스널컬러 시즌 | 축 B — 착용 계절 |
| --- | --- | --- |
| 질문 | 이 색이 **누구에게** 어울리는가 | 이 옷을 **언제(날씨)** 입는가 |
| 값 | 12계절 SeasonId (라이트 스프링 … 브라이트 윈터) | SeasonTag '봄/가을'·'여름'·'겨울'·'사계절' |
| 판정 대상 | 사람(얼굴) + 의류 HEX | 의류(형태·범주) |
| 판정 방법 | 사진·설문 융합 진단, 의류는 HEX↔시즌 팔레트 CIEDE2000 거리 | 이름 규칙 / 서버 규칙 / RF / 사용자 입력 |
| 추천에서의 역할 | personal 축 38% + 조화 축의 대비 선호 보정 | weather 축 22% + 하드컷 필터 |

**결론 요약.** 점수 계산 코드에서 두 축의 값이 직접 섞이는 지점은 없다(§4에서 검증 방법 서술). 그러나 ① 어휘가 전 층에서 충돌하고, ② UI에는 혼용이 낳은 실동작 버그가 있으며(사계절 옷이 계절 필터에서 소실), ③ 데이터 생성 스크립트에는 작성 시점에 두 개념을 실제로 혼동한 물증이 있고, ④ 논문에는 두 용어를 구분하는 정의가 없다. 그대로 두면 v2와 논문 심사에서 반드시 문제가 된다.

## 2. 두 축의 정확한 정의와 데이터 흐름

### 2.1 축 A — 퍼스널컬러 시즌 (12계절)

- 타입 `SeasonId`(src/types.ts), 라벨 `SEASON_LABELS`(wardrobeConstants.ts:97), 팔레트·특질 `SEASON_PROFILES`(personalColorWorkbook.ts), 설명·별칭 `SEASON_DETAILS`(seasonContent.ts).
- 생성 — `personalColorEngine.ts`가 사진+설문 융합으로 `FinalResult.seasonTop1Id/Top2Id`를 산출. **사람에 대한 판정**이다.
- 의류와의 접점("HEX로 퍼컬 범위 정하는 기능") — `scoreItemForPersonalColor`(recommendationEngine.ts:101)가 의류의 `representativeHex`/`dominantColors`(최대 3색 비율 가중)를 사용자 시즌 팔레트와 CIEDE2000 최근접 거리로 비교, `scorePaletteDistance`(:62) 구간 함수로 0~100 점수화하고 회피색 근접 시 감점(:81). 신뢰도가 낮으면 Top2 시즌 팔레트를 최대 35% 혼합(:89-96). 이 결과가 4축 중 personal(기본 38%)이 된다.
- 조화 축에도 개입 — `calculatePairHarmonyScore`(:562)가 `SEASON_PROFILES[seasonTop1Id].traits.contrast`로 보색/유사색 가감점을 준다. 주석(:511-512)의 표현이 "고대비 선호(**겨울 계열**), 저대비 선호(**여름 계열**)"이다.

### 2.2 축 B — 착용 계절 (SeasonTag)

- 타입 `SeasonTag`(wardrobeTypes.ts:41). **주석(:40)이 시스템 전체에서 유일하게 두 축의 구분을 명시한 지점이다** — "퍼스널컬러 시즌(12계절)과 다른 축이며, 날씨 기반 추천 필터에 사용됩니다."
- 생성 경로가 5개다. ① 이름 규칙 `getSeasonTag`(clothingMeta.ts:175, 패딩·코트·니트→겨울 등), ② 서버 규칙 계층(season_predictor.py `get_rule_season`), ③ RF(`season_model.joblib`, 3분류), ④ 카탈로그 human_label 458장 + 규칙 폴백 402장(generate_catalog.py:67), ⑤ 사용자 폼 입력.
- 소비 — `isWeatherEligible`(recommendationEngine.ts:46)의 **하드컷**(여름옷은 8도 이하 밴드에서 후보 제외, 겨울옷은 20도 이상에서 제외), `getWeatherScore`(:160)의 가감점(사계절 +8, 봄/가을 간절기 +6, 겨울옷@28도 이상 -30, 여름옷@8도 이하 -25), UI 필터·뱃지.

## 3. 혼용 실태 — 층별 증거

### 3.1 코드 계층 — 이름 충돌 (기능 오염은 없음)

- `wardrobeConstants.ts` 한 파일에 `SEASON_LABELS`(축 A, :97)와 `SEASON_TAGS`(축 B, :34)가 접두어 구분 없이 병존한다.
- `recommendationEngine.ts` 한 파일 안에서 같은 단어 '겨울'·'여름'이 두 의미로 쓰인다. :49-50과 :178-179의 `'겨울'`/`'여름'`은 seasonTag 값(축 B)이고, :511-512 주석의 "겨울 계열"/"여름 계열"은 퍼컬 윈터/서머(축 A)다. 식별자만 보면 `SEASON_CONTRAST`(축 A)와 `item.seasonTag === '겨울'`(축 B)이 같은 화면 폭 안에 있다.
- 서버는 축 B만 다루지만 파일·모델명이 `season_predictor.py`, `season_model.joblib`으로, 어느 계절인지 이름에 없다.

### 3.2 UI 계층 — 실동작 버그 포함

**[버그 D — 사계절 소실]** 카탈로그 선택 화면(WardrobeSection.tsx:348)과 데일리룩 피커(TryOn.tsx:477)의 계절 필터 버튼이 `['전체', '봄', '여름', '가을', '겨울']`이고, 매칭이 부분 문자열이다(WardrobeSection.tsx:322, TryOn.tsx:76).

```ts
.filter((i) => season === '전체' || i.seasonTag.includes(season))
```

- '봄'·'가을' 버튼은 `'봄/가을'.includes(…)`가 참이라 **우연히** 동작한다.
- `'사계절'`은 '봄'·'여름'·'가을'·'겨울' 어느 것도 부분 문자열로 포함하지 않는다. 따라서 **사계절 옷은 전체를 제외한 모든 계절 필터에서 사라진다.** 의미론은 정반대여야 한다 — 사계절은 어떤 계절 필터에서도 보여야 한다. 수동 등록 기본값과 예측 실패 폴백이 모두 '사계절'이므로(useManualClothing.ts:20, :52), 사용자 등록 옷의 상당수가 이 버그의 대상이 된다.
- 근본 원인이 바로 혼용이다 — UI는 사용자에게 익숙한 4계절(날씨) 멘탈 모델을 제시했는데, 데이터 taxonomy는 3+1('봄/가을' 병합 + '사계절')이라 서로 맞지 않고, 그 틈을 부분 문자열 매칭으로 봉합했다.

**어휘 충돌 동선.** 진단 결과 화면은 "4계절 대분류 + 12계절 세부 진단"(PersonalResult.tsx:23)이라는 축 A의 4계절을 말하고, 별칭은 "여름 쿨"·"겨울 브라이트"(seasonContent.ts)다. 옷장 화면은 의류 카드에 축 B의 "여름"·"겨울" 칩을 단다(WardrobeSection.tsx:281, AnchorOutfitFinder.tsx:79). "여름 쿨" 진단을 받은 사용자가 "여름" 태그 옷을 자기 톤에 맞는 옷으로 오독하는 동선이 자연스럽게 생긴다. 수동 등록 폼 라벨도 그냥 "계절 태그"(WardrobeSection.tsx:542)라 어느 계절인지 알 수 없다.

### 3.3 데이터 생성 계층 — 혼동의 물증

- `generate_catalog.py:113`이 산출 파일 헤더로 쓰는 주석이 **"퍼스널컬러 ML 모델 학습에 사용된 516개 의류 이미지"**다. 이 모델(RF)은 착용 계절 분류기이지 퍼스널컬러 모델이 아니다. 카탈로그를 만들던 시점에 작성자가 두 개념을 실제로 혼동했다는 직접 물증이다. 현행 `trainingCatalog.ts`는 JSON 리팩토링으로 이 주석이 사라졌지만, 스크립트를 재실행하면 부활한다.
- human_label의 라벨 어휘가 '봄/가을·여름·겨울'인데, 패션 문맥에서 "가을 색"은 통상 색채 미학(축 A의 어텀 팔레트 — 버건디·머스터드·브라운)을 뜻한다. 라벨링 안내 문서가 없는 상태에서 두 해석이 섞였을 위험이 있다. 검증은 불가능하지만 위 물증과 결합하면 개연성이 있고, RF 89.74%의 라벨 품질에 대한 잠재 위협이다.
- 규칙·RF의 색 밝기 휴리스틱(밝으면 여름옷 +1, 어두우면 겨울옷 +1 — season_predictor.py:270)은 **색이라는 축 A의 신호로 축 B를 추정**하는 지점이다. 색은 보온성과 독립이므로(흰 패딩은 겨울옷, 검은 린넨 셔츠는 여름옷) 개념 경계가 실제로 흐려진 설계다.

### 3.4 논문 계층

- 초록(논문.md:41) 한 문장 안에 "12계절 퍼스널컬러를 자동 진단하고, … segmentation 기반 계절 분류 결과와 결합하여"가 병존한다. 제목 후보 3(:13)도 동일 구조다.
- '용어', '착용 계절' 등 두 개념을 구분·정의하는 문구가 논문 전체에서 검색되지 않는다. "의류 계절 분류"라는 수식어가 사실상 유일한 구분 장치다.
- 심사 질의 위험 — "계절 분류 89.74%가 12계절 분류인가?"라는 질문이 나오기 딱 좋은 구성이다. 12계절 진단(기여 1)과 3계절 분류(기여 2)의 숫자가 나란히 있어 3/12 혼동, 정확도 비교 오독 여지가 있다.

## 4. 실동작 영향 평가

**점수 파이프라인 교차 오염 — 없음 (검증).** `scoreItemForPersonalColor`의 입력은 HEX·팔레트뿐이고 seasonTag를 읽지 않는다. `getWeatherScore`·`isWeatherEligible`의 입력은 seasonTag·type·상태뿐이고 seasonTop1Id를 읽지 않는다. src 전체에서 `seasonTag`와 `seasonTop1`/`SEASON_LABELS`의 사용처를 교차 검색한 결과 두 값이 만나는 코드는 없다.

**그러나 간접 피해 경로는 실재한다.**

1. 잘못된 축 B 값의 파괴력 — 축 B는 점수(22%)뿐 아니라 **하드컷**에 쓰인다. `의류인식_리버스해부.md` §3.3의 계통 편향(신발→항상 '여름' 고신뢰)과 버그 A(사용자 수정 무시)가 결합하면, 겨울 부츠가 '여름' 태그로 저장되고 `isWeatherEligible`(:49)에 의해 **추운 날 추천 후보에서 완전히 제외**된다. 감점이 아니라 소거라 사용자가 원인을 추적할 수 없다.
2. 버그 D — 사계절(등록 기본값) 옷이 카탈로그 선택·데일리룩 피커의 계절 필터에서 소실.
3. UX 오독 — "여름 쿨" 진단자에게 "여름" 칩이 톤 적합 신호로 읽히는 동선(§3.2).
4. 논문 방어력 — 용어 정의 부재로 기여 1·2의 구분이 심사에서 흔들릴 수 있음(§3.4).

## 5. v2 처방

1. **타입·식별자 분리.** 축 A는 `PersonalSeason`(12계절), 축 B는 `WearSeason` 또는 보온 등급 `WarmthLevel`로 개명하고, 코드에서 수식어 없는 `season`/`SEASON_*` 식별자를 금지한다. UI 문구도 "퍼컬 시즌" vs "착용 계절"로 고정한다("계절 태그" → "착용 계절").
2. **사계절 의미론 수정 + 매칭 방식 교체.** 필터는 enum 비교로 바꾸고, 사계절은 모든 계절 필터에 포함시킨다. 부분 문자열 매칭(`includes`)은 폐지한다.
3. **축 B를 명칭이 아닌 파생값으로.** 리버스frd.md 결정 ③(속성 유도)과 정합 — 소매 길이·소재·두께감 관측 → 보온 등급(서열형) → 날씨 밴드 직결. '봄/가을' 같은 계절 명칭이 데이터에서 사라지면 어휘 충돌 자체가 구조적으로 소멸하고, 8구간 날씨 밴드와의 해상도 불일치(3+1 taxonomy로는 간절기 구분 불가)도 해소된다. 기존 3계절 라벨은 논문 비교 실험용으로만 보존한다.
4. **논문에 용어 정의 절 추가.** 2장 초입에 "본 논문에서 퍼스널컬러 시즌은 인물의 색채 유형(12계절)을, 착용 계절은 의류의 기후 적합 구간(3분류)을 가리킨다"를 명시하고, 기여 2의 명칭을 "의류 착용 계절 분류"로 전 장에서 통일한다.
5. **색→착용계절 추론의 한계 명시.** 색 밝기 규칙의 반례(흰 패딩, 검은 린넨)를 논문 오류 분석에 포함해 정직하게 다루고, v2에서는 색을 착용 계절 신호에서 제외한다.
6. **라벨링 가이드 문서화.** P2 라벨 402장 완성 시 "착용 계절 = 이 옷을 주로 입는 기온 구간" 정의와 반례 예시를 명문화해 축 A 해석 오염을 차단하고, 기존 458장도 같은 기준으로 표본 재검한다.
