# v2 도메인 개념 고도화 — 계산 모델 설계서

작성 기준일 2026-07-03.

선행 문서는 `v2_md/도메인개념_색상계약_v2.md`다. 그 문서가 "개념을 섞지 않는다"는 계약을 고정했다면, 이 문서는 각 개념을 **실제 계산 규칙**까지 파고든다. 목표는 v2 추천 엔진과 어댑터를 구현할 때 이 문서만 보고 수식·표·의사코드를 그대로 옮길 수 있는 상태다.

원칙 하나를 먼저 박는다. 이 문서의 모든 수치는 **이론 기반 사전값(초기값)**이며, 10월 사용자 평가 데이터로 보정하는 대상이다. 초기값이라고 표시된 숫자를 바꾸는 것은 설계 변경이 아니라 튜닝이다.

## 1. 현재 구현과 목표의 격차 (실측)

`v2/src/domain/recommendationEngine.ts` 현재 코드를 기준으로 한 격차표다. 지금 엔진은 골든 패스 성립을 증명한 smoke 수준이고, 계약 문서가 요구하는 계산과는 거리가 있다.

| 축 | 현재 구현 | 계약이 요구하는 것 | 이 문서의 절 |
| --- | --- | --- | --- |
| 퍼컬 적합도 | RGB 유클리드 거리, 대표색 1개, Top1 팔레트만 | CIELAB Delta E 2000, dominant 가중, Top2 혼합, 회피색 감점, 중립 처리 | §2, §3 |
| 색상 조화 | hue 간격 하나로 4단 점수 | hue 관계 50% + 명도 균형 30% + 채도 균형 20%, 시즌 대비 선호, 중립·데님, 패턴 충돌 | §4 |
| 날씨 | ideal 96 / acceptable 78 / 그 외 48 계단 | 보온-밴드 거리 기반 연속 감점, 하드컷 금지, 아우터 규칙 | §5 |
| WarmthLevel | 프리셋에 수기 입력된 값 사용 | 관측 신호 우선순위 파생 + 금지 신호 명시 | §6 |
| N조합 | disjoint greedy, 부족하면 이슈 반환 | 완화 사다리(부분 재사용 허용), 세트 다양성 타이브레이크 | §7 |
| 안정성 | availability 3단 점수 | 유지 (v2에서 변경 없음) | — |

바꾸지 않는 것도 명시한다. 진단 엔진(사진+설문 융합)과 12계절 팔레트 HEX 원본, 4축 가중치 구조(0.38/0.22/0.28/0.12)는 이 문서의 대상이 아니다.

## 2. 퍼컬 스펙트럼 — 점 목록에서 영역으로

### 2.1 왜 영역인가

현재 팔레트는 시즌당 24개 HEX **점 목록**이다. 점 목록 기반 최근접 거리에는 구조적 약점이 있다.

- 두 스와치 사이의 색은 실제로는 시즌에 잘 맞는데도 거리가 멀게 나온다 (샘플링 구멍).
- 스와치 하나가 이상치면 그 근처의 부적합 색이 과대평가된다.
- "이 시즌은 명도 몇 이상, 채도 몇 이하" 같은 시즌의 **본질적 범위**가 데이터에 있는데 쓰이지 않는다.

그래서 퍼컬 스펙트럼을 "점 24개"에서 "**점 24개 + 그 점들이 정의하는 영역**"으로 재가공한다. 점은 정밀도를, 영역은 커버리지를 담당한다.

### 2.2 1단계 — 스와치 감사

영역을 만들기 전에 원본 스와치를 감사한다. 시즌당 1회, 빌드 타임 작업이다.

| 검사 | 규칙 | 처리 |
| --- | --- | --- |
| 중복 | 같은 시즌 안에서 스와치 쌍의 Delta E < 3 | 하나로 병합 |
| 이상치 | 시즌 내 다른 스와치들과의 평균 거리가 시즌 평균 + 2σ 초과 | 제외 후보로 표시, 수동 확인 |
| 형식 | HEX 파싱 실패, 6자리 아님 | 빌드 실패 |

감사 결과는 `v2/catalog/metadata` 옆에 시즌별 감사 리포트로 남기고, 감사를 통과한 스와치 집합만 이후 단계에 쓴다.

### 2.3 2단계 — LAB 변환 캐시

감사 통과 스와치를 전부 CIELAB과 LCh(명도 L*, 채도 C*, 색상각 h°)로 변환해 캐시한다. 런타임에 HEX→LAB 변환을 반복하지 않는다. 의류 쪽도 어댑터가 저장 시점에 대표색·dominant의 LAB 값을 함께 저장한다.

### 2.4 3단계 — 시즌 영역 표현

시즌 영역은 LCh 3축의 범위로 표현한다. 감사 통과 스와치로부터 자동 산출한다.

| 축 | 산출 규칙 (초기값) | 의미 |
| --- | --- | --- |
| L* 범위 | 스와치 L*의 10~90 백분위 ± 여유 6 | 이 시즌이 허용하는 밝기 대역 |
| C* 범위 | 스와치 C*의 10~90 백분위 ± 여유 5 | 허용 채도 대역 |
| h° 섹터 | 스와치 h°를 ±12° 창으로 덮는 각도 구간들의 합집합 | 허용 색상각 (섹터 여러 개 가능) |

무채색 스와치(C* < 12)는 h° 섹터 산출에서 제외한다. 각도가 무의미하기 때문이다.

### 2.5 4단계 — 멤버십 함수

후보 색이 영역에 얼마나 들어오는지를 0~1 멤버십으로 계산한다. 경계는 딱 자르지 않고 사다리꼴로 완만하게 내린다.

```
memL = trapezoid(L*, L범위, 경사폭 8)
memC = trapezoid(C*, C범위, 경사폭 6)
memH = C* < 12 ? 1.0 : maxOverSectors(trapezoid(h°, 섹터, 경사폭 10))
membership = min(memL, memC, memH)
regionScore = 100 × membership
```

`min`을 쓰는 이유는 한 축이라도 크게 벗어나면 시즌 색이 아니기 때문이다. 무채색은 h° 축을 통과 처리하고 §3.4의 중립 규칙이 따로 처리한다.

### 2.6 5단계 — 점수 결합

스와치 점수와 영역 점수는 **상한 결합**한다.

```
paletteScore = max(swatchScore, 0.92 × regionScore)   // 0.92는 초기값
```

- 스와치에 가까우면 스와치 점수가 이긴다 (정밀).
- 스와치 사이 구멍에 있는 색은 영역 점수가 구제한다 (커버리지).
- 영역 점수에 0.92를 곱하는 이유는, 검증된 점보다 추정된 영역을 약간 덜 신뢰한다는 뜻이다.

회피색은 같은 틀의 역방향이다. 시즌별 회피 스와치 목록(팔레트와 동일한 형식)을 두고, 회피 영역 멤버십이 높으면 §3.3의 감점을 적용한다.

## 3. 퍼컬 적합도 함수 v2 — 아이템 단위

계약 §3.3의 절차를 완전한 계산 규칙으로 만든다.

### 3.1 입력 색 가중

아이템의 색은 대표색 1개가 아니라 dominant 최대 3개를 쓴다.

```
색 목록 = dominant가 있으면 dominant(최대 3), 없으면 [대표색]
가중치 = 각 색의 ratio, ratio가 없으면 [0.60, 0.25, 0.15] 기본값
itemColorScore = Σ ( 가중치ᵢ × colorScore(색ᵢ) )
```

### 3.2 색 하나의 점수 — Delta E 매핑

색 하나의 `colorScore`는 §2.6의 `paletteScore`인데, 그중 `swatchScore`는 최근접 Delta E 2000을 계약 §3.3의 해석표에 맞춰 구간 선형으로 매핑한다.

| 최근접 Delta E | swatchScore (구간 내 선형) |
| ---: | --- |
| 0~5 | 100 → 92 |
| 5~12 | 92 → 78 |
| 12~22 | 78 → 60 |
| 22~35 | 60 → 42 |
| 35 초과 | 42에서 바닥 25까지 완만히 하강 |

바닥이 0이 아니라 25인 이유는 하드컷 금지 원칙이다. 팔레트 밖 색도 코디에서 역할이 있을 수 있으므로 감점으로 순위를 내리되 소거하지 않는다.

### 3.3 Top2 혼합과 회피 감점

```
// Top2 혼합 — 진단 신뢰도가 낮을수록 Top2를 섞는다 (최대 35%)
r = confidence >= 0.75 ? 0
  : confidence <= 0.40 ? 0.35
  : 0.35 × (0.75 - confidence) / 0.35        // 선형, 초기값
mixedScore = (1 - r) × scoreTop1 + r × scoreTop2

// 회피 감점 — 회피 스와치 최근접 거리 기준
회피 Delta E < 8   → -15
회피 Delta E 8~15  → -8
그 외              → 0
// 결정 경계 — 팔레트가 회피색보다 가깝거나 같으면 감점하지 않는다
팔레트 최근접 <= 회피 최근접 → 감점 0
최종 = clamp(mixedScore + 회피감점, 15, 100)
```

`top2`가 없거나 unknown이면 r = 0으로 고정한다. top1 복제로 때우지 않는다 (계약 §7.3).

회피색 데이터는 새로 발명하지 않고 v1 `seasonContent.ts`의 `SEASON_DETAILS.worstColors`(시즌당 5개)를 승계한다. v1 결과 화면의 "피해야 하는 색상"과 v1 의류 적합도 감점에 이미 쓰이던 사용자 대면 데이터라, v2 설명 문구와 감점 로직이 같은 데이터를 봐야 하기 때문이다. 다만 실측 결과 승계 회피색 60개 중 11개가 자기 시즌 팔레트와 Delta E 15 이내로 가깝다 (예: 다크 윈터 회피색 페일 핑크 `#F9CCC9` vs 팔레트 아이시 핑크, 최근접 9.7). 이것이 위의 결정 경계 규칙이 필요한 이유다 — 경계가 없으면 팔레트 색이 만점과 감점을 동시에 받는 모순이 생긴다. 경계 덕분에 팔레트 스와치 자신은 절대 감점받지 않음이 테스트로 보장된다.

### 3.4 중립색과 데님

무채색·저채도(C* < 12) 색은 팔레트 거리로 죽이지 않는다. 화이트·블랙·그레이는 어느 시즌에서도 코디 유틸리티가 있기 때문이다.

```
중립 기본 점수 = 78 (초기값)
시즌 명도 정합 보정 = 시즌 lightness 축과 색 L*의 방향이 맞으면 +10, 반대면 -6
// 예: light-summer(명도 0.95)에 밝은 그레이 → +10, 검정 → -6
// 예: true-winter는 팔레트에 순백·순흑이 있으므로 스와치 점수가 이미 높아 이 규칙과 max 결합
neutralScore = clamp(78 + 보정, 60, 92)
colorScore = max(paletteScore, neutralScore)   // 중립일 때만
```

데님은 소재 속성이지 색이 아니므로 퍼컬 축에서는 색으로만 판정하되, `isDenim=true`인 하의에는 코디 유틸리티 가산 +4를 준다 (초기값, 계약 §3.3의 "소폭 가산").

## 4. 색상 조화 v2 — 코디 단위

계약 §5의 성분 비중(hue 50% / 명도 30% / 채도 20%)과 타입 기본 점수를 그대로 쓰고, 그 위에 세 가지를 추가한다.

### 4.1 쌍(pair) 점수

두 아이템의 대표색으로 계산한다.

```
hue 성분   = 조화 타입 기본 점수 (계약 §5.1 표: mono 80, analogous 82, tension 55, triadic 76, complementary 88)
명도 성분  = 100 - |L*₁ - L*₂ 이상 편차|   // 목표 편차 25±20을 만점 대역으로, 벗어날수록 감점
             // 상하의가 너무 같은 명도면 분리감이 없고, 너무 다르면 단절된다
채도 성분  = 100 - max(0, (C*₁ + C*₂) - 120) × 0.5   // 고채도 두 개 충돌 방지, 초기값
pairScore = hue×0.5 + 명도×0.3 + 채도×0.2
```

한쪽이 중립(무채색)이면 hue 성분을 계산하지 않고 `neutral` 타입 85점을 쓴다. 양쪽 다 중립이면 hue 성분 88점(무채색 코디는 안전한 조합)에 명도 성분만 검사한다.

### 4.2 시즌 대비 선호 보정

같은 배색이라도 시즌에 따라 어울림이 다르다. 계약 §3.1의 contrast 축 값(-0.38 ~ +1.00)을 그대로 쓴다.

```
complementary/tension 쌍  → hue 성분 += 6 × contrast    // 고대비 시즌은 보색을 잘 받음
monochromatic/analogous 쌍 → hue 성분 += -4 × contrast   // 저대비 시즌은 유사색이 안전
```

bright-winter(contrast 1.00)는 보색 +6, soft-summer(contrast -0.38)는 보색 -2.3에 유사색 +1.5가 된다. 표 하나로 12계절 전부 자동 커버되고 시즌별 하드코딩 분기가 사라진다.

### 4.3 패턴 충돌

강한 패턴(stripe, plaid, graphic)이 한 코디에 2개 이상이면 조화 점수에서 -8 (초기값). `unknown` 패턴은 solid로 취급하되 추천 근거에 "패턴 미확인"을 남기지 않는다 — 사용자에게는 조용히, `confidenceNotes`에만 남긴다.

### 4.4 코디(3개 이상) 집계

모든 쌍의 평균은 시각적 중요도를 무시한다. 상의-하의 쌍이 코디 인상의 중심이므로 가중 집계한다.

```
harmonyScore = 상의-하의 pairScore × 0.6 + 나머지 쌍 평균 × 0.4   // 초기값
```

신발이 없으면 상의-하의 쌍 점수 그대로다. 아우터가 있으면 아우터-상의 쌍을 "나머지 쌍"에 포함한다.

## 5. 날씨 축 v2 — 연속 감점 모델

### 5.1 계단을 거리로

현재의 3단 계단(96/78/48)을 보온-밴드 순서 거리 기반 연속 점수로 바꾼다. WarmthLevel과 WeatherBand는 둘 다 순서형이므로 수직선 위에 놓을 수 있다.

```
warmthPos: very-light=0, light=1, mid=2, warm=3, heavy=4
bandTarget: very-hot=0, hot=0.5, warm=0.75, mild=1.75, cool=2.4, chilly=2.75, cold=3.5, freezing=4

weatherScore = clamp(100 - 22 × |warmthPos - bandTarget|, 30, 100)
```

- 밴드 목표값이 0.5 간격이 아닌 이유는 계약 §4.2의 ideal/보조 허용 표가 비대칭이기 때문이다. 예를 들어 mild의 ideal은 mid·light이고 acceptable은 warm인데, 목표를 2.0에 두면 light와 warm이 같은 점수를 받아 순서가 깨진다. 1.75로 옮기면 mid 94.5 > light 83.5 > warm 72.5로 계약 순서가 성립한다. warm·cool·chilly도 같은 이유로 조정했다 (초기값).
- 정확히 맞으면 100, 거리에 비례해 감점, 바닥 30.
- 바닥이 0이 아닌 이유는 하드컷 금지다. 오분류 한 번으로 아이템이 영구 실종되던 v1 문제를 감점 방식이 구조적으로 막는다.
- 계약 §4.2의 ideal/보조 허용 표와 결과가 정합한지 테스트로 고정한다 (§8).

### 5.2 아우터 규칙

아우터는 "있으면 점수 계산"이 아니라 밴드에 따라 요구/선택/비권장이 갈린다.

| 밴드 | 아우터 | 규칙 |
| --- | --- | --- |
| freezing, cold | 필수 | 아우터 없는 코디는 날씨 축 -25 |
| chilly, cool | 권장 | 아우터 있으면 +6, 없어도 감점 없음 |
| mild | 선택 | 가감 없음 |
| warm, hot, very-hot | 비권장 | 아우터 있으면 -15 (heavy 아우터는 -25) |

아우터가 포함된 코디의 상의 보온은 한 단계 올려 평가한다 (레이어링 효과, 초기값 +1).

```
효과적 상의 warmthPos = min(4, 상의 warmthPos + 1)   // 아우터 동반 시
```

### 5.3 프리셋에 대한 요구

이 규칙이 들어오면 **추천 성립 보장 검증도 밴드별로 아우터를 포함해야 한다.** freezing~cool 밴드 시연에서 아우터 없는 프리셋은 전 코디가 -25를 받아 설득력이 죽는다. 프리셋 큐레이션 합격 기준에 "8개 밴드 전부에서 겹치지 않는 N조합 성립"을 명시하고, 검증 스크립트가 자동 확인한다.

## 6. WarmthLevel 파생 고도화

계약 §4.3의 파생 순서를 신호 표로 구체화한다. 핵심은 우선순위와 **금지 신호**다.

### 6.1 신호 우선순위

| 순위 | 신호 | 예 | 산출 |
| --- | --- | --- | --- |
| 1 | 사용자 확정값 | 편집 화면에서 직접 선택 | 그대로 채택, 이후 신호 무시 |
| 2 | 종류(typeLabel) 기본표 | 반팔티→light, 니트→warm, 패딩→heavy | 1차 후보 |
| 3 | 소재 보정 | 린넨 -1단계, 울/플리스 +1단계, 초기값 | 후보 ±1 |
| 4 | 구조 관측 | 민소매 -1, 기모·퀼팅 +1 | 후보 ±1, 관측이 있을 때만 |
| 5 | v1 SeasonTag 폴백 | 여름→light, 봄/가을→mid, 겨울→warm, 사계절→mid | 2~4가 전부 없을 때만, `confidenceNotes`에 경고 |

보정 결과는 5단계 범위로 clamp한다. 2~4단계에서 신호가 서로 충돌하면(예: 종류는 반팔인데 소재가 울) 종류를 우선하고 충돌 사실을 `confidenceNotes`에 남긴다.

### 6.2 금지 신호

다음은 어떤 경우에도 WarmthLevel 산출에 쓰지 않는다. v1이 흰 패딩·검은 린넨에서 깨진 원인이다.

- 색 밝기·색상 (흰 옷 ≠ 여름 옷).
- 노출도·피부 픽셀 비율 (촬영 조건이지 옷 속성이 아님).
- 촬영 배경·계절감 있는 소품.

테스트로 고정한다. "L* 95의 패딩은 heavy, L* 20의 린넨 셔츠는 light"가 통과해야 한다.

## 7. 겹치지 않는 N조합 — 완화 사다리와 다양성

### 7.1 현재 규칙의 문제

현재는 완전 disjoint greedy라서, 신발이 2켤레뿐인 옷장에서는 3번째 코디가 성립하지 않고 이슈만 반환한다. 실사용 옷장(신발이 적은 게 정상)에서 자주 깨진다.

### 7.2 완화 사다리

부족하면 조건을 한 단계씩 풀되, 무엇을 재사용했는지 `reusedItemIds`에 기록하고 UI가 "신발은 앞 코디와 같아요"라고 말할 수 있게 한다.

```
0단계: 완전 disjoint (현재 규칙)
1단계: 신발 재사용 허용
2단계: 신발 + 아우터 재사용 허용
3단계: 하의 재사용 허용
그래도 부족 → issues에 카테고리별 부족 수량을 명시해 반환
```

상의는 끝까지 재사용하지 않는다. 상의가 같으면 사용자 눈에 "같은 코디"로 보이기 때문이다.

### 7.3 세트 다양성 타이브레이크

점수 상위만 뽑으면 비슷한 색 이야기의 코디 3개가 나올 수 있다 (전부 네이비 계열 등). 후보 점수 차이가 3점 이내면(초기값) 이미 선택된 코디들과 **hue 버킷 구성이 다른 후보**를 우선한다.

```
후보 A, B의 점수 차 <= 3 이고
B의 hue 버킷 집합이 기선택 코디들과 겹침이 더 적다면 → B 선택
```

점수 순위를 크게 훼손하지 않으면서 "여러 개 보여주는 의미"를 지키는 규칙이다.

## 9. 구현 순서 제안

엔진 교체는 한 번에 하지 않는다. 각 단계가 독립적으로 테스트 가능하다.

1. **색 수학 기반** — HEX→LAB/LCh 변환, Delta E 2000, 스와치 캐시. (G1, G2)
2. **퍼컬 축 교체** — dominant 가중, Top2 혼합, 회피, 중립. (G3~G5)
3. **영역 모델** — 스와치 감사 + 영역 산출 + 상한 결합. (G6)
4. **조화 축 교체** — 성분 분해, 대비 선호, 패턴, 집계. (G7, G8)
5. **날씨 축 교체** — 연속 감점 + 아우터. (G9~G11)
6. **조합기 고도화** — 완화 사다리 + 다양성. (G12)
7. **프리셋 재검증** — 새 엔진으로 성립 보장 재확인, 필요 시 프리셋 보강(특히 아우터). (G13)

1~2만 끝나도 smoke 엔진보다 계약 정합성이 크게 오른다. 3~6은 순서를 바꿔도 된다. 7은 마지막이 아니라 5 이후 아무 때나 당겨도 된다.

## 10. 열려 있는 질문

- 영역 산출의 백분위·여유 폭(§2.4)은 12시즌 전부에서 시각적으로 점검해야 한다. `도메인개념_시각화.html`에 영역 오버레이를 추가하는 것이 자연스러운 다음 단계다.
- 명도 성분의 "목표 편차 25±20"(§4.1)은 톤온톤 코디를 과소평가할 수 있다. 모노톤 선호 사용자를 위한 예외인지, 초기값 조정인지 평가 데이터로 판단한다.
- 아우터 레이어링 +1단계(§5.2)는 얇은 가디건과 두꺼운 코트를 같게 취급한다. 아우터의 warmthLevel별로 가산 폭을 달리하는 확장이 후보다.
- Delta E 2000 구현을 직접 작성할지 검증된 소형 라이브러리를 쓸지는 구현 시점에 결정한다. 직접 작성 시 참조 구현 대비 오차 테스트가 필요하다.
