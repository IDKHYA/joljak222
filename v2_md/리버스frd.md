# 리버스 FRD — 퍼스널컬러 기반 코디 추천 시스템 v2 재구축 명세

작성 기준일 2026-07-02. 이 문서는 현 저장소(v1)를 리버스 엔지니어링하여 요구사항을 추출하고, 완전히 새로 만드는 v2의 착수 기준이 되는 단일 문서다. 루트에 흩어진 기존 문서 30여 개를 이 문서가 대체하며, 처분 기준은 부록 A에 있다.

**독자와 사용법.** 작성자 본인과 후속 개발 세션이 읽는다. v2 작업을 시작하는 세션은 이 문서만 읽으면 되도록 자기완결적으로 썼다. v1 코드는 §3의 승계 목록에 있는 것만 참조하면 된다.

**재구축 3원칙.**
1. UI는 계승한다 — 현 화면 구조와 룩앤필을 100% 유지 지향(로직 변경이 요구하면 변경 가능).
2. 로직·데이터·모델은 백지에서 재선정한다 — v1 방법론에 성역 없음.
3. 기술 선정 기준은 무료 최우선, 로컬 실행 선호, 성능은 그다음. 유료 고성능 옵션은 참고로만 병기.

## 1. 프로젝트 정체성

### 1.1 한 줄 정의

얼굴 사진과 설문으로 12계절 퍼스널컬러를 진단하고, 사용자의 옷장·오늘의 날씨·색채 조화를 함께 반영하여 **근거를 설명할 수 있는** 코디를 추천하는 웹 서비스.

### 1.2 독립 프로젝트로서의 가치

- 퍼스널컬러 진단 앱, 디지털 옷장 앱, 코디 추천 앱은 각각 존재하지만, 세 가지를 **하나의 의사결정 흐름**(내 톤 → 내 옷 → 오늘 날씨 → 오늘 입을 옷)으로 묶은 서비스는 드물다. 통합이 곧 차별점이다.
- 옷 데이터 확보 경로가 셋이다 — 검수된 카탈로그, 쇼핑몰·인터넷 이미지 URL 수집, 직접 촬영. "옷을 하나하나 찍어야 한다"는 디지털 옷장의 고질적 진입 장벽을 카탈로그와 URL 수집이 흡수한다.
- 추천이 블랙박스가 아니다. 모든 추천 점수가 4축(퍼스널컬러 적합·날씨·색 조화·안정성)으로 분해되고, 각 축의 계수는 색채 이론 근거 또는 "경험칙 초기값 + 향후 학습"으로 명시된다.
- 새 옷 구매 유도가 아니라 보유 의류 활용도를 높이는 방향이라 지속가능 소비 관점의 명분도 있다.

### 1.3 졸업작품 관점 — v1에서 이미 달성한 것

### 1.4 논문 관점 (마감 2026-11)

- 실험(전문가 일치율, 리커트 만족도)은 아직 수행 전이므로, **지금이 방법론을 교체할 수 있는 마지막 시점**이다. v2에서 방법을 바꾸면 논문 3~4장을 새로 쓰고 실험도 v2 기준으로 설계한다.
- v1의 RF 계절 분류기(89.74%)는 폐기가 아니라 **논문의 베이스라인**으로 승격한다. "고전 ML 파이프라인(v1) 대비 패션 특화 임베딩 기반 접근(v2)의 비교"가 그 자체로 실험 챕터가 된다. 같은 516장 라벨 데이터로 두 방법을 비교하면 추가 라벨링 비용 없이 실험이 성립한다.
- 기여 후보(v2 기준)는 세 가지다. ① 사진·설문 하이브리드 12계절 진단(승계·보강), ② 패션 특화 임베딩 기반 의류 속성 인식(신규, v1 RF와 비교 — 명칭은 "착용 계절"로 통일), ③ 신뢰도 적응형 4축 다기준 추천(승계·사용자 평가로 가중치 정당화).
- URL 수집(FR-10)이 쌓는 상품명·카테고리 텍스트는 속성 인식의 약한 라벨(weak label)로 재사용할 수 있어, 추가 라벨링 비용 없는 평가 데이터 확보 경로도 된다.
- **논문 최소 경로(이중 트랙).** 서비스 일정이 밀려도 논문이 성립하도록 분리한다 — 진단 엔진(완성) + 라벨 516장 3자 비교(서비스 탑재와 무관하게 독립 실행 가능) + 추천 가중치 사용자 평가. 이 셋만 있으면 기여 ①②③이 전부 방어된다.

### 1.5 집중 배분 — 빡집중과 덜 집중

전 절의 요구를 같은 강도로 만들면 11월을 못 지킨다. 배분 원칙은 하나다 — **시연에서 관객이 보는 순서대로 힘을 쓴다.** 골든 패스(§4.0) 위의 1분이 패스 밖의 한 달보다 중요하다.

**빡집중 — 여기서 승부한다.**
1. **P0 데이터 구출.** 유일한 비가역 리스크. 라벨 516장과 원본 이미지는 이 컴퓨터 밖에 사본이 없다. 다른 모든 것은 다시 만들 수 있지만 이것은 잃으면 끝이다.
2. **카탈로그 품질.** 시연에서 관객이 보는 모든 옷이 카탈로그다. 메타데이터 실측값, 이미지 품질, 스타일 일관성이 곧 서비스의 체감 품질이다.
3. **옷장 빌더와 추천 성립 보장.** 골든 패스의 유일한 신규 기능이자 관객의 첫 상호작용. 프리셋은 "어떤 퍼컬·어떤 기온에서도 겹치지 않는 조합 N개가 성립"하도록 큐레이션되어야 하며, 이것은 UI가 아니라 데이터 설계 문제다(FR-9).
4. **추천 결과의 설득력.** 심사자가 판단하는 순간은 점수 내부가 아니라 화면이다 — 조합이 사람 눈에 어울려 보이는가, 근거 문장이 읽히는가. 엔진의 내부 정교함보다 이 체감이 우선이다.
5. **논문 실험 조기 병행.** 3자 비교는 라벨 516장만 있으면 서비스 진척과 무관하게 돌릴 수 있다. 8월부터 병행 시작해 11월 리스크를 미리 소진한다(§8 P2).

**덜 집중 — 동작하면 충분하다.**
1. 사진 등록 정교화 — 부가 경로다. 착용샷 부위 분할은 MVP에서 제외한다(§6.1).
2. 로컬 VLM(Ollama) — 선택 의존성. 무료 엔드포인트 배치 호출로 충분하다.
3. 저장 계층 재설계 — v1 코드와 테스트를 그대로 이식하고 끝낸다. 재설계는 필요가 증명될 때만(§7).
4. 옷장 관리 부가 기능(상태 관리 등) — v1 수준 이식으로 종료.
5. 라벨 402장 신규 라벨링 — 착용 계절 직접 분류가 폐기되어 서비스에 불필요하고, 논문 비교도 동일 516장 유지가 오히려 깨끗하다. 라벨링 예산은 516장 재감사와 프리셋 큐레이션으로 재배정한다(§8 P2).

**하지 않는다.**
- 생성형 가상 착용(VTON), 사이트별 스크래퍼, 4축 가중치 학습의 서비스 기능화(논문 분석으로만 수행), v1 저장소 안에서의 버그 수정(v1은 동결 — 수정은 전부 v2에서).

## 2. v1 해부 — 무엇이 있고 왜 재시작인가

### 2.1 v1 구성 요약

- 프론트 — React 19 + TypeScript + Vite 6, Tailwind 4 + shadcn 계열. 화면은 `src/features/`(home, personal, wardrobe, recommendation, try-on, saved-outfits), 상태는 `src/hooks/` 6개, 로직은 `src/services/`.
- 서버 — FastAPI(`server/background_remove_api.py`). rembg u2netp 일반 누끼, SegFormer(sayeed99/segformer-b3-fashion) 정밀 추출, 계절/재질 예측(`season_predictor.py`).
- 데이터 — 카탈로그 890개(상의·하의 등 860 + 아우터 30), 이미지 `public/catalog/` PNG 861장. 12계절 팔레트·traits(`personalColorWorkbook.ts`).
- 테스트 — vitest 42개(추천 29, clothingMeta 8, imageStore 5). `npm run lint`는 `tsc --noEmit`.

### 2.2 구조적 문제 — 재시작의 근거

계획 없이 증축된 결과로 다음 문제가 누적되었다. 부분 수리로는 해소되지 않아 재시작을 결정했다.

**(1) 재현 불가능한 데이터 파이프라인 — 가장 치명적.**
카탈로그 생성 스크립트 전부가 이 컴퓨터의 절대경로에 묶여 있다.

```
scripts/generate_catalog.py:5   RESULT_DIR = r'C:\Users\dheod\Documents\ForMe\project_image_model\이미지 데이터\최종\케이스 1\결과'
scripts/generate_catalog_v2.py:5 BASE_DIR  = r'C:\Users\dheod\Documents\ForMe\project_image_model\이미지 데이터\2차 확장'
scripts/check_categories.py:5   path = r'C:\Users\dheod\Downloads\lastbose\통합_퍼컬_옷장\src\data\trainingCatalog.ts'
```

원본 이미지·라벨 데이터가 저장소 밖(`Documents\ForMe\project_image_model`)에 있고, 스크립트가 가리키는 저장소 경로조차 현 폴더(`…옷장 - 복사본`)가 아닌 옛 폴더다. 즉 지금 이 저장소에서는 카탈로그를 재생성할 수 없고, 다른 컴퓨터에서는 아예 불가능하다. RF 모델도 산출물(`server/artifacts/season_model.joblib`)만 저장소에 있고 학습 코드는 저장소 밖에 있다.

**(2) 의류 등록 파이프라인의 설계 부재.**
등록 경로가 두 갈래(rembg 일반 누끼 / SegFormer 정밀 추출)인데, 계절·재질 예측은 정밀 경로에만 붙어 있다. 어느 경로로 등록했느냐에 따라 같은 옷의 메타데이터 품질이 달라진다. 예측 로직 자체도 어긋나 있다.
- RF의 95개 특징 중 노출도(옷 픽셀/전체 픽셀), 소매 비율, bbox 위치 등은 **착용샷을 전제**로 설계되었는데, 실제 등록 이미지의 다수는 상품컷·누끼 이미지라 이 특징들이 무의미하게 계산된다.
- 규칙 기반 계절 점수(`get_rule_season`)의 출력이 RF의 입력 특징으로 다시 들어가는 순환 구조다.
- 재질 추론이 라벨→재질 고정 매핑(sweater→knit, jacket→nylon)으로, 예측이라 부르기 어려운 수준이다.
- 계절 taxonomy가 3분류(봄/가을·여름·겨울)로 통합되어 있는데 추천 엔진의 날씨 밴드와의 연결이 느슨하다.

코드 레벨 전수 해부(확정 버그 3건 — 사용자 계절 수정 무시, 이전 예측값 잔존, isDenim/material 모순 — 과 상품컷 계통 편향 증명, RF 95특징 분해, 문제 18건 종합표)는 `의류인식_리버스해부.md`에 있다. v2 등록 파이프라인 작업 시 그 문서의 §6이 체크리스트다.

**(3) 카탈로그 데이터 결함.**
`representativeColor`에 색 이름 대신 HEX가 들어가 있고(생성 스크립트 결함), `isNeutral`이 890개 전부 false다. 인간 라벨은 860장 중 458장만 완료. 신규 등록 의류는 `clothingMeta.buildColorMeta`로 정정되지만 기존 카탈로그는 스크립트 재실행이 불가능해(문제 1) 고칠 수 없는 상태다. 추가 실측(2026-07-02, 표본 12장 픽셀 검사) — 카탈로그 PNG 860장은 투명 누끼가 아니라 **RGB 회색 배경 상품컷**이고 해상도도 낮다(대부분 418×313). 데일리룩 캔버스에 올리면 회색 상자가 그대로 보이고, 마스크 없이 대표색을 추출하면 배경 회색(RGB 199~225 — 근백색 임계로는 안 걸러짐)이 섞인다. 고해상도 원본 구출(P0)과 배치 누끼(P1)가 필요한 실증 근거다.

**(4) 라이선스 회색지대.**
정밀 추출에 쓰는 sayeed99/segformer-b3-fashion은 NVlabs SegFormer(비상업 연구용) 라이선스를 승계한다고 표기되어 있어 공개 배포 시 문제가 될 수 있다.

**(5) 코드·문서 위생.**
`App.tsx` 547줄에 라우팅·상태·핸들러가 집중. `src/features/` 일부 파일과 `checklist.md`·`context-notes.md`에 인코딩 깨진 한글 문자열 존재. 루트에 성격·신선도가 제각각인 문서 30여 개가 중복·상충(설문 8문항/5문항, 특징 수 97/95 등의 불일치 이력).

### 2.3 반면 건강한 부분

## 4. 요구사항

구현 기술과 무관하게 시스템이 충족해야 하는 요구다. v2에서 기술이 바뀌어도 이 절은 유효하다.

### 4.0 MVP 골든 패스 — 시연 크리티컬 패스

시연(졸업작품 발표·전시)에서 관객이 실제로 밟는 단일 흐름이며, v2 개발 우선순위의 최상위 기준이다.

```
퍼컬 측정 → 퍼컬 결과 확인·이미지 저장
  → 옷장 획득 — 스타일 프리셋 옷장 선택 또는 미니 설문 옷장 빌더 (FR-9)
  → 코디 추천 — 겹치지 않는 조합 N개 (FR-4)
  → 마음에 드는 조합을 데일리룩 저장 (FR-6)
  → 데일리룩 제작 → 완성 이미지 저장
```

원칙 세 가지.
- 이 경로 위의 옷은 전부 카탈로그 의류다. **사진 등록(FR-2a)과 URL 수집(FR-10)은 크리티컬 패스 밖의 부가 경로**이며, 실패하거나 미완성이어도 시연이 죽지 않는다. v1이 폭발한 원인이 임의 사진 인식을 크리티컬 패스에 둔 것이므로, 이 배치가 v2의 핵심 교정이다.
- 골든 패스는 외부 네트워크 없이 완주 가능해야 한다(NFR-9).
- "겹치지 않는 조합"은 같은 옷이 두 조합에 재사용되지 않는 N개의 코디 세트로 정의한다(가정 — 다르면 본 줄을 갱신).

### 4.1 기능 요구 (FR)

**FR-1 퍼스널컬러 진단.**
카메라 사진과 설문을 융합해 12계절 중 Top1·Top2 시즌, 신뢰도, 추천·회피 팔레트를 산출한다. 판정 근거(사진 신호, 설문 신호, 융합 비율)를 사용자에게 노출한다. 외부 유료 API 없이 동작해야 한다. 사진 품질이 낮으면 사진 비중을 자동으로 낮추고 재촬영을 권한다. **진단 결과 카드는 이미지 파일로 저장할 수 있어야 한다**(골든 패스 2단계 — v1에 없으면 신규 구현).

**FR-2 의류 스키마와 수집 원칙.**
추천 엔진이 소비하는 의류 스키마를 계약으로 먼저 고정하고, 모든 수집 경로가 같은 스키마를 채운다. 스키마 필드는 대표색 상위 3개(비율 포함), 카테고리, 보온 등급, 패턴, 중립색 여부를 기본으로 하되, **각 필드는 추천 점수 기여가 역추적으로 증명된 것만 유지한다**(v1의 material·isDenim 등은 기여 재심사 후 존폐 결정). 수집 경로는 세 가지다 — ① 카탈로그(사람이 검수한 완성 행, FR-8) ② URL 수집(FR-10) ③ 사진 등록(FR-2a). 자동 인식은 **초안 작성기**이고 사용자 수정이 **최종 권한**이다 — 어떤 예측값도 사용자가 확정한 값을 덮어쓰지 못하며, 관측→병합→저장은 단일 경로 하나만 둔다(v1 확정 버그 A·B·C의 구조적 재발 방지). 착용 계절은 직접 분류하지 않고 관측 속성(소매 길이·두께감·카테고리)에서 **보온 등급으로 파생**하며, 퍼스널컬러 시즌과 착용 계절은 타입·식별자·UI 문구를 분리한다(`계절개념_혼용_보고서.md` 처방 1~3).

**FR-2a 사진 등록 (부가 경로).**
이미지 1장으로 의류 초안을 생성한다 — 누끼 PNG, 카테고리, 대표색(색 이름 + HEX + 상위 후보 3개), 패턴, 보온 관측치. 상품컷·착용샷 구분 없이 **단일 파이프라인으로 동일 품질**이 나와야 한다. 수정 이력은 학습 데이터로 축적한다.

**FR-3 옷장 관리.**
복수 옷장 생성·이름 변경·삭제, 의류 추가·삭제, 상태 관리(보유중/세탁중/보관중/추천 제외). 카탈로그에서 내 옷장으로 복사 가능.

**FR-4 코디 추천.**
상의×하의(×아우터) 조합을 4축 점수로 평가해 상위 결과를 제시한다. 각 추천은 축별 점수 분해와 자연어 근거 문장을 반드시 동반한다(설명 가능성은 협상 불가 요구). 진단 신뢰도가 낮으면 퍼스널컬러 축 비중을 자동으로 낮춘다. 날씨(현재 위치 기온)를 반영하고 위치 권한 거부 시 폴백이 있어야 한다. 추가 요구 두 가지 — ① **겹치지 않는 조합 N개(기본 3) 세트 제시**. 같은 옷이 두 조합에 재사용되지 않아야 하며, v1에 없는 알고리즘이므로 단위 테스트와 함께 신규 구현한다(골든 패스 4단계). ② **날씨 수동 오버라이드**. 기온을 손으로 설정해 추천을 제어할 수 있어야 한다 — 위치 거부 폴백이자 시연 제어 수단(NFR-9).

**FR-5 단일 옷(앵커) 추천.**
특정 옷 하나를 고정하고 어울리는 조합을 찾는다. 결과 다양성을 보장한다(동일 상대 옷 반복 제한). 옷장/카탈로그/혼합 소스 선택 가능.

**FR-6 데일리룩.**
저장한 코디를 누끼 PNG 레이어로 한 장의 룩 이미지로 배치·편집·저장한다. **완성본은 PNG 이미지 파일로 내보낼 수 있어야 한다**(골든 패스 종착점 — 관객이 들고 가는 산출물).

**FR-7 데이터 영속성.**
모든 사용자 데이터는 브라우저 로컬에 저장된다(서버 계정 없음). 이미지 대용량은 용량 제한이 있는 저장소를 회피해 저장하고, 스키마 변경 시 자동 마이그레이션한다.

**FR-8 카탈로그.**
자체 라벨링 이미지 기반 카탈로그를 제공한다. 카탈로그 메타데이터는 저장소 내 파이프라인으로 원클릭 재생성 가능해야 한다. 골든 패스의 주연 데이터이므로 스키마 전 필드가 실측값이어야 하며(v1의 isNeutral 일괄 false 같은 자리 채움 금지), 스타일 분류는 옷장 빌더(FR-9)가 요구하는 범위만 갖춘다 — **890벌 전수 태깅이 아니라 프리셋별 수동 큐레이션이 기본**이고, 나머지는 임베딩 제로샷 초안으로 충분하다.

**FR-9 옷장 빌더 (신규 — 골든 패스 필수).**
사용자가 옷을 한 벌도 등록하지 않은 상태에서 즉시 쓸 수 있는 옷장을 만들어 준다. 두 방식 — ① 스타일 프리셋(스타일 단위로 미리 구성한 카탈로그 옷장을 고른다) ② 미니 설문(3~5문항으로 취향을 받아 카탈로그에서 자동 구성한다). 산출물은 일반 옷장과 동일한 구조여서 이후 추가·삭제·수정이 자유롭다.
- 프리셋은 3~4종(예: 캐주얼·미니멀·스트릿), 종당 30~40벌을 수동 큐레이션한다. 전수 스타일 태깅보다 싸고 품질이 높다.
- **추천 성립 보장이 프리셋의 합격 기준이다.** 어떤 12계절 퍼컬, 어떤 기온 밴드에서도 겹치지 않는 조합 N개가 성립해야 한다(대략 상·하·아우터 각 파트가 모든 기온 밴드에서 적합 의류 N벌 이상). 이것은 UI가 아니라 데이터 큐레이션 요구이며, **검증 스크립트로 자동 확인**한다 — 시연 중 "추천 결과 없음"이 나오는 순간 발표가 죽는다.

**FR-10 URL 수집 (신규 — 부가 경로).**
쇼핑몰 상품 페이지 URL 또는 인터넷 이미지 URL을 붙여넣으면 서버가 이미지를 가져와 FR-2 파이프라인으로 초안 메타데이터를 생성한다.
- 상품 페이지 파싱은 범용 규칙만 쓴다 — Open Graph(og:image·og:title) → JSON-LD(schema.org/Product) → 페이지 내 최대 크기 이미지 휴리스틱 순. **사이트별 전용 어댑터는 만들지 않는다**(유지보수 폭발 방지).
- 상품명 텍스트를 속성 추정에 활용한다 — v1의 이름 신호 규칙(청·denim·코트 등)을 승계하고, VLM JSON 관측(§6.2 [5])에 상품명을 함께 넣는다. 쇼핑몰 상품명은 종류·소재·핏 정보를 담고 있어 이미지 단독보다 초안 품질이 높다.
- 차단·파싱 실패는 정상 시나리오다 — 명확한 실패 안내와 함께 "이미지를 저장해서 사진 등록으로" 폴백을 제시한다. 성공률 100%를 목표하지 않는다.
- 브라우저 CORS 제약 때문에 반드시 서버 경유이며, http/https 외 스킴과 사설 대역 주소 요청은 거부한다(SSRF 방지).
- 가져온 이미지는 사용자 로컬 옷장에만 저장하고 재배포하지 않는다.

## 6. 기술 재선정 (2026-07 조사 기준)

선정 기준은 ① 무료 ② 로컬(소비자 GPU 8~16GB 또는 CPU) ③ 성능. 조사는 2026년 7월 초 웹 문헌 기준이며 출처는 각 항목에 표기했다.

### 6.2 의류 수집·분석 파이프라인 v2 — 핵심 재설계

v1의 문제(경로별 품질 편차, 착용샷 전제 특징, 순환 특징, 계절 직접 분류)를 해소한다. 구조는 "수집 어댑터 3종이 공통 분석 파이프라인을 거쳐 같은 스키마(FR-2)를 채운다"이다.

```
수집 어댑터 (같은 스키마를 채우는 세 입구)
  A. 카탈로그 — 사람이 검수한 완성 행. 골든 패스 주연 (FR-8)
  B. URL — 서버가 og:image/JSON-LD/최대 이미지 순으로 이미지+상품명 획득 (FR-10)
  C. 사진 — 사용자 촬영·업로드 (FR-2a)
        ↓ B·C는 아래 공통 분석 진입 (A는 배치로 같은 파이프라인을 통과해 생성)
[1] BiRefNet 누끼 → 투명 PNG + bbox — 카탈로그 배치(P1)와 URL·사진(P2) 공통. 카탈로그는 RGB 회색 배경이라 생략 불가(실측, §2.2 (3))
[2] (MVP 제외) 착용샷 부위 분할 — 사진 등록 강등으로 필요 소멸, 후순위 옵션으로만 유지
[3] K-means 대표색 → 색 이름 매핑(COLOR_META 방식 승계), isNeutral
[4] Marqo-FashionSigLIP 임베딩 제로샷 관측 — 카테고리·패턴·소매 길이·두께감
[5] (선택) VLM JSON 관측 — 상품명 텍스트 + 이미지 → 소재·세부 속성
    (로컬 Qwen3-VL 또는 build.nvidia.com 무료 엔드포인트, 사전·배치 호출 위주)
[6] 보온 등급 파생 — 관측 속성(소매·두께·카테고리)의 명시적 규칙 → WarmthLevel
    (착용 계절 직접 분류 없음. 색·노출도는 보온 신호에서 제외)
  → 초안 제시 → 사용자 확인·수정(최종 권한) → 저장 — 병합 경로는 이 한 곳뿐
```

설계 의도.
- **착용 계절 직접 분류를 폐기하고 관측→파생으로.** v1은 색 밝기·노출도 같은 촬영 조건에서 계절을 추론해 흰 패딩·검은 린넨 반례와 상품컷 계통 편향이 났다. 파생 규칙은 사람이 읽고 고칠 수 있어 오류 수정이 재학습이 아니라 규칙 한 줄 수정이 된다. 계절 명칭이 예측 대상에서 사라지므로 퍼컬 시즌과의 어휘 충돌도 구조적으로 소멸한다.
- **주 입력은 상품컷이라는 사실을 전제로 설계.** 카탈로그·URL 수집이 주력이고 착용샷은 예외 케이스다. v1은 반대로 착용샷 전제 특징을 상품컷에 적용해 편향이 났다.
- **단일 병합 경로**가 v1 확정 버그 A(사용자 수정 무시)·B(이전 예측 잔존)·C(재질 모순)의 재발을 구조적으로 차단한다.
- 학습·평가 스크립트를 전부 v2 저장소 안에 두어 NFR-4를 충족한다.
- **논문 트랙은 서비스와 분리해 유지.** 동일 라벨 516장에서 v1 RF(95 수공 특징) vs 임베딩+분류 헤드 vs VLM 제로샷의 3자 비교. 서비스에 무엇이 탑재되든 실험은 독립적으로 성립한다.
- FashionSigLIP은 150M이라 CPU에서도 돌아 서버 요구사항이 오히려 낮아진다.
- **P1 카탈로그 배치는 [1][3][4][6]이다.** 분할·VLM 없이 성립하고, [1]은 일회성 오프라인 배치(860장)라 시연 런타임에는 어떤 모델도 필요 없다. 누끼 산출물(투명 PNG)은 색 추출 정확도와 데일리룩 레이어 품질의 공통 전제다.

리스크와 대비. 임베딩 제로샷 관측의 정확도가 기대에 못 미치면 관측 항목을 줄이고 사용자 확인 단계에 더 기대면 된다 — 초안 작성기 위치라서 서비스가 죽지 않는다. 논문 쪽은 3자 비교 결과가 어느 방향이든 그 자체로 내용이 되므로 손해가 없다.

### 6.3 퍼스널컬러 진단 — 승계 + 2건 보강

방법 유지(로컬 계산, MediaPipe). 보강 후보는 ① ITA° 정식 도입(현 b* 근사를 L* 포함 표준 지표로 — 논문 §4 정합성), ② 조명 보정 품질 지표를 신뢰도에 더 직접 반영. 둘 다 융합 구조는 바꾸지 않는다.

### 6.4 추천 엔진 — 로직 승계 + 가중치 학습 경로 확보

4축 설계와 상수를 그대로 이식하되, v2에서는 사용자 평가 데이터(10월 실험)로 4축 가중치(현재 휴리스틱 0.38/0.22/0.28/0.12)를 보정하는 절차를 설계에 포함한다. "이론 기반 사전값 + 평가 데이터로 학습"이 논문 기여 ③의 서사다.

추가 — 의류색↔시즌 적합도 함수(퍼컬 축)의 재가공을 검토한다. 현재의 "시즌 스와치 점 목록과의 최근접 CIEDE2000 거리"를, 스와치 감사(중복·이상치 제거) → LAB 사전 변환 캐시 → **시즌별 적합 영역(명도·채도·온도 축의 범위) 표현**으로 개선한다. 스와치에 없는 색도 자연스럽게 판정되고 회피색도 같은 틀에서 표현된다. 이 함수는 진단 로직과 독립이므로 완성된 진단은 건드리지 않는다.

날씨 축은 seasonTag 하드컷 대신 보온 등급(§6.2 [6])↔기온 밴드 직접 매핑으로 교체하고, 부적합 시 완전 소거가 아니라 강한 감점을 기본으로 한다(오분류 한 번에 아이템이 영구 실종되는 v1 문제 방지).

### 6.5 스택

프론트 React 19 + TS + Vite 유지(UI 계승이 목적이므로 당연). 서버 FastAPI 유지하되 엔드포인트를 등록 파이프라인 v2에 맞춰 재정의(`/api/clothing/analyze` 단일 진입 + 단계별 옵션). Ollama는 선택 의존성(없으면 VLM 단계 생략 또는 API 폴백).
