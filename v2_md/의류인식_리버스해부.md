## 1. 전체 데이터 흐름 지도 — 의류가 시스템에 들어오는 5개 경로

같은 "옷 등록"인데 경로마다 생성되는 메타데이터가 다르다. 이것이 파이프라인의 근본 문제다.

| # | 경로 | 트리거 | 누끼 | 계절 예측 | 재질 예측 | 카테고리/종류 감지 |
| --- | --- | --- | --- | --- | --- | --- |
| 1 | 자동 분석 | 파일 선택 즉시 (`handleFileChange` → `autoAnalyzeOnUpload`) | SegFormer (`targetPart='auto'`) | O | O | O (픽셀 다수결 + fine label) |
| 2 | 누끼 따기 버튼 | `removeManualBackground` | rembg u2netp | **X** | **X** | X |
| 3 | 정밀 추출 버튼 | `extractManualClothingPrecisely` | SegFormer (사용자 선택 카테고리 기반) | O | O | X (사용자 카테고리 신뢰) |
| 4 | 카탈로그 선택 | `fromCatalog` (App.tsx:118) | 없음 (저장된 메타 복사) | 스크립트 생성값 | 전부 `unknown` | 스크립트 생성값 |
| 5 | 수동 입력만 | 분석 실패/미사용 | 없음 | 사용자 선택 | 텍스트 규칙 | 사용자 선택 |

경로 1이 기본값(파일 선택 즉시 실행)이고 경로 2·3은 보조 버튼이다. 즉 **일반 누끼(rembg)가 "빠른 기본 경로"라는 문서상 설계와 달리, 실제 기본 경로는 SegFormer 정밀 추출이다.** 문서(README·CLAUDE.md)와 구현이 이미 어긋나 있다.

최종 저장 시 `addManualItem`(App.tsx:234-267)이 경로별 산출물을 병합하는데, 이 병합 규칙에 확정 버그가 있다(§4.2).

## 2. 서버 계층 해부

### 2.1 공통 전처리

- 프론트 `resizeImageFileForUpload`(clothingImageApi.ts:4) — 긴 변 1280px, JPEG 품질 0.9로 재인코딩 후 업로드.
- 서버 `resize_for_model`(api:53) — 긴 변 1600px 상한. 프론트에서 이미 1280으로 줄여 보내므로 사실상 무동작.
- SegFormer 추론은 `run_fashion_segmentation`(api:194)에서 **다시 640px로 축소**해 수행하고, 결과 라벨맵을 NEAREST로 원본 크기로 되돌린다. 즉 실제 분할 해상도는 640px이다.

### 2.2 일반 누끼 — `POST /api/background/remove` (api:289)

rembg u2netp → `harden_alpha`(api:78)로 알파 이진화. 임계값 두 개를 쓴다.

- `ALPHA_KEEP_THRESHOLD=56` — 이 미만 알파는 0, 이상은 255로 강제. 반투명(시스루·레이스) 원단의 계조가 소실된다.
- `ALPHA_EDGE_THRESHOLD=24` — 이 값으로 구한 bbox **사각형** 내부만 유지. 주석은 "가장자리 1px 유지"라고 하지만 실제 동작은 bbox 클리핑이다(주석-구현 불일치).

응답에 `bbox`, `colors`(대표색), 모델명, 버전(`hard-alpha-v3`)을 담는다. **계절·재질·카테고리 필드는 없다.** 이 경로로 등록하면 해당 메타는 프론트 텍스트 규칙과 사용자 입력에만 의존한다.

### 2.3 대표색 추출기 — `extract_dominant_colors` (api:226)

1. 이미지를 1/4로 축소, 알파 96 미만 픽셀 제외.
2. **근백색 픽셀 제외** — `max(r,g,b)>245 and min(r,g,b)>235`. 배경 잔여물 제거 목적이지만, 조명이 밝은 흰 옷은 본체 픽셀이 통째로 걸러져 **그림자 픽셀이 대표색이 되는** 왜곡이 있다. 흰 티가 그레이·베이지로 등록되는 증상의 근원.
3. 각 채널을 24 단위로 버킷 양자화(±12 오차) 후 빈도 상위 5색 반환.

한편 `season_predictor.get_dominant_colors`(sp:213)는 **같은 서버 안에서 전혀 다른 방식**(K-means + 실루엣 점수로 k 자동 선택, 비율 4% 미만 탈락)으로 대표색을 또 뽑는다. 같은 이미지에서 API 응답 색과 RF 입력 색이 다를 수 있다. 대표색 추출기가 2개 공존하는 셈이다.

### 2.4 정밀 추출 — `POST /api/clothing/extract` (api:318)

SegFormer(sayeed99/segformer-b3-fashion, Fashionpedia 계열 47클래스) 라벨맵에서 요청 부위 id만 남긴다.

| targetPart | 포함 라벨 id | 비고 |
| --- | --- | --- |
| upper | 1,2,3,6 + 28,29,32,34 | 셔츠/티/스웨터/조끼 + 후드·칼라·소매·네크라인 |
| lower | 7,8,9 | 팬츠/반바지/스커트 |
| outer | 4,5,10,13 + 28,29,32,34 | 가디건/재킷/코트/케이프 + 공유 4종 |
| shoes | 24 | |
| bag | 25 | |
| accessory | 14~20,23,25,26 | 안경·모자·타이·장갑·시계·벨트·양말·가방·스카프 |
| auto | 픽셀 다수결로 위 그룹 중 택1 | §2.5 |

구조적 한계 — **28,29,32,34(후드·칼라·소매·네크라인)가 upper와 outer에 동시 소속**이다. 시맨틱(인스턴스 아님) 분할이라 "재킷 안에 셔츠"인 착용샷에서 upper를 요청하면 재킷의 소매·칼라 픽셀이 함께 추출된다. 레이어드 착용샷에서 정밀 추출 품질이 무너지는 근본 원인.

후처리 `clean_boolean_mask`(api:121) — 5×5 모폴로지 close/open → 구멍 메움 → 연결 성분 상위 3개만 유지. dress/jumpsuit(11,12)는 upper·lower 요청 시 자동 합류(api:342).

`fineLabels`는 픽셀 200개 초과 라벨만 담는다(api:365). 반면 계절 예측기 내부의 `infer_labels_from_pred`(sp:311)는 **임계값 없이 1픽셀이라도 있으면 라벨로 인정**한다. 같은 응답 안에서 두 기준이 다르다.

### 2.5 카테고리 자동 감지와 "96.7%"의 실체

`detect_dominant_category`(api:164)는 5개 카테고리의 메인 라벨 픽셀 수를 세어 최다 카테고리를 고른다(전부 0이면 None → auto는 'upper' 폴백). 로직 자체는 건전하다.

다만 checklist.md의 "카테고리 29/30=96.7%"는 `scripts/eval_clothing_extract.py` 기준으로 **카탈로그 누끼 PNG 30장(카테고리당 6장)** 평가다. 이미 배경이 제거된 정제 이미지 대상이므로, 사용자 실사진(배경·착용·조명 포함)에 대한 성능 근거로는 쓸 수 없다. 참고로 이 스크립트는 상대경로 기반이라 scripts/ 중 유일하게 이식 가능하다.

## 3. 계절·재질 예측기 해부 — `season_predictor.py`

### 3.1 2계층 구조

```
predict(pred, mask, image, target_part)                    (sp:377)
  ├─ 규칙 계층 get_rule_season → (계절, 점수표, high/low)   (sp:260)
  ├─ 재질 = 라벨→재질 고정 매핑                             (sp:322)
  ├─ RF 모델 없으면 → 규칙 결과 반환 (conf 0.7/0.4 하드코딩)
  └─ RF 있으면 → 95특징 행 구성 → predict_proba → 최대 확률 클래스
```

출력 taxonomy는 `봄/가을·여름·겨울` 3분류 + `미분류`(label_mapping.json).

### 3.2 규칙 계층 전체 점수표

**의류 라벨 점수** (LABEL_SEASON_SCORE, sp:41)

| 라벨 | 점수 |
| --- | --- |
| shorts | 여름 +3 |
| top/t-shirt, shirt/blouse, skirt, dress, vest | 여름 +1, 봄/가을 +1 |
| jacket, cardigan | 봄/가을 +2 |
| jumpsuit, cape, pants | 봄/가을 +1 |
| coat | 겨울 +3 |
| sweater | 겨울 +2, 봄/가을 +1 |
| leg warmer, glove, scarf | 겨울 +2 |
| tights/stockings | 겨울 +1 |

**디테일 점수** (sp:51) — hood·zipper 겨울 +1, pocket·buckle 봄/가을 +1.

**색 밝기 점수** (sp:270) — 상위 2색에 대해 밝기 >180이면 여름 +1.0×비율(봄/가을 +0.5×비율), <80이면 겨울 +1.0×비율, 그 외 봄/가을 +1.0×비율.

**소매 비율** (sp:285) — 소매 픽셀/의류 픽셀 <0.08이면 여름 +2, ≥0.20이면 봄/가을 +1 겨울 +1.

**노출도** (sp:293) — 의류 픽셀/전체 픽셀 <0.10이면 여름 +2, ≥0.40이면 겨울 +2, 그 외 봄/가을 +0.5.

신뢰도는 최다 점수/총점 >0.6이면 high(0.7), 아니면 low(0.4)의 2단계 하드코딩이다.

### 3.3 [증명] 상품컷 계통 편향 — 워크스루 2건

소매·노출도 규칙은 "착용샷에서 피부 노출이 많으면 여름"이라는 가정이다. 상품컷·플랫레이에서는 이 가정이 **계통 오차**로 뒤집힌다.

**워크스루 1 — 흰 반팔티 상품컷** (프레임의 45% 차지, 소매 비율 22%, 대표색 흰색 밝기 230·비율 0.9)

| 규칙 | 여름 | 봄/가을 | 겨울 |
| --- | --- | --- | --- |
| top/t-shirt 라벨 | +1 | +1 | |
| 색 밝기 >180 | +0.9 | +0.45 | |
| 소매 ≥0.20 | | +1 | +1 |
| 노출 ≥0.40 (프레임 가득) | | | +2 |
| **합계** | **1.9** | **2.45** | **3.0** |

→ 여름옷인 흰 반팔티가 **겨울**로 판정된다(신뢰도 low 0.4). 상품컷은 옷이 프레임을 채우는 것이 정상인데, 노출도 규칙이 이를 "노출 없는 겨울 복장"으로 해석하기 때문이다.

**워크스루 2 — 운동화 상품컷** (auto 감지 shoe, 화면의 8%, 대표색 밝기 200·비율 0.7 + 120·비율 0.3)

| 규칙 | 여름 | 봄/가을 | 겨울 |
| --- | --- | --- | --- |
| 라벨 점수 (MAIN_IDS에 shoe 없음 → fine label 0건) | 0 | 0 | 0 |
| 색 밝기 | +0.7 | +0.65 | |
| 소매 0 (<0.08) | +2 | | |
| 노출 8% (<0.10) | +2 | | |
| **합계** | **4.7** | **0.65** | **0** |

→ 4.7/5.35 = 0.88 > 0.6이므로 **여름, 신뢰도 high(0.7)**. 색이 어두운 겨울 부츠여도 겨울은 +1 미만이라 결과가 뒤집히지 않는다. 즉 **신발·가방류는 무엇을 올려도 사실상 "여름 고신뢰"로 판정**되고, 이 값이 UI의 `seasonTagFromAI`(useManualClothing.ts:52)로 그대로 계절 태그에 들어간다. 겨울 부츠가 여름 태그를 달고 저장된다.

### 3.4 RF 계층 — 95특징 전수 분해

`server/artifacts/feature_columns.json`의 95개 컬럼을 그룹별로 분해하면 다음과 같다.

| 그룹 | 개수 | 내용 | 평가 |
| --- | --- | --- | --- |
| label_ratio_* | 24 | 의류·디테일 라벨별 픽셀 비율 | 의류 자체 신호. 유효 |
| color_1~4_* | 20 | 상위 4색의 R/G/B/밝기/비율 | 유효하나 RGB 원값(지각 균등 아님) |
| color_stats_* | 13 | 밝기·채도 통계, 웜/쿨/중립 비율, 엣지 밀도 | 유효 |
| quality_* | 8 | **image_pixel_count 등 절대 픽셀 수 3종** + 가시성 비율 | 절대값은 해상도 의존 → 위험 |

| fullbody_expanded, colors_count | 2 | | |

### 3.5 구조 결함 5가지

1. **순환 특징.** 규칙 계층의 계절 점수·판정·신뢰도 9개 컬럼이 RF 입력으로 재유입된다. RF가 규칙의 편향(§3.3)을 학습으로 교정하는 게 아니라 **편향을 특징으로 상속**한다. "룰 19.2% → RF 89.74%" 비교도 독립 비교가 아니다.
2. **해상도 민감 특징.** `quality_image_pixel_count` 등 절대 픽셀 수가 특징이다. 학습 이미지와 서빙 이미지(프론트 1280 리사이즈)의 해상도가 다르면 같은 옷도 다른 특징 벡터가 된다.
3. **학습-서빙 분포 불일치.** 학습은 외부 데이터셋(레포 밖 `project_image_model`)의 정제된 추출 결과, 서빙은 사용자 실사진이다. 노출도·bbox 위치 같은 구도 특징은 분포가 완전히 다르다.
4. **학습 데이터 편중의 물증.** fine_* 원핫 컬럼이 상의 4종(shirt/blouse, sweater, top/t-shirt, vest)뿐이다. 원핫 컬럼은 학습 데이터에 등장한 값만 생성되므로, **학습 셋에서 하의·아우터의 fine label이 사실상 등장하지 않았다**는 뜻이다(part별 라벨 추출 로직상 하의는 pants/shorts/skirt가 나와야 하는데 컬럼이 없음). 89.74%가 상의 편중 데이터에서 나온 수치일 가능성이 높다.
5. **범위 밖(OOD) 파트.** 신발·가방·액세서리는 part_* 원핫이 전부 0인 학습에 없던 입력인데도 `predict()`가 그대로 실행되어 확률을 반환한다. 거부(reject) 경로가 없다.

### 3.6 재질 예측 — 예측이 아니라 조회표

`MATERIAL_FROM_FINE_LABEL`(sp:56) 고정 매핑이 전부다.

| 라벨 | 재질 | 문제 |
| --- | --- | --- |
| sweater, cardigan | knit | 무난 |
| coat | wool | 패딩 코트도 wool |
| jacket, vest | nylon | 데님·레더 재킷도 nylon |
| **pants, shorts, skirt** | **cotton** | **청바지도 cotton** — §4.2 버그 C의 원인 |
| shirt/blouse, top, dress, jumpsuit | cotton | |

이미지의 시각 정보(텍스처·광택)는 전혀 쓰지 않는다. "재질 예측"이라는 이름의 카테고리→재질 조회표다.

## 4. 프론트 계층 해부

### 4.1 등록 상태 머신 — `useManualClothing.ts`

파일 선택(`handleFileChange`:84) 시 상태를 초기화하고 즉시 `autoAnalyzeOnUpload`를 실행한다. 초기화 대상은 `segmentation, aiAnalyzed, aiConfidence`뿐이고 **`predictedSeasonTag`·`predictedMaterial`은 초기화하지 않는다**(버그 B의 원인). 자동 분석 성공 시 카테고리·종류·색·계절 태그·segmentation·예측값을 일괄 반영하고, 실패 시 `describeFailure`가 헬스체크로 "서버 꺼짐"과 "분석 실패"를 구분해 안내한다.

### 4.2 [확정 버그 3건] — 저장 병합 규칙

**버그 A — 사용자의 계절 수정이 조용히 무시된다.**
- 폼의 계절 태그 select는 `manual.seasonTag`만 수정한다(WardrobeSection.tsx:542). `predictedSeasonTag`를 지우는 수단이 UI에 없다.
- 저장 시 `addManualItem`은 `predictedSeasonTag`를 우선한다(App.tsx:254-256).

```ts
seasonTag: (manual.predictedSeasonTag && manual.predictedSeasonTag !== '미분류')
  ? manual.predictedSeasonTag   // 예측이 있으면 무조건 이것
  : manual.seasonTag,           // 사용자가 고른 값은 예측이 없을 때만
```

→ 분석이 "겨울"로 예측한 옷을 사용자가 드롭다운에서 "여름"으로 고쳐도 **겨울로 저장**된다. §3.3의 계통 편향과 결합하면 치명적이다 — 신발은 항상 "여름 고신뢰"로 예측되는데(워크스루 2), 사용자가 고쳐도 반영되지 않는다.

**버그 B — 이전 이미지의 예측값이 다음 아이템에 유출된다.**
`handleFileChange`(useManualClothing.ts:88)가 `predictedSeasonTag`·`predictedMaterial`을 초기화하지 않으므로, 이미지 A 분석 성공(예측 "겨울") 후 이미지 B를 올렸는데 자동 분석이 실패하면(서버 꺼짐 등) A의 예측값이 남는다. 이 상태로 저장하면 **B가 A의 계절·재질을 물려받는다.** 버그 A의 우선 병합 규칙 때문에 사용자가 알아챌 방법도 없다.

**버그 C — isDenim=true인데 material='cotton'인 모순 아이템이 생성된다.**
저장 시 재질은 `(manual.predictedMaterial) ?? meta.material`(App.tsx:258)로 서버 예측이 텍스트 추론을 이긴다. 청바지 사진을 자동 분석하면 SegFormer가 pants로 인식 → 재질 조회표가 **cotton** 반환(§3.6) → 사용자가 종류를 "청바지"로 골라 `meta.material='denim'`이 나와도 cotton이 저장된다. 한편 `isDenim`은 meta에서 오므로 true. 결과는 `{ isDenim: true, material: 'cotton' }`. 추천 엔진의 데님 관련 로직과 재질 로직이 서로 다른 판단을 보게 된다.

### 4.3 종류 매핑의 정보 손실 체인 — 청바지 사례

`FINE_LABEL_TO_TYPE`(wardrobeConstants.ts:156)이 SegFormer 라벨을 앱 종류로 바꾸는데 다대일 강제 매핑이다.

- `pants` → **'슬랙스'** 고정. 청바지·조거팬츠도 전부 슬랙스가 된다.
- `top, t-shirt, sweatshirt` → **'반팔티'** 고정. 긴팔티·맨투맨도 반팔티가 된다.

청바지 자동 등록의 전체 체인은 다음과 같다.

```
청바지 사진 → SegFormer 'pants' → 종류 '슬랙스' (데님 텍스트 신호 소실)
→ buildColorMeta 텍스트 = "하의 슬랙스 #HEX" → /청|데님|denim/ 불일치
→ isDenim은 denimByHex에만 의존 (최근접 색이 '데님'(#5C7898)일 때만 true)
→ 중청은 운 좋게 데님 판정, 흑청·생지는 블랙/차콜 최근접 → isDenim=false
→ 재질은 어차피 서버 예측 'cotton'이 덮어씀 (버그 C)
```

즉 자동 경로로 등록한 검은 청바지는 "슬랙스, cotton, 데님 아님"이 된다. v1 문서들이 자랑하는 데님 워시 구조화(연청/중청/진청/흑청)는 **이름을 직접 입력한 경우에만** 제대로 작동한다.

### 4.4 색·패턴·계절 규칙 (clothingMeta.ts)

- **색 정규화** — `nearestColorMeta`가 임의 HEX를 26색 팔레트(COLOR_META)에 CIEDE2000 최근접 매핑. 설계 자체는 건전하고 테스트도 있다(clothingMeta.test.ts 8건). 한계는 팔레트 커버리지 — 오렌지 계열이 없어 주황이 베이지로 매핑되는 기록이 있다(context-notes 2026-06-05).
- **패턴** — `inferPatternType`은 **순수 텍스트 정규식**이다(stripe/체크/그래픽 키워드). 이미지는 전혀 보지 않으므로, 자동 등록(이름 없음) 아이템은 전부 `solid`가 된다. RF 특징의 edge_density가 있지만 패턴 분류엔 쓰이지 않는다.
- **계절(이름 규칙)** — `getSeasonTag`(clothingMeta.ts:175)는 종류 문자열 기반(패딩·코트·니트→겨울 등)이다. 단, 자동 등록 시엔 버그 A의 병합 규칙 때문에 이 값이 아니라 서버 예측이 쓰인다. **계절 판정기가 시스템에 3개**(이름 규칙, 서버 규칙 계층, RF) 존재하고 우선순위가 경로마다 다르다.

## 5. 카탈로그 생성 경로 — 결함의 근원 (`scripts/generate_catalog.py`)

카탈로그 890개의 메타 결함은 전부 이 스크립트의 하드코딩에서 나왔다(139줄, v2·outer 스크립트도 동일 패턴).

```python
f'representativeColor: "{e["rep_hex"]}", ...'   # 색 이름 자리에 HEX (121-129행)
f'patternType: "solid" as const, '              # 전부 solid 고정
f'material: "unknown" as const, isNeutral: false, isDenim: false, '  # 전부 고정
```

- `representativeColor`에 색 이름 대신 HEX — clothingMeta의 `colorMetaForInput`을 거치지 않고 문자열을 직접 조립했기 때문. 알려진 결함의 정확한 위치.
- `isNeutral/isDenim/patternType/material` 일괄 고정 — 추천 엔진의 중립색 보너스, 데님 로직, 패턴 페널티가 카탈로그 아이템에 대해서는 전부 무력화된다.
- `human_label` 없으면 `data.get('season')` 폴백(67행) — 라벨 미완 402장의 seasonTag는 §3.2 규칙 계층의 출력이다. 즉 **카탈로그 계절 태그의 절반 가까이가 편향 증명된 규칙의 산출물**이다.
- 헤더 주석은 "516개"라는데 실제 생성은 860개(주석 부패), import 경로도 `from '../App'`으로 현행 구조와 불일치.

## 7. v2 재설계로의 연결

이 해부가 리버스frd.md의 두 가지 미결 사항을 뒷받침한다.

1. **계절 직접 분류 → 속성 유도 전환의 근거 보강.** 편향의 원인이 전부 "촬영 조건을 계절 신호로 쓴 것"(#3)과 "판정기 난립"(#9)이다. 소매 길이·소재·두께감 같은 의류 속성을 관측하고 계절을 규칙으로 유도하면 #3·#5·#8·#9가 구조적으로 소멸한다.
2. **비교 실험의 전제 조건.** v1 RF를 논문 베이스라인으로 쓰려면 #5(순환 특징)와 #6(상의 편중)을 논문에 정직하게 기술해야 한다. 특히 #6은 P0 데이터 감사에서 파트별 라벨 분포를 세어 확정할 것.
3. **UI 계약에 추가할 것.** 예측값 표시 UI(WardrobeSection.tsx:463-472)는 유지하되, "예측 해제/수정이 저장에 반영"이 v2 UI 계약의 필수 항목이다(버그 A 재발 방지).
