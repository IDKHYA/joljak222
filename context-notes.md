### labTemperatureIndex로 바꾼 이유

기존 `colorTemperatureIndex`는 `clamp(((r-b) + (r-g)*0.35) / 255, -1, 1)`. RGB 채널 차이를 직접 뺀 방식이라 조명 색온도에 크게 흔들림.

Lab b* 축은 노랑(웜)↔파랑(쿨)을 perceptually uniform space에서 표현. 조명 변화에 덜 민감하고 사람 눈의 웜/쿨 지각과 상관관계가 더 높음. a* 보정(-0.18 비율)으로 분홍 쿨 뉘앙스도 소폭 반영.

### harmonyScore 진화 경로

1단계 (커밋 전) — 84/72 이진값. 사실상 상수.

2단계 (8df20c9) — Lab ΔE로 이상 범위를 정의하고 inRange=82/else=55 step function. 개선됐지만 경계에서 27점 절벽.

3단계 (7e51ac6) — hue 각도로 Itten 색상 이론 5개 타입 분류. step function 제거, 타입마다 고유 기본 점수. 시즌 대비 선호도로 타입별 가산/감산.

절충 사항. 중성색(isNeutral=true)은 hue가 의미 없어서 별도 처리(기본 85). 그러나 아우터와 신발은 여전히 조화 계산에 미포함 — 상의↔하의만 측정.

### CIEDE2000 채택 이유

CIE76은 1976년 단순 유클리드 거리. 파란 계열(쿨 퍼스널컬러의 핵심 영역)에서 사람 눈과의 상관관계가 낮음. PubMed 비교 연구 기준 CIE76 상관계수 0.176 vs CIEDE2000 0.289.

CIEDE2000은 RT(회전항)으로 파란 계열 비균일성을 보정하는 2001년 ISO 표준. 숫자 스케일이 CIE76보다 작아 보정 상수 재조정 필요 (곱수 3.2→4.5, 임계값 18/28→10/16).

`geminiService scorePaletteMatch`는 사진 진단 알고리즘 쪽이라 의도적으로 CIE76 유지. 진단 결과와 추천 점수를 다른 공식으로 계산해도 각 시스템이 내부적으로 일관성 있으면 문제없음.

### 미완 사항

- `dominantColors` 배열이 `ClothingItem`에 있지만 `scoreItemForPersonalColor`는 `representativeHex` 단일값만 씀. 체크무늬/그라데이션 의류에서 정확도 손실 있음.

### 개념 정정 (중요)

퍼스널 컬러 계절 ≠ 옷의 계절. 완전히 다른 시스템.

- 퍼스널 컬러 계절 (light-spring, soft-summer 등): 사람 피부톤에 어울리는 색상 스펙트럼을 계절에 비유한 것. 실제 날씨와 무관.
- 옷의 계절 (봄/가을, 여름, 겨울): 옷의 두께/재질/소매 길이 등으로 실제 입기 적합한 날씨를 분류한 것.

이 두 가지는 추천 알고리즘에서 독립적인 두 축으로 작동해야 함.
  - 퍼컬 적합도 (Delta E) → 이 색상이 사용자 퍼컬에 맞는가
  - 날씨/계절 적합도 → 지금 날씨에 입을 수 있는 옷인가 (Python 모델이 담당)

### CatalogItem 인터페이스 이동 이유

원래 `CatalogItem`이 `App.tsx`에 있고 `trainingCatalog.ts`가 그걸 import했는데, App도 trainingCatalog를 import하므로 순환 참조가 생겼다. TypeScript 5.8은 순환 참조 시 타입을 `unknown`으로 추론해버려서 Map 생성자 반환 타입이 `Map<unknown, unknown>`이 되는 문제가 발생했다. 해결책은 `CatalogItem`을 `trainingCatalog.ts`로 이동해 App이 단방향으로만 import하도록 만드는 것.

### Map 명시적 타입 필요 이유

TS 5.8에서 `new Map(array.map(item => [item.id, item]))` 은 tuple 추론이 실패하면 `Map<unknown, unknown>` 이 된다. `new Map<string, ScoredClothingItem>(...)` 처럼 제네릭을 명시해야 함.

### v2_shoe_shoe 중복 원인

### 소분류 필터 설계

### dominantColors 3색 팔레트 매칭 전환 이유

기존 `scoreItemForPersonalColor`는 `item.representativeHex` 단일값만 비교했다. 체크무늬/스트라이프처럼 여러 색이 섞인 의류는 대표색 하나로는 실제 착장 느낌을 반영하지 못함.

전환 방식.
- `scoreSingleHex(hex, paletteLabs, avoidLabs)` 헬퍼로 단일 색상 점수 계산을 분리
- `dominantColors`가 있으면 상위 3색을 `ratio` 가중 평균으로 최종 점수 산출
- 없으면 기존과 동일하게 `representativeHex` 단일값 사용 (하위 호환)

타입 수정 사항.
- `paletteLabs / avoidLabs`를 `[number, number, number][]`로 잘못 선언했다가 `LabColor[]`로 수정
- `rgbToLab()` 반환이 `LabColor` (`.l .a .b` 객체)이므로 tuple 캐스트 `as [number, number, number]` 불필요 → 제거
- `LabColor` 타입을 `colorUtils`에서 named import 추가

# 2026-05-25 이 옷으로 코디 찾기 기능.

## 왜 이 기능인가.

기존 추천은 옷장 전체를 상의×하의×아우터로 전수 조합해 점수순으로 보여 준다. 사용자가 "이 옷 하나를 중심으로 뭘 입을까"를 묻는 흐름은 없었다. 기준 옷을 고정하고 어울리는 짝을 찾는 것은 실제 옷 입기 사고방식에 더 가깝고, 카탈로그까지 풀에 넣으면 "내가 가진 이 옷에 살 만한 옷"까지 제안할 수 있다.

## 설계 결정과 근거.

- 엔진을 새로 만들지 않고 재사용한다. `buildRecommendations`의 점수 합성 블록을 `scoreOutfit` 헬퍼로 추출해 `buildAnchoredRecommendations`와 공유한다. 점수·reason·fitReason 산식이 두 경로에서 동일하게 유지되는 게 핵심이라, 합성 로직을 한 곳에 둔다.
- `scoreOutfit` 추출은 기존 점수를 바꾸면 안 된다. 그래서 조화 회귀 스냅샷(89/85/84)을 그대로 통과시키는 것을 추출 성공 기준으로 삼는다. outfit.id 문자열 포맷만 `items.map(id).join('-')`로 바뀌는데, id는 React key 용도라 점수와 무관하고 테스트도 id를 단정하지 않는다.
- 기준 카테고리는 v1에서 상의·하의·아우터만. 신발·액세서리는 코디의 포인트 아이템이라 상하의 전체를 다시 찾아야 해서 조합 규칙이 더 복잡하다. 과거에 신발을 자동 조합에서 뺀 결정(2026-05-15)과도 일관된다. 신발/액세서리 기준은 빈 배열을 반환하고, 진입 버튼도 세 카테고리에만 노출한다.
- 날씨 축은 '상관없음'으로 고정한다. 이 기능의 질문은 "이 옷에 색이 어울리는 옷"이지 "오늘 날씨에 맞는 옷"이 아니다. band를 '상관없음'으로 두면 `getWeatherScore`가 보유 상태만 반영하고 기온 필터는 끈다. 결과적으로 퍼컬 38 + 조화 28의 색 기반 축이 사실상 추천을 주도한다.
- 후보 풀 출처는 사용자가 토글로 고른다. 처음엔 기본값을 정해 주려 했으나, 사용자가 "그냥 선택하게 하자"고 해서 옷장 / 카탈로그 / 둘 다를 화면에서 직접 고르게 한다. 기본은 둘 다.
- 카탈로그 아이템은 추천 풀에 들어가려면 `ScoredClothingItem`이어야 한다. `fromCatalog`로 `ClothingItem`을 만든 뒤 `scoreItemForPersonalColor`로 퍼컬 점수를 매겨 풀에 넣는다. 카탈로그는 크므로 점수화 결과를 `useMemo`로 캐시한다.
- 결과 카드는 기존 코디 카드 렌더를 재사용한다. 출처 구분은 이미 있는 `sourceType`('catalog' | 'upload')으로 보유/카탈로그 뱃지를 더한다.

## 미뤄 둔 것.

- 신발·액세서리 기준 코디.
- 조화 축의 보조색(dominantColors) 반영. 패턴 데이터 신뢰도가 받쳐 주는 Tier 3 이후로 둔다.

# 2026-06-01 논문 기반 추천 알고리즘 고도화 (A·B·C·D).

논문(`논문.md`)과 코드를 대조해 갭을 찾고, 코드를 논문 수준 이상으로 끌어올렸다. 핵심 설계 결정과 근거는 다음과 같다.

## 공통 — 회귀 안전 원칙.
- 테스트 `mockResult`에는 `seasonTop2Id`·`confidence`가 없다. 그래서 "신뢰도/Top2 부재 → 조정 0(=기존 동작)"을 기본값으로 두면 기존 23개 테스트와 조화 회귀 스냅샷(89/85/84)이 그대로 통과한다. 개선은 실제 앱의 `FinalResult`(항상 confidence·Top2 보유)에서만 발동한다. 이게 "안전하게 개선"의 핵심.

## A. Top1+Top2 신뢰도 융합.
- 근거. `FinalResult`는 §3-2대로 Top1·Top2·confidence를 모두 갖는데(`types.ts` 127~131), 추천은 `seasonTop1Id`만 썼다(`recommendationEngine.ts` 98·510행). "진단은 확률적인데 추천은 1등만 본다"는 모순.
- 데이터. `result.palette === SEASON_PROFILES[seasonTop1Id].palette`(`personalColorEngine.ts` 439). Top2는 `SEASON_PROFILES[seasonTop2Id].palette`, 회피색은 `SEASON_DETAILS[*].worstColors`로 대칭 접근.
- 설계. `top2Share = clamp((CONF_FULL−confidence)/(CONF_FULL−CONF_FLOOR),0,1) × TOP2_MAX_SHARE`. CONF_FULL=0.9, CONF_FLOOR=0.5, TOP2_MAX_SHARE=0.35. 팔레트 점수·회피 페널티를 Top1/Top2 가중 평균으로 합성. confidence가 높으면(≥0.9) top2Share=0 → 기존과 동일.
- 신뢰도는 이미 gap(=Top1−Top2 점수차)을 크게 반영(`gap×1.35`)하므로, confidence 단일 신호로 Top2 근접도를 대리한다.

## B. 신뢰도 기반 4축 가중 동적화.
- 근거. §4-1-4 "사진 품질 낮으면 보조신호 비중↑"의 추천판. 진단이 불확실하면 색 적합도(주관·진단 의존)를 덜 믿고 색 조화(객관 지표)에 더 기댄다.
- 설계. `resolveScoreWeights(result)`: personal을 최대 0.12까지 낮추고 그만큼 harmony로 이전. 신뢰도 부재/null → 38/22/28/12 그대로. weather·stability 고정.
- baseScore도 같은 동적 가중을 써서 `qualityAdjustment`(=score−baseScore)가 "가중 변화"가 아니라 "정밀 조화 vs 레거시 조화 차이"만 나타내게 유지.
- 발표 영향. 38/22/28/12는 이제 "완전 신뢰 시 기준값"이며 불확실할수록 조화로 이동한다는, 오히려 더 강한 스토리.

# 2026-07-02 MVP 골든 패스 확정과 FRD 대개정 — 수집 3경로 체계.

- 결정(확인 완료). 시연 MVP는 단일 골든 패스 — 퍼컬 측정 → 결과 확인·이미지 저장 → 옷장 획득(스타일 프리셋 선택 또는 미니 설문 빌더) → 추천(겹치지 않는 조합 N개) → 데일리룩 저장·제작·이미지 저장. 이 경로 위 옷은 전부 카탈로그이며, 사진 등록은 부가 경로로 강등. v1이 폭발한 원인(임의 사진 인식을 크리티컬 패스에 둠)의 직접 교정.
- 문제 재정의. 진짜 문제는 인식 실패가 아니라 스키마 부재 — 추천 엔진이 소비하는 의류 스키마를 계약으로 먼저 고정하고, 카탈로그(검수 완성 행)/URL/사진이 같은 스키마를 채우는 초안 작성기로 위치. 사용자 수정이 최종 권한, 관측→병합→저장 단일 경로(버그 A·B·C 구조적 차단).
- 신규 요구 FR-10 URL 수집. 쇼핑몰 상품 페이지·인터넷 이미지 URL → 서버가 og:image→JSON-LD→최대 이미지 순 범용 파싱으로 이미지+상품명 획득 → 공통 파이프라인. 사이트별 어댑터 금지, 실패는 정상 시나리오(사진 등록 폴백 안내), 서버 경유 필수(CORS)+SSRF 가드, 이미지는 로컬 저장만.
- 우선순위 역전. 부가 경로 안에서 URL > 사진 — 상품컷은 배경이 깨끗하고 상품명 텍스트(종류·소재·핏)가 딸려 와 구현이 쉽고 초안 품질이 높으며 차별성이 큼. 상품명·카테고리는 속성 인식의 약한 라벨로 논문 평가 데이터로도 재사용.
