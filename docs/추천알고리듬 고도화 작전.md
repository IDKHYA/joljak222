## 0. 이 문서는

바로 코드부터 손대지 않기 위해 먼저 쓴다. 내가 어떤 생각으로 헤매다 무엇을 깨달았는지, 그래서 엔진의 현재 상태가 어떤지, 무엇부터 고칠지를 전부 적어둔다. 다음 작업 세션(나 자신이든 에이전트든)이 이 문서만 읽고도 맥락을 다시 세우지 않고 이어갈 수 있게 하는 게 목적이다.

---

## 1. 출발점 — 내가 빠져 있던 곳

발표를 앞두고 나는 한동안 "주변 기술"을 파고 있었다. 솔직히 말하면 이런 것들이었다.

- 옷 카테고리 분류 모델 고도화
- 이미지 인식 모델 고도화 (SegFormer, 누끼, 색 추출)
- 청바지처럼 더 세밀한 색감 분류 모델
- 체크무늬 옷의 패턴·다중 색상 인식 (OpenCV FFT/Hough 까지 검토)
- 카탈로그 데이터 규칙화
- 데일리 룩 · Try-On 기능

각각은 다 의미가 있었지만, 어느 순간 "근데 내가 이걸 왜 하고 있지"라는 생각이 들었다. 패턴 인식이 어려운 기술이냐 아니냐를 따지고 있었는데, 정작 그게 발표의 논지와 어떻게 연결되는지는 흐릿했다. 곁가지 기술을 하나씩 정복하는 데 집중력이 새고 있었다.

---

## 2. 깨달은 점 (전환점)

그러다 분명해졌다.

> 지금까지 한 모든 것(카테고리 고도화, 이미지 인식, 패턴 인식)은 **전부 추천 알고리듬을 더 잘 굴리기 위한 입력 데이터**였다.

즉 주변 기술은 목적이 아니라 수단이었다. 그리고 한 발 더 들어가니 이렇게 정리됐다.

- 옷 추천을 고도화하려면 기본적으로 **옷에서 나오는 정보 데이터**가 많아야 한다.
- 이건 **퍼스널컬러** 프로젝트니까 그중에서도 **색상 정보**가 핵심이다.
- 따라서 내가 집중해야 하는 건 **"퍼컬과 함께 옷을 추천해주는 이 도메인" 그 자체**다.

곁가지가 아니라 엔진. 데이터 수집이 아니라, 그 데이터를 점수로 바꾸는 논리. 거기가 졸작의 논지이고, 내가 깊게 파야 할 곳이었다.

---

## 3. 다시 세운 목표

집중 영역을 "퍼컬 + 옷 추천 도메인"으로 못박고, 구체적으로 네 가지를 하기로 했다.

1. **엔진 전체 진단** — 4축이 실제로 어떻게 도는지, 어디가 강하고 어디가 약한지 먼저 지도부터 그린다.
2. **설명 문장 대폭 고도화** — 추천 결과가 "왜 이 옷이 네 퍼컬에 맞는지"를 제대로 말하게 한다.
3. **데이터 고도화 가능성 열어두기** — 무료 LLM 비전 API로 옷 인식 데이터를 더 정확하고 많이 확보하는 길을 닫지 않는다.
4. **알고리듬을 더 유기적이고 수치화 가능하게** — 축들이 서로 대화하고, 모든 점수가 근거로 추적되게 만든다.

## 4. 엔진 전체 진단 (현재 상태)

### 4.1 4축 구조

최종 점수는 `recommendationEngine.ts:344`의 `buildRecommendations`에서 합산된다.

```
score = personal × 0.38 + weather × 0.22 + harmony × 0.28 + stability × 0.12
```

색 기반 축(퍼컬 + 조화)이 **66%**를 차지한다. 방향 자체는 "퍼컬로 옷을 추천한다"는 논지와 일치한다. 기반은 서 있다.

### 4.2 축별 강점 · 약점

| 축 | 강점 (발표에서 내세울 것) | 약점 (고도화 대상) |
| --- | --- | --- |
| **퍼컬 38%** (`scoreItemForPersonalColor`) | CIEDE2000 + 팔레트 Delta E, dominantColors 비율 가중 평균(`:96`), 회피색 페널티까지 — 이론적으로 탄탄하다 | 점수 전체가 `100 - 거리 × 4.5`라는 **임의 상수**(`:72`)에 매달린다. 24색 중 최근접 1색만 보니 웬만한 색은 고득점 → 변별력이 떨어진다. neutral/denim +8 보너스(`:110`)도 근거가 없다 |
| **조화 28%** (`calculateHarmonyScore`) | Itten 색상환 분류 + **시즌 대비 선호와 연결**(`:303`). 퍼컬↔조화가 대화하는 유일한 지점이라 가장 "유기적"이다 | hue **각도만** 본다(명도·채도 무시). 대표색 1개만 써서 38% 축과 **불일치**한다(거긴 dominantColors). 베이스 점수 80/82/55/76/88이 임의값이다. 무채색은 그냥 85 고정(`:295`)이다 |
| **날씨 22%** (`getWeatherScore`) | 기온 구간 룰 + 보유 상태 반영, 경계 구간 완충까지 챙긴다 | `type.includes(키워드)` 문자열 매칭이라 취약하다. 다만 허용 범위 |
| **안정성 12%** | 단순하다(전부 보유중 92 / 아니면 68). 가중치가 낮아 무난하다 | 이진값이라 손댈 가치가 적다 |

### 4.3 가장 약한 곳 둘

**(1) 설명 문장이 죽어 있다 — 목표 #2가 정조준하는 지점.**
`recommendationEngine.ts:356`의 `reason`은 고정 템플릿이다. "퍼스널 컬러 적합도와 ◯◯ 날씨 조건을 함께 반영했습니다."가 전부다. 정작 엔진은 시즌 이름, personalScore, `harmonyType`의 한글 라벨(`HARMONY_TITLE_KO`/`HARMONY_BADGE_KO`가 이미 정의돼 있다), 아이템별 등급, `avoidRisk`를 **다 계산해놓고 버린다**. 아이템별 `fitReason`도 "트루 윈터 팔레트 기준 87점"이 끝이다(`:117`). 퍼컬 앱에서 "왜 이 옷이 네 퍼컬에 맞는지"가 가장 중요한 출력인데, 그걸 한 줄도 못 하고 있다. **새 데이터 없이 이미 있는 숫자만 엮어도** 설명이 살아난다.

**(2) 헤드라인 숫자가 임의 상수로 굴러간다 — 목표 #4("수치화")가 겨냥하는 지점.**
`× 4.5`, `+8`, `80/82/55/76/88` 전부 "왜 그 값인지"에 대한 설명이 없다. 발표에서 "그 점수는 어떻게 나온 거죠?"라는 질문이 오면 막힌다. 수치화의 핵심은 모든 숫자가 근거로 추적되는 것이다. 예를 들어 Delta E는 2 미만이면 인지 불가, 10 미만이면 근접, 20 이상이면 다른 색이라는 지각 임계가 있으니, 점수 곡선을 이 임계에 맞추면 상수가 정당화된다.

### 4.4 데이터 측면 — 측정 vs 추측 (LLM 질문의 재구성)

| 필드 | 출처 | 점수 반영 |
| --- | --- | --- |
| 색 (representativeHex / dominantColors) | **이미지에서 측정** (K-means / 누끼 API) | 퍼컬 38% + 조화 28% = **66%** |
| patternType | `inferPatternType` = **텍스트 정규식 추측** (`App.tsx:214`) | 조화 축 패턴 페널티에만 소폭 |
| material | `inferMaterial` = **텍스트 추측** | **점수에 전혀 안 쓰임** (표시용) |
| seasonTag | `getSeasonTag` = **타입 룰 추측** (`App.tsx:306`) + 업로드는 AI 예측 | 날씨 22% |

**핵심 발견.** 제일 중요한 색 정보는 이미 이미지에서 측정하고 있다. LLM이 정확히 채워줄 수 있는 건 패턴·소재·카테고리인데, 패턴은 점수 영향이 작고 소재는 점수에 아예 안 물려 있다. 그래서 **LLM 데이터를 먼저 넣어도 추천 숫자는 거의 안 움직인다.** 패턴을 의미 있는 변수로 만들려면 "데이터 확보(LLM/CV) + 점수에 더 깊게 배선" 두 가지를 함께 해야 하는 더 큰 작업이다.

### 4.5 진단 한 줄 결론

색 데이터는 이미 측정되고 있다. 추천 품질의 병목은 데이터 부족이 아니라 (1) 점수 공식 안의 정당화 안 된 상수와 (2) 엔진이 계산해놓고 버리는 설명 문장이다. 따라서 "유기적·수치화"의 레버리지는 데이터 수집이 아니라 **공식과 설명**에 있다.

## 5. 작전 — 우선순위 로드맵

### Tier 1. 설명 문장 재설계 (효과 크고 위험 낮음, 도메인 그 자체)

- **목표.** `reason`과 아이템별 `fitReason`을, 엔진이 이미 계산한 값으로 동적 생성한다.
- **구체 작업.** 시즌 이름 + 매칭/회피 색 계열 + `harmonyType` 한글 라벨 + 적합 점수대를 엮어 한 문장으로 합성한다. 예. "트루 윈터 팔레트의 네이비 계열과 잘 맞고(적합 92), 상·하의가 보색 대비로 또렷한 인상을 줍니다."
- **건드릴 파일.** `recommendationEngine.ts` (reason/fitReason 생성부), 필요 시 표시 컴포넌트.
- **검증.** `npm run lint` + `npm test` (reason 포맷 단위 테스트 추가) + 화면에서 실제 문장 확인.
- **리스크.** 낮음. 새 데이터·새 의존성 없음.

### Tier 2. 수치화 · 유기화 (알고리듬 고도화의 본체)

- **목표.** 임의 상수를 근거 있는 매핑으로 바꾸고, 축들이 일관된 데이터를 쓰게 한다.
- **구체 작업.**
  - 퍼컬 점수의 `× 4.5` 선형 스케일을 Delta E 지각 임계(2 / 10 / 20)에 맞춘 곡선으로 교체하고 근거를 주석화한다.
  - 조화 점수가 hue뿐 아니라 명도·채도 대비도 반영하게 한다.
  - 조화 계산도 대표색 1개가 아니라 dominantColors를 쓰게 해 38% 축과 일관성을 맞춘다.
  - 무채색 고정 85를 명도 대비 기반으로 세분화한다.
- **건드릴 파일.** `recommendationEngine.ts`, 필요 시 `colorUtils.ts`.
- **검증.** 점수가 바뀌므로 회귀 테스트 필수. 변경 전후 대표 케이스 점수 스냅샷 비교.
- **리스크.** 중간. 점수 분포가 달라지니 시연용 결과를 다시 확인해야 한다.

### Tier 3. LLM 데이터 보강 (1·2 끝난 뒤, 패턴을 진짜 변수로 만들 거면)

- **목표.** 무료 비전 LLM으로 업로드 의류의 패턴·소재·카테고리를 추측이 아닌 관측으로 채운다.
- **전제.** 데이터만 넣지 말고 **점수 배선까지** 함께 해야 효과가 난다(특히 소재는 현재 점수 미반영).
- **건드릴 파일.** 데이터 파이프라인(`App.tsx`의 메타 빌더), 새 LLM 서비스 모듈, 엔진의 패턴/소재 반영부.
- **리스크.** 높음(상대적). API 키·네트워크 지연 등 시연 리스크. 발표 전이면 신중히.

## 6. 검증 원칙

이전 모듈 분리 때 확립한 패턴을 그대로 따른다.

- 변경마다 `npm run lint`(tsc --noEmit) 클린 유지.
- `npm test`(vitest) 통과 유지. 점수·문장 로직은 단위 테스트로 고정한다.
- `npm run build` 성공 확인.
- 한 논리 단위마다 의미 있는 커밋. 점수가 바뀌는 변경은 회귀 스냅샷을 함께 남긴다.
