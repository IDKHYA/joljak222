# 4.3 디지털 옷장과 의류 등록 모듈 및 추천 알고리즘 관계 상세분석서

## 1. 모듈 개요

디지털 옷장과 의류 등록 모듈은 사용자의 의류 데이터를 구조화하여 저장하고, AI 추천 모듈이 이를 기반으로 날씨, 퍼스널컬러, 보유 상태, 색상 조화 조건에 맞는 코디 조합을 생성하도록 연결하는 핵심 데이터 계층이다.

현재 구현은 서버 DB가 아닌 브라우저 `localStorage` 기반의 클라이언트 저장 구조이다. 다만 데이터 모델은 `Wardrobe`, `ClothingItem`, `CatalogItem`, `SavedOutfit`처럼 명확한 엔티티 단위로 나뉘어 있어, 이후 서버 DB로 이전할 때도 테이블 구조로 그대로 매핑할 수 있다.

주요 구현 파일은 다음과 같다.

- `src/App.tsx`: 옷장, 의류, 추천, 저장 코디의 상태 및 알고리즘 처리
- `src/data/musinsaCatalogData.ts`: 무신사 이미지 폴더와 JSON 메타데이터 소스 정의
- `src/services/colorUtils.ts`: 색상 거리 계산 및 퍼스널컬러 적합도 계산 보조
- `src/hooks/useWeather.ts`, `src/lib/weather.ts`: 실시간 날씨와 기온 구간 산출

## 2. 데이터 저장 구조

### 2.1 저장 방식

현재 데이터는 `localStorage`에 JSON 문자열로 저장된다. 저장 키는 `STORAGE_KEYS` 상수로 관리한다.

| 저장 대상 | localStorage key | 설명 |
|---|---|---|
| 퍼스널컬러 현재 결과 | `integrated_personal_color_result` | 사용자의 현재 퍼스널컬러 분석 결과 |
| 퍼스널컬러 이력 | `integrated_personal_color_history` | 과거 분석 기록 |
| 옷장 목록 | `integrated_wardrobes` | 사용자가 만든 옷장 단위 |
| 의류 아이템 | `integrated_clothing_items` | 실제 추천에 사용되는 옷 데이터 |
| 저장 코디 | `integrated_saved_outfits` | 추천 결과 중 사용자가 저장한 코디 |

저장과 로드는 `loadJson`, `saveJson` 유틸 함수로 처리된다. React 상태가 변경될 때 `saveJson`을 호출하여 브라우저 저장소와 UI 상태를 동기화한다.

### 2.2 옷장 데이터 모델

`Wardrobe`는 옷을 묶는 컨테이너이다.

```ts
interface Wardrobe {
  id: string;
  name: string;
  createdAt: string;
}
```

추천 화면에서 여러 옷장을 선택하면, 선택된 옷장 ID에 속한 의류만 추천 후보로 사용된다. 즉 옷장은 단순 화면 분류가 아니라 추천 후보군을 제한하는 필터 역할을 한다.

### 2.3 카탈로그 의류 데이터 모델

`CatalogItem`은 관리자가 미리 준비한 의류 DB의 원본 데이터이다. 현재 무신사 이미지 폴더와 JSON 분석 파일을 합쳐 생성된다.

```ts
interface CatalogItem {
  catalogItemId: string;
  name: string;
  category: ClothingCategory;
  subcategory: string;
  imageUrl: string;
  color: string;
  size: string;
  brand: string;
  representativeColor: string;
  representativeHex: string;
  seasonTag: string;
  patternType: string;
  isNeutral: boolean;
  isDenim: boolean;
  sourceType: 'catalog';
}
```

카탈로그 데이터는 사용자가 옷장에 추가하기 전의 후보 데이터이다. 실제 사용자의 옷장에 들어가면 `ClothingItem`으로 변환된다.

### 2.4 사용자 의류 데이터 모델

`ClothingItem`은 추천 알고리즘이 실제로 사용하는 핵심 데이터이다.

```ts
interface ClothingItem {
  id: string;
  wardrobeId: string;
  imageUrl: string;
  category: ClothingCategory;
  type: string;
  color: string;
  size: string;
  brand: string;
  createdAt: string;
  representativeColor: string;
  representativeHex: string;
  seasonTag: string;
  patternType: string;
  availabilityStatus: AvailabilityStatus;
  isNeutral: boolean;
  isDenim: boolean;
  sourceType: 'catalog' | 'upload';
  catalogItemId?: string;
}
```

추천 알고리즘에서 특히 중요한 필드는 다음이다.

| 필드 | 추천 알고리즘 사용 방식 |
|---|---|
| `wardrobeId` | 선택된 옷장만 추천 후보로 제한 |
| `category` | 상의, 하의, 아우터, 신발 분리 조합 |
| `type` | 기온별 의류 종류 필터링 |
| `representativeHex` | 퍼스널컬러 적합도 및 색상 조화 계산 |
| `color` | UI 표시 및 색상명 기반 보조 판단 |
| `seasonTag` | 계절감/기온 보조 점수 |
| `availabilityStatus` | 추천 제외, 세탁중, 보관중 처리 |
| `isNeutral` | 무채색 안정 조합 가산 |
| `isDenim` | 데님 안정 조합 가산 |
| `catalogItemId` | 같은 카탈로그 옷 중복 추천 방지 |

## 3. 의류 등록 흐름

### 3.1 관리자 준비 DB 기반 등록

관리자가 준비한 이미지 폴더와 JSON 메타데이터는 `src/data/musinsaCatalogData.ts`에서 정의된다.

현재 등록된 소스는 다음과 같다.

| 소스 ID | 폴더 | 카테고리 | 타입 |
|---|---|---|---|
| `sweatshirts` | 맨투맨/스웨트 | 상의 | 맨투맨 |
| `denimPants` | 데님 팬츠 | 하의 | 청바지 |
| `knits` | 니트/스웨터 | 상의 | 니트 |
| `outers` | 아우터 | 아우터 | 재킷 |
| `shortSleeveTshirts` | 반소매 티셔츠 | 상의 | 반팔티 |

각 소스는 두 종류의 파일을 사용한다.

- 이미지 파일: 실제 의류 사진
- `_result.json`: 카테고리, 색상, dominant color 등 분석 메타데이터

빌드 시점에 `import.meta.glob`이 이미지 파일을 asset URL로 수집하고, 같은 번호의 JSON 파일을 찾아 색상 정보를 병합한다. 예를 들어 `001-상품명.jpg`는 `001-상품명_result.json`과 번호 기준으로 연결된다.

이 방식은 이미지 바이너리를 JSON에 저장하지 않는다. JSON에는 분석 정보만 있고, 이미지는 앱 번들 asset으로 참조된다.

### 3.2 카탈로그에서 옷장으로 저장

사용자가 "나만의 옷장 만들기" 화면에서 카탈로그 옷을 선택하면 `fromCatalog` 함수가 `CatalogItem`을 `ClothingItem`으로 변환한다.

변환 시 추가되는 정보는 다음과 같다.

- 사용자가 선택한 `wardrobeId`
- 앱 내부용 의류 ID
- 보유 상태 기본값 `보유중`
- 원본 추적용 `catalogItemId`
- 추천 알고리즘에 필요한 색상, 계절, 패턴, 데님 여부

이후 `persistClothing`이 `clothingItems` 상태를 갱신하고 `integrated_clothing_items`에 저장한다.

### 3.3 직접 업로드 등록

사용자가 직접 사진을 업로드하는 경우에도 최종 저장 구조는 동일하게 `ClothingItem`이다.

직접 등록 시 사용자가 입력하거나 선택하는 정보는 다음이다.

- 카테고리
- 세부 타입
- 색상
- 사이즈
- 브랜드
- 계절 태그
- 보유 상태
- 이미지 URL

색상과 타입을 기반으로 `buildColorMeta`가 대표 색상, HEX, 계절 태그, 패턴, 무채색 여부, 데님 여부를 계산한다. 이렇게 직접 등록한 옷도 카탈로그 기반 옷과 동일하게 추천 후보가 된다.

## 4. 저장된 옷 데이터의 보정

`reconcileStoredClothing`은 앱 시작 시 저장된 의류 데이터를 현재 카탈로그 기준으로 보정한다.

이 함수가 필요한 이유는 다음과 같다.

- 카탈로그 이미지 경로가 변경될 수 있음
- JSON 색상 분석 결과가 갱신될 수 있음
- 과거에 잘못 저장된 카테고리 또는 색상 정보가 남아 있을 수 있음
- 더 이상 사용하지 않는 샘플 카탈로그 데이터가 localStorage에 남아 있을 수 있음

현재 로직은 `catalogItemId`가 `musinsa-`로 시작하는 데이터만 활성 카탈로그와 매칭하여 최신 정보로 덮어쓴다. 과거 샘플 카탈로그 데이터는 추천 후보에서 제거된다.

## 5. 추천 알고리즘과 옷장 데이터의 연결

### 5.1 추천 후보 수집

추천 화면에서 사용자가 여러 옷장을 선택하면 다음 순서로 후보가 만들어진다.

1. 전체 의류 목록 `scoredItems`에서 선택된 `wardrobeId`에 속한 옷만 필터링
2. `availabilityStatus`가 `추천제외` 또는 `세탁중`인 옷 제거
3. `catalogItemId` 또는 `imageUrl` 기준으로 중복 옷 제거
4. 날씨가 `상관없음`이 아니면 기온별 의류 종류 필터 적용
5. 카테고리별로 상의, 하의, 아우터, 신발 분리

이 단계에서 옷장 데이터는 단순히 표시용이 아니라 추천 후보 집합을 만드는 직접 입력값이다.

### 5.2 퍼스널컬러 점수화

`ClothingItem`은 추천 전에 `ScoredClothingItem`으로 확장된다.

```ts
interface ScoredClothingItem extends ClothingItem {
  personalFitScore: number | null;
  fitGrade: FitGrade | null;
  fitReason: string;
  avoidRisk: boolean;
}
```

퍼스널컬러 결과가 있으면 의류의 `representativeHex`를 사용하여 사용자의 팔레트 색상과 색상 거리(`deltaE`)를 계산한다. 거리가 가까울수록 `personalFitScore`가 높아진다.

즉 의류 등록 단계에서 저장된 대표 색상 HEX는 추천 품질을 결정하는 핵심 필드이다.

### 5.3 기온별 의류 제외 필터

현재 추천은 날씨를 단순 점수 가산이 아니라 후보 제외 필터로 사용한다.

기온별 규칙은 `WEATHER_RULES`에 정의되어 있다.

| 기온 구간 | 허용 의류 종류 |
|---|---|
| 4도 이하 | 패딩, 코트, 니트, 가디건 |
| 5~8도 | 코트, 재킷, 니트, 맨투맨 |
| 9~11도 | 블레이저, 재킷, 니트, 긴팔티, 셔츠 |
| 12~16도 | 블레이저, 셔츠, 긴팔티, 니트, 맨투맨 |
| 17~19도 | 셔츠, 가디건, 긴팔티, 맨투맨 |
| 20~22도 | 반팔티, 셔츠, 블라우스 |
| 23~27도 | 반팔티, 반바지, 블라우스, 스커트 |
| 28도 이상 | 반팔티, 반바지, 샌들, 스커트 |

특정 기온 구간을 선택하면 해당 구간과 바로 한 단계 낮은 구간의 의류 종류까지 허용한다.

예를 들어 `20~22도`에서는 다음이 허용된다.

- 현재 구간: 반팔티, 셔츠, 블라우스
- 한 단계 낮은 구간: 셔츠, 가디건, 긴팔티, 맨투맨

반대로 더 높은 온도 구간의 옷은 포함하지 않는다. 따라서 추운 쪽으로는 약간 보수적으로 허용하지만, 더운 쪽 옷을 무리하게 포함하지 않는 구조이다.

`상관없음`을 선택하면 이 기온 필터를 적용하지 않고 모든 보유 의류를 조합 후보로 사용한다.

### 5.4 코디 조합 생성

`buildRecommendations`는 필터링된 의류를 조합하여 추천 코디를 만든다.

조합 기본 구조는 다음이다.

- 상의 1개
- 하의 1개
- 아우터 0개 또는 1개
- 신발 0개 또는 1개

아우터와 신발은 없어도 코디 생성이 가능하지만, 상의와 하의는 필수 조합 축으로 사용된다. 생성된 조합은 `outfitUniqueKey`로 중복 여부를 검사한다. 같은 옷 조합이 이미 생성되었다면 추가하지 않는다.

### 5.5 추천 점수 산정

후보 제외 필터를 통과한 조합에 대해서만 점수를 계산한다.

최종 점수는 다음 요소를 가중 합산한다.

| 점수 요소 | 설명 | 가중치 |
|---|---|---|
| `personalScore` | 코디 아이템들의 퍼스널컬러 평균 적합도 | 42% |
| `weatherScore` | 기온/계절감 보조 점수 | 28% |
| `harmonyScore` | 상의와 하의 색상 조화, 무채색 안정성 | 20% |
| `stabilityScore` | 보유중 상태 중심의 안정성 | 10% |

현재 날씨 필터는 먼저 부적합 후보를 제거하고, 점수는 통과한 조합들 사이의 우선순위를 정하는 용도로 쓰인다.

## 6. 저장 코디와 추천 결과의 재사용

추천 결과는 `OutfitRecommendation`으로 생성되고, 사용자가 저장하면 `SavedOutfit`으로 축약 저장된다.

```ts
interface SavedOutfit {
  id: string;
  title: string;
  score: number;
  itemIds: string[];
  colorHexes: string[];
  weatherBand: RecommendationWeatherBand;
  mode: RecommendationMode;
  savedAt: string;
}
```

저장 코디는 의류 전체 데이터를 복사하지 않고 `itemIds`만 저장한다. 화면에 표시할 때는 현재 `items` 목록에서 ID를 다시 찾아 이미지를 보여준다.

이 방식의 장점은 다음이다.

- 의류 이미지나 색상 정보가 수정되면 저장 코디 표시도 최신화됨
- 저장 데이터 크기가 작음
- 같은 옷의 중복 저장을 줄일 수 있음

단점은 다음이다.

- 원본 옷이 삭제되면 저장 코디에서 해당 아이템을 찾을 수 없음
- 서버 DB로 전환할 때 외래키 무결성 처리가 필요함

## 7. 서버 DB 전환 시 권장 테이블 구조

현재 localStorage 구조를 서버 DB로 이전한다면 다음과 같은 테이블 구조가 적합하다.

### 7.1 wardrobes

| 컬럼 | 타입 | 설명 |
|---|---|---|
| id | string | 옷장 ID |
| user_id | string | 사용자 ID |
| name | string | 옷장 이름 |
| created_at | datetime | 생성일 |

### 7.2 clothing_items

| 컬럼 | 타입 | 설명 |
|---|---|---|
| id | string | 의류 ID |
| user_id | string | 사용자 ID |
| wardrobe_id | string | 소속 옷장 |
| image_url | string | 이미지 경로 |
| category | string | 상의/하의/아우터 등 |
| type | string | 반팔티, 니트, 청바지 등 |
| color | string | 색상명 |
| representative_hex | string | 대표 색상 HEX |
| season_tag | string | 계절 태그 |
| pattern_type | string | 패턴 |
| availability_status | string | 보유중/세탁중/보관중/추천제외 |
| is_neutral | boolean | 무채색 여부 |
| is_denim | boolean | 데님 여부 |
| source_type | string | catalog/upload |
| catalog_item_id | string nullable | 원본 카탈로그 ID |
| created_at | datetime | 생성일 |

### 7.3 catalog_items

| 컬럼 | 타입 | 설명 |
|---|---|---|

## 8. 현재 구조의 장점과 한계

### 장점

- 의류 데이터 모델이 추천 알고리즘에 필요한 필드를 이미 포함한다.
- 카탈로그 기반 등록과 직접 업로드 등록이 같은 `ClothingItem` 구조로 통합된다.
- 무신사 이미지와 JSON 메타데이터를 자동 병합하여 색상 기반 추천에 사용할 수 있다.
- 여러 옷장을 선택해도 `catalogItemId`/`imageUrl` 기준으로 중복 추천을 방지한다.
- 기온 구간은 점수 보정이 아니라 후보 제외 필터로 작동해 추천 결과의 현실성이 높다.

### 한계

- 현재 저장소가 localStorage라서 사용자 계정 간 동기화가 불가능하다.
- 이미지 asset은 빌드 시점에 포함되므로 런타임에 폴더만 추가해서 자동 반영되지는 않는다.
- `import.meta.glob` 경로는 정적 문자열이어야 하므로 새 카테고리 추가 시 `src/data/musinsaCatalogData.ts` 수정이 필요하다.
- 저장 코디는 `itemIds` 기반이라 원본 의류 삭제 시 일부 이미지가 누락될 수 있다.
- 색상 분석 JSON의 품질에 따라 추천 품질이 달라진다.

## 9. 결론

디지털 옷장과 의류 등록 모듈은 추천 알고리즘의 입력 데이터베이스 역할을 한다. 사용자가 옷을 등록하면 단순 이미지 목록이 아니라 카테고리, 세부 타입, 대표 색상, 계절 태그, 보유 상태, 원본 카탈로그 ID가 함께 저장된다.

추천 알고리즘은 이 데이터를 기반으로 다음 순서로 코디를 생성한다.

1. 선택된 옷장의 의류만 가져온다.
2. 세탁중/추천제외 상태를 제거한다.
3. 같은 이미지 또는 같은 카탈로그 옷을 중복 제거한다.
4. 기온 구간에 맞지 않는 옷 종류를 제외한다.
5. 상의/하의/아우터/신발을 조합한다.
6. 퍼스널컬러, 날씨, 색상 조화, 보유 상태 점수로 정렬한다.
7. 사용자가 저장한 코디는 별도 `SavedOutfit`으로 관리한다.

따라서 의류 등록 데이터의 정확도, 특히 `category`, `type`, `representativeHex`, `availabilityStatus`의 품질이 추천 결과 품질을 직접 결정한다.
