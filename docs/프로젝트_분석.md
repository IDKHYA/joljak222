# 프로젝트 분석 — 옷 추천 · 옷 인식 중심 심층 리뷰

> 작성 기준: 실제 소스 코드(추측 아님)를 읽고 분석. 파일·행 번호를 함께 표기해 추적 가능하게 함.
> 분석 범위: 옷 추천 엔진(`recommendationEngine.ts`), 옷 인식 백엔드(`background_remove_api.py`, `season_predictor.py`), 카탈로그 데이터(`trainingCatalog.json`), 색상 유틸(`colorUtils.ts`), 호출부(`App.tsx`, `AnchorOutfitFinder.tsx`).
> 이 문서는 분석/리뷰 문서이며, 이 문서를 쓰는 시점에 코드는 아무것도 변경하지 않았다.

---

## 0. 한 줄 총평

색 이론·색공간(Lab/CIEDE2000)·세그멘테이션·Random Forest를 제대로 엮은, 학부 졸업작품 기준 상위권 설계다.
다만 세 가지 구조적 약점이 있다.

1. 추천 엔진의 매직넘버가 데이터로 검증되지 않은 손튜닝 값이다.
2. 카탈로그 데이터의 한계가 알고리즘의 일부(패턴 페널티 등)를 사실상 죽은 코드로 만든다.
3. 성능 캡(후보 제한)이 앵커 경로에만 있고 메인 추천 경로에는 없다.

인식 파트는 파이프라인 설계는 좋으나, 2단 추론(SegFormer → RF)의 비용 대비 효용이 데이터로 입증돼야 한다.

## 2. 옷 추천 파트 — 상세 분석

### 2-1. 치명적 문제 — 카탈로그가 알고리즘의 일부를 죽인다

검증 결과 **860개 카탈로그 전 품목의 `patternType`이 `'solid'` 하나뿐**이다(`trainingCatalog.json` 전수 확인).

- `calculatePatternPenalty` (`recommendationEngine.ts` 413~419)는 graphic / stripe / plaid 충돌을 감점하도록 짜여 있으나, 카탈로그엔 solid밖에 없어 **항상 0을 반환**한다. 사실상 죽은 코드다.
- 따라서 발표에서 "패턴 충돌 페널티"를 강조하면 "그 데이터에 패턴이 존재하긴 하나요?"라는 질문에 무너진다.
- 같은 맥락으로 `seasonTag`에 `'봄/가을'`이 존재하는데(`trainingCatalog.json`), `getWeatherScore`의 `'사계절'` +8 보너스(138행)와 `isWeatherEligible`의 하드 컷(49~50행)은 `'여름'/'겨울'`만 검사한다. `'봄/가을'` 품목은 날씨 가산·감점에서 사실상 중립으로 빠진다. 의도인지 누락인지 불명확하다.

**개선 방향.**
- 카탈로그 생성 단계에서 패턴을 실제로 분류해 넣거나(인식 파트와 연결), 패턴 페널티의 적용 범위를 "수동 등록 의류에만 적용"으로 솔직히 한정한다.
- 발표 안전성 측면에서는 후자(범위 한정 + 향후 계획으로 제시)가 낫다.

### 2-2. 성능 캡이 한쪽 경로에만 있다 (잠재적 프리징)

- 앵커 경로 `buildAnchoredRecommendations`는 `MAX_PRIMARY_CANDIDATES=60`, `MAX_OUTER_CANDIDATES=20`으로 후보를 잘라 안전하다(635~636, 660~662).
- 그러나 **메인 경로 `buildRecommendations`(602~626)에는 후보 캡이 없다.** `tops × bottoms × outerOptions`를 전수 순회한 뒤 마지막에 `slice(0, 60)`만 한다(625행).
- 카탈로그 규모는 상의 582 · 하의 75 · 아우터 30이다. 선택된 옷장이 카탈로그에 가까워지면 약 582 × 75 × 31 ≈ **135만 조합**이 만들어지고, 각 조합마다 `scoreOutfit`이 다수의 HSL/Lab 변환·deltaE를 돌린다. 이 O(n³) 연산이 **메인 스레드(useMemo, `App.tsx` 207행)에서** 통째로 실행된다.
- 즉 앵커에서 고쳤던 프리징이 메인 경로에 그대로 잠재한다. 지금은 사용자가 옷장을 작게 쓰기 때문에 안 터질 뿐, 구조적 폭탄이다.

**개선 방향.**
- 앵커와 동일하게 `personalFitScore` 상위로 후보를 먼저 자른다.
- 그래도 무거우면 Web Worker로 옮겨 메인 스레드를 막지 않는다.

### 2-3. 점수 산식의 검증 공백 (가장 솔직해야 할 부분)

- 가중치(38/22/28/12), 조화 기본점(보색 88 / 유사 82 / 단색 80 / 삼색 76 / tension 55), `IDEAL_LIGHTNESS_GAP=0.22` 등은 **색 이론으로 "방향"은 정당하나, 숫자 자체는 손튜닝**이다. A/B 테스트나 사용자 평가 데이터가 없다.
- `scoreOutfit`은 `score`(분해 가중합)와 `baseScore`(레거시 `legacyHarmonyScore` 기반, `calculateBaseScore` 542~552)를 **둘 다 계산**하고 `qualityAdjustment = score - baseScore`를 노출한다(566~597). 두 조화 점수 체계(정밀 `calculateHarmonyScore` vs 단일 룩업 `HARMONY_BASE_SCORES`)가 공존해 인지·연산 중복이 있다.

**개선 방향.**
- 발표에서는 숫자를 "벤치마크 결과"가 아니라 "색 이론 기반 사전값(prior) + 향후 사용자 피드백으로 학습"이라고 정확히 포지셔닝한다.
- `baseScore`/`qualityAdjustment`가 UI에서 실제로 쓰이지 않으면 제거를 검토한다(죽은 복잡도).

### 2-4. 기타 정합성 이슈

- `getHueBucket`은 채도 `< 0.08`을 무채색으로 처리하는데(237행), 같은 함수 주석에는 "0.15 미만"이라고 적혀 있어(233행) **주석과 코드가 불일치**한다. 사소하나 발표 자료 신뢰도에 영향.
- `scoreOutfit`의 `mode` 인자는 `title` 문자열 조립에만 쓰인다(582행). 토글 UI는 이미 제거했으나 인자는 남아 있어 "mode가 점수에 영향을 준다"는 오해 소지가 있다. 주석으로 명시돼 있어 위험도는 낮다.

## 3. 옷 인식 파트 — 상세 분석

### 3-1. 구조 요약

`/api/clothing/extract`(`background_remove_api.py` 302~377) 흐름은 다음과 같다.

1. SegFormer-b3-fashion으로 픽셀별 의류 카테고리 분류(`run_fashion_segmentation` 178~202).
2. 요청 부위 라벨만 마스크로 남기고 정리(`clean_boolean_mask`).
3. `season_predictor.predict`에서 47개 계열의 ROI·색·형상·품질 피처 추출 후 Random Forest(`season_model.joblib`, 2.6MB)로 계절 분류.
4. RF가 없으면 규칙 기반 폴백(`get_rule_season` 260~308).

### 3-2. 좋은 점

- 룰 폴백을 항상 두어 모델 파일이 없어도 동작한다(401~403).
- 피처가 단순 색이 아니라 소매 비율·노출 면적·bbox·edge_density 등 "계절 단서"를 직접 인코딩한다(`extract_color_stats`, `get_rule_season` 283~302). 도메인 통찰이 들어가 있다.
- `silhouette_score`로 KMeans의 k를 자동 선택한다(`get_dominant_colors` 234~245). 색 개수를 고정하지 않은 점이 영리하다.

### 3-3. 문제점 / 의심되는 가성비

1. **2단 추론의 비용 대비 효용.** SegFormer-b3(무거움) + scipy/cv2/sklearn KMeans + RF를 동원해 결과는 고작 `봄/가을 · 여름 · 겨울` **3-클래스**를 맞힌다. CPU에서 이미지당 수 초가 든다. 게다가 RF 피처의 상당수가 SegFormer 라벨 비율에서 파생되므로, "SegFormer 라벨 → 규칙 매핑"만으로도 상당 부분이 커버된다. 실제로 `get_rule_season`이 이미 그 일을 한다.
2. **89.74% 정확도의 출처·검증셋이 불투명.** human_label 분포·테스트셋 크기·클래스 불균형이 문서에 없으면, 발표의 "규칙 대비 4.7배" 주장이 흔들린다.
3. **계절 3분류와 12-시즌 퍼스널컬러가 의미상 분리돼 있다.** 옷의 '봄/가을' 계절 태그는 기온 의미이고, 퍼스널컬러 '봄'은 색 톤 의미인데 라벨이 겹쳐 보여 혼동 위험이 있다.
4. **`extract_color_stats`가 픽셀마다 `colorsys.rgb_to_hsv`를 파이썬 루프로 돈다**(166행). numpy/cv2로 벡터화하면 수십 배 빨라진다. 명백한 가성비 손실.
5. 해상도 기준이 경로마다 다르다. SegFormer `max_side=640`(178행), rembg `MAX_IMAGE_SIDE=1600`(30행). 일관성이 부족하다.

**개선 방향 / 대체안.**
- 계절 RF의 실제 기여도를 검증셋으로 확인한다. 룰 폴백 대비 정확도 향상이 미미하면 RF를 제거하고 SegFormer 라벨 + 규칙으로 대체한다. 그러면 모델 2.6MB·sklearn/pandas 의존성·추론 1단계를 제거할 수 있다.
- 단, 발표용으로 "RF로 정확도 +X%p"를 입증할 데이터가 있으면 RF 유지가 정당하다.
- 색 통계 추출을 numpy 벡터화로 교체한다.

## 4. 기술적으로 더 가성비 있게 대체 가능한 파트

| 현재 | 비용 | 대체안 | 기대 효과 |
|---|---|---|---|
| 계절 RandomForest (다수 피처) | 모델 2.6MB + sklearn/pandas + 추론 1단 | SegFormer 라벨 + 규칙(이미 존재) | 의존성·추론 단계 제거, 유지보수 비용↓ (정확도 데이터로 먼저 검증) |
| `extract_color_stats` 픽셀 파이썬 루프 | O(픽셀) 느림 | numpy/cv2 벡터 변환 | 수십 배 속도 |
| 메인 추천 O(n³) 전수 조합 | 메인스레드 프리징 위험 | 앵커처럼 후보 캡 + (선택)Web Worker | 안정성 |
| rembg(u2netp) + SegFormer 2개 모델 병존 | RAM·로딩 2배 | 용도 정리 후 한쪽 제거 검토 | 메모리·코드 절감 |
| `baseScore` / `qualityAdjustment` 이중 조화 계산 | 연산·인지 비용 | 미사용 시 제거 | 단순화 |

## 5. 권장 우선순위 (분석 결론)

1. **(발표 신뢰도)** 패턴 페널티·문항 수 변경·89.74% 등 "데이터로 뒷받침되지 않는 주장"을 솔직히 범위 한정하고 검증셋을 명시한다. 지난 발표 실패의 재발 방지가 목적이다.
2. **(잠재 버그)** 메인 `buildRecommendations`에도 후보 캡을 둔다. 언젠가 반드시 터지는 구조다.
3. **(가성비)** 계절 RF의 효용을 데이터로 확인하고, 미미하면 룰로 대체한다. 색 통계는 벡터화한다.
4. **(정합성)** `getHueBucket` 주석/코드 불일치, `'봄/가을'` 날씨 처리 누락을 점검한다.

## 부록 — 확인된 수치 사실(Fact)

- 카탈로그 총 860벌. 상의 582 · 하의 75 · 아우터 30 · 신발 47 · 액세서리 126.
- 카탈로그 `patternType`은 전부 `'solid'` (단일 값).
- 카탈로그 `seasonTag` 종류: `봄/가을`, `여름`, `겨울`, `사계절`.
- 전 품목에 `dominantColors`(상위 색 + ratio)가 채워져 있음 → `scoreItemForPersonalColor`의 다색 가중 평균(102~116)이 카탈로그에서 실제로 동작함.
- 계절 모델 아티팩트: `server/artifacts/season_model.joblib` 2.6MB, `feature_columns.json`, `label_mapping.json` 존재.
- 추천 가중치 `SCORE_WEIGHTS`: 퍼스널 0.38 / 날씨 0.22 / 조화 0.28 / 안정성 0.12.
- 조화 분해 가중치 `HARMONY_COMPONENT_WEIGHTS`: hue 0.5 / 명도 0.3 / 채도 0.2.
