# 이미지 저장소와 옷장 데이터 아키텍처 설계

## 목적

옷 카탈로그와 사용자 옷장 데이터가 많아졌을 때 이미지 용량, 추천 성능, 사용자 업로드, 누끼 처리, 카탈로그 확장을 안정적으로 다루기 위한 서비스 구조를 정리한다.

현재 프로젝트는 로컬 이미지 파일과 JSON 또는 TypeScript 데이터를 매칭해서 옷 데이터를 보여주는 구조다. MVP 단계에서는 빠르게 만들 수 있지만, 아우터, 상의, 하의, 신발, 시계, 목도리, 모자, 장갑, 가방처럼 카테고리가 늘어나고 사용자가 직접 옷을 계속 추가하면 저장소와 데이터베이스 구조를 분리해야 한다.

## 핵심 방향

이미지 파일과 옷 메타데이터를 같은 곳에 저장하지 않는다.

이미지 파일은 Object Storage에 저장하고, 데이터베이스에는 이미지 경로와 옷의 구조화된 정보만 저장한다.

```text
이미지 파일
-> Object Storage

옷 메타데이터
-> Postgres 같은 관계형 데이터베이스

색상, 재질, 패턴, 카테고리 분석 결과
-> DB 컬럼 또는 JSONB

추천용 벡터와 임베딩
-> Vector DB 또는 Postgres pgvector

썸네일, 누끼 이미지, 미리보기 이미지
-> Object Storage의 파생 파일

누끼와 분석 작업
-> 백그라운드 큐와 Worker
```

## 현재 구조의 한계

현재 구조는 로컬 폴더 안에 이미지와 데이터가 같이 존재한다.

```text
로컬 이미지 폴더
+ 로컬 JSON 또는 TypeScript 데이터
+ React에서 직접 import 또는 경로 참조
```

이 방식은 데이터가 적을 때는 단순하고 빠르다. 하지만 장기적으로는 다음 문제가 생긴다.

- 이미지가 많아질수록 프로젝트 용량이 커진다.
- Git 저장소에 이미지가 섞이면 저장소가 무거워진다.
- 사용자 업로드 파일을 앱 코드와 분리하기 어렵다.
- 원본, 누끼, 썸네일, 분석 결과의 버전 관리가 꼬인다.
- 모바일에서 전체 카탈로그를 들고 있으면 느려진다.
- 추천과 검색 로직이 커질수록 프론트 코드가 비대해진다.

## 미래 서비스 구조

추천하는 구조는 이미지 저장, 메타데이터 저장, 분석 처리, 추천 계산을 분리하는 것이다.

```text
Frontend
  |
  | API 요청
  v
Backend API
  |
  +-- Postgres
  |     - users
  |     - wardrobes
  |     - clothing_items
  |     - clothing_assets
  |     - clothing_analysis
  |     - outfits
  |     - dailylook_layouts
  |
  +-- Object Storage
  |     - original
  |     - thumbnail
  |     - cutout
  |     - preview
  |
  +-- Queue
        |
        v
      ML Worker
        - resize
        - background removal
        - segmentation
        - color extraction
        - material extraction
        - pattern extraction
        - embedding
```

## 이미지 저장 전략

원본 이미지는 저장한다. 나중에 누끼 모델이나 색상 분석 모델이 좋아졌을 때 다시 처리해야 하기 때문이다.

누끼 결과도 저장한다. 매번 모델을 다시 돌리면 비용과 시간이 너무 크기 때문이다.

썸네일도 저장한다. 목록 화면과 모바일 성능을 위해 필요하다.

권장 파생 이미지는 다음과 같다.

```text
original: 원본 보관용
thumbnail: 목록용 256px
card: 카드용 512px
cutout: 데일리룩 배치용 PNG 또는 WebP
preview: 결과 공유용
```

앱에서는 원본을 바로 쓰지 않는다. 목록에서는 `thumbnail`, 카드에서는 `card`, 데일리룩 만들기에서는 `cutout`을 사용한다.

## 카탈로그와 사용자 옷장 분리

카탈로그 옷과 사용자 옷은 같은 형태의 데이터로 다룰 수 있지만, 소유권과 공개 범위는 반드시 분리해야 한다.

카탈로그 옷은 공통 데이터다.

```text
owner_type: catalog
visibility: public
```

사용자 옷은 개인 데이터다.

```text
owner_type: user
visibility: private
```

카탈로그 이미지는 public bucket과 CDN으로 제공할 수 있다. 사용자 업로드 이미지는 private bucket에 저장하고 signed URL로 접근시키는 구조가 안전하다.

## 업로드와 분석 흐름

사용자가 옷을 추가할 때 모델 분석을 즉시 전부 끝내려고 하면 앱이 느려진다. 업로드와 분석은 분리해야 한다.

```text
사용자 업로드
-> 원본 이미지를 Object Storage에 저장
-> DB에 clothing_item 생성
-> DB에 clothing_asset original 생성
-> 분석 작업을 Queue에 등록
-> 목록에는 processing 상태로 표시
-> Worker가 누끼, 색상, 재질, 패턴을 분석
-> cutout, thumbnail, card 이미지를 저장
-> clothing_analysis와 clothing_assets를 갱신
```

초기에는 FastAPI 내부 BackgroundTasks로 시작할 수 있다. 데이터가 많아지면 Redis Queue, Celery, BullMQ 같은 큐로 분리하고, 운영 단계에서는 별도 Worker 서버나 Cloud Run Jobs, AWS Lambda, ECS Worker 같은 구조로 확장한다.

## 추천 시스템 구조

추천은 프론트에서 전체 데이터를 가져와 필터링하는 방식으로 가면 안 된다.

추천 계산은 서버에서 수행하고, 프론트는 결과만 받는 구조가 좋다.

```text
1차 필터
카테고리, 계절, 날씨, 성별, 스타일, 보유 여부

2차 점수
퍼컬 적합도, 색 조합, 재질, 패턴, 온도 점수

3차 다양화
같은 색과 같은 카테고리 반복 방지

4차 자동 배치
outer, upper, lower, shoes, accessory 슬롯별 레이아웃 생성
```

나중에 데이터가 많아지면 벡터 검색도 고려한다. 비슷한 무드의 옷, 특정 아우터와 어울리는 하의, 사용자가 자주 고른 룩과 비슷한 조합은 임베딩 기반 검색이 유리하다.

## 데일리룩 자동 배치 구조

데일리룩 자동 배치는 사용자가 만들기 전에 추천 결과 단계에서 기본 생성되어야 한다.

추천된 조합이 저장될 때 다음 값도 함께 만든다.

```text
outfit
-> outfit_items
-> dailylook_layouts
```

`dailylook_layouts.layers_json`은 다음 구조를 가진다.

```text
itemId
assetId
category
slot
x
y
scale
rotation
zIndex
visible
```

사용자는 이 기본 배치를 보고, 필요할 때만 데일리룩 만들기 화면에서 수정한다.

## 단계별 전환 계획

1단계는 현재 코드에서 이미지 참조 구조를 추상화하는 것이다.

```text
imageUrl
thumbnailUrl
cutoutUrl
assetIds
sourceType
ownerType
categoryGroup
slot
```

2단계는 로컬 JSON 데이터를 DB로 옮길 수 있게 구조를 정리하는 것이다.

3단계는 Object Storage를 붙이는 것이다.

4단계는 누끼와 분석 작업을 백그라운드 Worker로 분리하는 것이다.

5단계는 추천 계산을 서버 중심으로 옮기는 것이다.

## 현실적인 기술 선택

초기 확장 단계에서는 Supabase가 현실적이다.

```text
Postgres
Storage
Auth
RLS
CDN
Edge Function
pgvector 확장 가능성
```

직접 서버를 크게 만들기 전까지는 Supabase로 데이터베이스, 파일 저장소, 사용자 권한, 이미지 접근 제어를 빠르게 구성할 수 있다.

더 큰 운영 구조가 필요해지면 AWS S3, CloudFront, RDS Postgres, Redis Queue, ECS 또는 Lambda Worker로 확장할 수 있다.

## 결론

앞으로는 앱 코드 안의 이미지와 JSON을 직접 매칭하는 구조에서 벗어나야 한다.

장기 구조는 Object Storage에 이미지, DB에 메타데이터, Worker에 분석, CDN에 이미지 전달, 서버에 추천 계산을 두는 방식이 맞다.

지금 프로젝트에서 가장 먼저 해야 할 일은 DB를 바로 붙이는 것이 아니라 `ClothingItem`과 이미지 참조 구조를 Storage와 DB로 옮길 수 있게 추상화하는 것이다.
