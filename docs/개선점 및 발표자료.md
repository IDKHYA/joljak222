## 0. 한 줄 메시지 (발표 오프닝)

"personalcolorcheckpage 버전에서는 추천 점수가 코드 안에 박힌 숫자였습니다. 이번 버전은 **모든 점수 계수가 이름과 색채 이론 근거를 갖는, 설명 가능하고 테스트된 추천 엔진**으로 바뀌었습니다. 그리고 숫자를 바꾼 게 아니라 같은 숫자에 근거를 붙였다는 걸 회귀 테스트로 증명했습니다."

지난 발표에서 "이 숫자 왜 이 값이냐"에 답을 못 했던 부분을, 이번엔 근거표로 답한다.

## 2. [핵심] 옷 추천 엔진 — 수치 명분화 (Before / After)

### 2.1 구조 — 인라인 매직 넘버 → 문서화된 모듈 + 테스트

| 구분 | Before (baseline 5c11bfe) | After (현재) |
| --- | --- | --- |
| 위치 | `App.tsx` 안에 추천 로직 인라인 | `src/services/recommendationEngine.ts` 단일 모듈(696줄) |
| 점수 계수 | 코드에 박힌 숫자(`0.38`, `0.5`, `-0.2`, `6`, `-10`…) | **이름 있는 상수 28개**(조화 근거화 17개) + 근거 주석 |
| 테스트 | 추천 로직 단위 테스트 없음 | **단위 테스트 23개** (점수·분류·조합·다양성 회귀) |
| 설명 문장 | 고정 템플릿 문자열 | 계산값 기반 동적 생성 |

### 2.2 최종 점수 = 4축 가중합 (왜 이 비중인가)

```
최종 점수 = 퍼스널컬러 0.38 + 날씨 0.22 + 색상 조화 0.28 + 착용 안정성 0.12
```

- Before: `Math.round(personalScore * 0.38 + weatherScore * 0.22 + harmonyScore * 0.28 + stabilityScore * 0.12)` — 숫자가 식 안에 직접 박혀 있었다.
- After: `SCORE_WEIGHTS = { personal: 0.38, weather: 0.22, harmony: 0.28, stability: 0.12 }` 상수.
- 근거 멘트. "퍼스널컬러가 이 앱의 정체성이라 가장 큰 0.38. 색 조화는 코디 완성도의 핵심이라 0.28. 날씨는 실제 착용 가능성이라 0.22. 안정성(보유/상태)은 보정 역할이라 가장 작은 0.12."

### 2.3 색상 조화 점수 — 단순 보정 → 3요소 분해 (명분화의 중심)

Before(baseline)의 `calculateHarmonyScore`는 **기본 점수 + 시즌 대비 보정**이 전부였고, 보정값이 매직 넘버였다.

```js
// Before (baseline, App.tsx)
let score = HARMONY_BASE_SCORES[harmonyType];
if (harmonyType === 'complementary') score += preferredContrast > 0.5 ? 6 : preferredContrast < -0.2 ? -10 : 0;
if (harmonyType === 'analogous')     score += preferredContrast < -0.2 ? 6 : 0;
```

After(현재)는 조화 점수를 **hue 50% + 명도 30% + 채도 20%**로 분해하고, 모든 숫자에 이름과 근거를 붙였다.

```
조화 = hue관계(0.5) + 명도균형(0.3) + 채도균형(0.2)  → 시즌 대비 보정 → 안정성 가산 → 패턴 페널티
```

근거 멘트. "hue 관계가 조화의 1차 결정 요인이라 비중이 가장 큽니다(Itten 색상환). 명도 대비는 상하의 분리감(가독성), 채도 균형은 과한 충돌을 억제하는 미세 조정이라 그다음입니다."

### 2.4 조화 상수 근거표 (발표에서 "이 숫자 왜?"에 그대로 답하는 표)

| 상수 | 값 | 근거 |
| --- | --- | --- |
| `HARMONY_COMPONENT_WEIGHTS` | hue .5 / 명도 .3 / 채도 .2 | hue가 조화의 1차 요인, 명도는 가독성, 채도는 충돌 억제 |
| `HARMONY_BASE_SCORES` | 보색 88 / 유사 82 / 모노 80 / 삼색 76 / tension 55 | 보색·유사색이 가장 안정적, 어중간한 거리(tension)가 가장 약함 |
| 분류 임계 | ≤15 모노 / ≤45 유사 / ≤90 tension / ≤135 삼색 / >135 보색 | hue 각도 구간을 base score와 1:1 대응 |
| `IDEAL_LIGHTNESS_GAP` | 0.22 | 상하의가 구분되면서도 과하지 않은 명도차 |
| `LOW_LIGHTNESS_CONTRAST` | 8%p 미만 → -28 | 명도차 너무 작으면 한 덩어리로 뭉개짐(강한 감점) |
| `HIGH_LIGHTNESS_CONTRAST` | 55%p 초과 → -10 | 위아래가 지나치게 갈라짐(약한 감점) |
| `SATURATION_DIFF_SLOPE` | 45 | 채도 차이가 클수록 통일감 저하(선형 감점) |
| `DOUBLE_POINT_SATURATION` | 둘 다 >0.65 → -12 | 포인트가 둘이라 과한 충돌 |
| `FLAT_SATURATION` | 둘 다 <0.12 → -6 | 둘 다 무채라 밋밋 |
| `BALANCE_SCORE_FLOOR` | 45 | 색이 안 맞아도 다른 축이 살릴 수 있어 0이 아닌 45 바닥 |
| `SEASON_CONTRAST` | 고대비 0.5↑ / 저대비 -0.2↓ | 겨울 계열=고대비 선호, 여름 계열=저대비 선호 |
| 보색·유사 보정 | 보색 +6/-10, 유사 +6 | 고대비 시즌엔 보색 가산, 저대비 시즌엔 보색 감점·유사 가산 |
| 안정성 가산 | 포인트+받침 +6 / 받침만 +3 | 무채·데님이 강한 포인트색(채도 0.45↑)을 받칠 때 가장 안정 |

### 2.5 "값은 안 바꿨다"는 증명 (리팩터링 신뢰성)

매직 넘버를 상수로 바꾸면서 점수가 바뀌지 않았음을 **회귀 스냅샷 테스트**로 박제했다. 대표 케이스 조화 점수 **89 / 85 / 84**가 리팩터링 전후로 동일하다.

발표 멘트. "근거를 붙이는 작업이 결과를 흔들지 않았다는 걸 테스트로 증명했습니다. 즉 설명 가능성을 얻으면서 기존 동작은 그대로 보존했습니다."

### 2.6 설명 문장 동적화 — 고정 문구 → 계산값 기반 (LLM 미사용)

| 필드 | Before (baseline) | After (현재) |

baseline에는 추천 결과에 설명 패널·점수 분해·출처 표시가 없었다(아래 요소 모두 baseline에서 0건으로 확인).

## 3. 단일 옷 코디 추천 (baseline에 없던 신규 기능)

"이 옷 한 벌"을 기준으로 어울리는 코디를 찾아 주는 기능. baseline에는 없다(전수 추천만 있었음).

- 기준 옷을 반드시 포함하는 조합만 생성. 옷장/카탈로그/둘 다 출처 토글.
- 기존 4축 엔진(`scoreOutfit`)을 그대로 재사용해 전수 추천과 점수 산식 공유.
- **성능 명분화(수치화 강점)**. 아우터 기준 + 카탈로그면 상의 582 × 하의 75 ≈ **43,650 조합**이 렌더 중 계산돼 멈췄다. 상대 후보를 퍼스널컬러 적합도 상위로 미리 잘라(`60`/`20`) **최대 3,600 조합(약 92% 감소)**으로 묶었다. 속도와 추천 품질이 함께 올라간다.
- 다양성 버그 수정. 기준 옷이 모든 코디에 들어가 표준 다양성 제한에 먼저 걸려 결과가 3개로 고정되던 것을, 상대 옷만 제한하도록 바꿔 최대 24개로.

## 4. 데일리룩 — 처음 선보이는 개인화 기능

데일리룩은 코드로는 만들어 뒀지만 **중간 발표에서는 공개하지 않은** 기능이다. 관객 기준으로는 이번에 처음 선보이므로, 발표에서는 신규 개인화 기능으로 소개해도 된다.

추천으로 끝나지 않고, 고른 코디를 보드 위에서 자유롭게 꾸미고 텍스트까지 얹어 하나의 룩 이미지로 완성하는 기능이다.

- 저장한 코디를 플랫레이 보드에 **자동 배치**. 위치·크기·회전·앞뒤 순서 조정, 이전 배치 복원.
- **텍스트 레이어**. 문구 추가, 내용·색·크기·회전 편집, 드래그 이동·리사이즈, 표시/숨김·삭제.
- **옷 추가 picker**. 옷장·카탈로그 탭, 카테고리·시즌 필터, 퍼컬 추천순 정렬, 적합도 80↑ 표시, 일괄 추가.
- 누끼(배경 제거)가 없는 옷은 **자동 PNG 처리**, 완성 보드를 한 장의 PNG로 확정 저장.

발표 멘트. "추천으로 끝나지 않고, 고른 코디를 보드에 자동 배치하고 텍스트까지 얹어 나만의 룩 이미지로 완성합니다. 누끼 처리부터 이미지 저장까지 앱 안에서 끝납니다."

> 내부 메모(슬라이드 아님). 코드 자체는 personalcolorcheckpage baseline에도 존재한다. 다만 중간 발표에서 공개하지 않았으므로 관객에게는 처음 선보이는 기능으로 소개한다. 깃 히스토리 질문이 나오면 "구현은 이전부터, 공개는 이번이 처음"으로 답하면 정확하다.

## 5. 퍼컬 측정 설문 8→5 — 팩트체크 결과 (주의)

⚠️ **팩트체크 결론. 설문 8→5 축소는 personalcolorcheckpage baseline "이전"에 이미 끝나 있었다.** baseline(`5c11bfe`)을 직접 열어 보니 지금과 똑같은 5문항(vein_color·jewelry_reaction·vibrant_colors·contrast_preference·depth_preference)이었다. 따라서 "baseline 대비 8→5로 줄였다"는 정확한 표현이 아니다. 이번 세션에 한 일은 코드에 잘못 남아 있던 "8문항" 주석을 "5문항"으로 **정정**한 것뿐이다(Questionnaire.tsx, personalColorEngine.ts).

→ 발표에서 8→5를 쓰려면 "personalcolorcheckpage 대비 개선"이 아니라 "프로젝트 전반의 설계 결정"으로 분리해서 말해야 사실과 맞는다.

### 5.1 "정확성 유지" 근거 (측정값이 아니라 구조적 논거)

코드에 정확도 벤치마크(8문항 vs 5문항 A/B 데이터)는 없다. 그래서 "정확성 유지"는 측정 수치가 아니라 아래 구조로 설명해야 한다.

1. **5문항이 진단 4축을 모두 덮는다.** temperature(vein_color·jewelry_reaction), clarity(vibrant_colors), contrast(contrast_preference), lightness(depth_preference). `QUESTIONNAIRE_AXES = [temperature, lightness, clarity, contrast]` 전부 커버.
2. **설문은 보조(안정화) 신호다.** 실제 색 데이터는 사진 분석(skin·hair·eyes·lips의 Lab)이 담당하고, 설문은 카메라·조명 오차를 안정화하는 보조 신호다(코드 주석에 명시).
3. **융합 비율이 동적이다.** 사진과 설문은 `photoQuality` 기반 동적 비율로 융합되고, 사진/설문에 서로 다른 trait 가중치를 쓴다. 사진 품질이 좋으면 사진이 우위라, 문항 수 감소가 핵심 정확도에 주는 영향이 작다.

발표 멘트(정직 버전). "문항을 5개로 줄여도 진단 4축을 모두 덮고, 설문은 사진을 보정하는 보조 신호라 핵심 정확도는 사진이 좌우합니다. 단, 이건 측정값이 아니라 설계 근거입니다."

→ 더 강한 주장을 원하면 8문항·5문항 버전의 시즌 판정 일치율을 샘플로 비교한 수치가 필요하다(현재 없음). 필요하면 만들어 줄 수 있다.

## 8. 예상 질문 대응

- "이 가중치(38/22/28/12)는 왜 이 값인가." → 2.2의 근거 멘트. 정체성(퍼컬) > 완성도(조화) > 실용(날씨) > 보정(안정).

## 9. 정확성·기여 메모 (발표 전 점검용)

- baseline = `5c11bfe`로 확정(git merge-base로 HEAD의 조상임을 확인). `5c11bfe..HEAD` = 26 커밋이 개선분.
- 추천 엔진 모듈화·점수 공식 근거화·설명 패널은 baseline 이후 여러 커밋에 걸쳐 이뤄졌다(특정 1인의 단일 작업으로 단정하지 말 것).
- "상수 근거화(이번 세션)"는 **값을 바꾸지 않고 이름·근거만 부여**한 작업이며, 점수 불변을 테스트로 증명했다.
- 데일리룩은 baseline 코드에 이미 존재하지만 중간 발표에서 미공개 → 관객 기준 "처음 공개"로 소개 가능. 깃 히스토리 질문엔 "구현은 이전부터, 공개는 이번"으로 답한다.
- 설문 8→5 축소는 baseline(`5c11bfe`)에 이미 5문항이라 baseline 대비 변화가 아니다. 이번 세션 기여는 "8문항" 잘못된 주석을 5문항으로 정정한 것뿐이다. "정확성 유지"는 측정값이 아닌 구조적 근거(5.1).
- **의류 계절 분류 89.74%의 정확한 범위.** 이 수치는 직접 라벨링한 516장(`논문.md`)으로 학습한 RandomForest(트리 300, balanced)의 정확도이며, 분류는 **3계절(겨울·봄가을·여름) ** 문제다(`server/artifacts/label_mapping.json`로 확인). 12계절이 아니다. 데이터셋이 작고 클래스가 3개뿐이라, 절대치보다 "룰 베이스라인 19.2% 대비 약 4.7배"라는 **상대 향상폭**으로 말하는 편이 방어에 유리하다. 학습/검증 분할 방식은 자료에 명시돼 있지 않으니, 질문이 나오면 "보유 라벨셋 기준"이라고만 답하고 분할 수치를 지어내지 말 것.
- **특징 수는 95개가 맞다.** 모델 ground truth(`feature_columns.json`)가 95개다. `논문.md`의 "97개"는 오기이므로 발표 자료(docx/슬라이드)의 "95개"를 기준으로 말할 것. (논문 본문은 별도 정정 필요 — 발표 범위 밖.)
- **패턴 페널티의 실제 적용 범위.** `calculatePatternPenalty`는 구현돼 있으나, 현재 카탈로그 860벌의 `patternType`이 전부 `solid`라 카탈로그 추천에서는 발동하지 않는다. 실제 발동은 패턴이 있는 수동 등록 의류에 한정된다. 발표에서는 "조화 엔진에 패턴 충돌 감점이 설계돼 있고, 패턴 자동 태깅(LLM 비전)은 향후 과제"로 범위를 분명히 할 것. "지금 카탈로그 추천이 패턴까지 본다"고 말하면 사실과 다르다.
