# 컴퓨터와 모바일 환경에 따른 퍼스널 컬러 측정 문제 보고서

## 1. 개요

이 문서는 `통합_퍼컬_옷장` 프로젝트의 퍼스널 컬러 측정 기능에서 발생한 컴퓨터 웹 환경과 모바일 웹 환경의 차이, 실제 증상, 원인 분석, 그리고 적용한 해결 내용을 정리한 보고서입니다.

문제가 발생한 기능은 `퍼스널 컬러 분석` 화면의 사진 분석 모듈입니다. 이 모듈은 브라우저 카메라를 켜고, MediaPipe Face Landmarker로 얼굴과 주요 랜드마크를 인식한 뒤, 흰 종이 기준 영역을 이용해 조명 색을 보정하고 얼굴 색상 샘플을 분석합니다.

컴퓨터 웹 환경에서는 비교적 안정적으로 얼굴 인식, 랜드마크 표시, 자동 촬영 카운트다운이 동작했습니다. 반면 모바일 웹 환경에서는 카메라 비율, 얼굴 가이드 표시, 얼굴 인식 지속성, 자동 촬영 카운트다운에서 문제가 반복적으로 발생했습니다.

## 2. 발생한 주요 문제

### 2.1 모바일에서 카메라 화면 비율이 이상하게 표시됨

모바일 브라우저에서 카메라 화면을 열면 사용자가 기대한 세로 셀카 화면이 아니라, 가로로 긴 카메라 스트림이 세로 UI 안에 들어가는 문제가 있었습니다.

초기에는 화면을 꽉 채우기 위해 `object-fit: cover` 방식으로 처리했지만, 이 경우 가로 스트림을 세로 프레임에 맞추면서 화면이 확대되어 얼굴 일부가 잘리는 문제가 생겼습니다. 특히 이마, 눈, 얼굴 하단이 프레임 밖으로 나가 사용자가 얼굴을 제대로 맞추기 어려웠습니다.

반대로 `object-fit: contain`을 사용하면 얼굴은 덜 잘리지만, 비디오 주변에 검은 배경 또는 여백이 보이는 문제가 있었습니다. 사용자는 모바일에서 사진을 찍는 순간만큼은 얼굴이 정확히 잘 보여야 한다고 요구했기 때문에, 단순히 화면을 꽉 채우는 것보다 얼굴이 잘리지 않는 방향이 더 중요했습니다.

### 2.2 모바일 웹에서 네이티브 카메라 앱처럼 동작하지 않음

사용자는 “셀카 카메라를 켜면 원래 세로 비율이고, 초광각처럼 넓게 나오면 되는 것 아닌가”라고 지적했습니다. 그러나 모바일 웹의 `getUserMedia` API는 네이티브 카메라 앱과 다르게 동작합니다.

네이티브 카메라 앱은 기기 제조사가 제공하는 카메라 앱이기 때문에 전면 카메라, 초광각 카메라, 줌, 보정, 회전, 미리보기 비율 등을 직접 제어할 수 있습니다. 반면 모바일 브라우저는 보안과 표준 API 제한 때문에 특정 렌즈를 확정적으로 선택하거나, 초광각 렌즈를 강제로 켜거나, 네이티브 카메라 앱과 동일한 미리보기 비율을 보장할 수 없습니다.

즉, 모바일 브라우저에서는 전면 카메라를 요청해도 실제 스트림이 `640x480`, `1280x720` 같은 가로 비율로 들어올 수 있습니다. 이 상태에서 UI만 세로로 만들면 화면 확대 또는 여백 문제가 발생합니다.

### 2.3 모바일에서 얼굴 인식 마크가 제대로 보이지 않음

모바일 화면에서는 얼굴을 어디에 맞춰야 하는지 보여주는 가이드가 충분히 명확하지 않았고, 실제 얼굴이 인식되고 있는지 확인하기 어려웠습니다.

컴퓨터에서는 화면 크기가 넓고 카메라 비율도 상대적으로 안정적이어서 얼굴 박스와 랜드마크가 더 잘 보였습니다. 하지만 모바일에서는 프레임 크기가 작고, 비디오 비율이 자주 다르며, 하단 내비게이션과 안내 문구까지 함께 표시되어 얼굴 인식 상태를 직관적으로 확인하기 어려웠습니다.

### 2.4 모바일에서 얼굴 인식이 처음에는 되다가 멈춤

가장 중요한 문제는 모바일에서 얼굴 인식이 처음에는 되는 것처럼 보이다가, 카운트다운이 `3`, `2` 정도까지 내려간 뒤 더 이상 인식이 지속되지 않는 현상이었습니다.

사용자 관찰 기준으로는 다음과 같은 흐름이었습니다.

1. 모바일 브라우저에서 퍼스널 컬러 측정 화면 진입
2. 카메라 권한 허용
3. 처음에는 얼굴이 인식되고 카운트다운이 시작됨
4. `3`, `2` 정도까지 진행됨
5. 이후 얼굴/눈동자 인식 마크가 멈추거나 사라짐
6. 자동 촬영까지 안정적으로 이어지지 않음

이 현상은 단순 UI 문제가 아니라 얼굴 인식 루프와 MediaPipe 추론 방식이 모바일 환경에서 불안정하게 동작한 문제로 판단했습니다.

## 3. 원인 분석

### 3.1 모바일 브라우저 카메라 스트림의 제약

모바일 웹에서 카메라를 켤 때 사용하는 API는 `navigator.mediaDevices.getUserMedia()`입니다. 이 API는 다음과 같은 제약이 있습니다.

- 전면 카메라를 요청할 수는 있지만, 반드시 원하는 렌즈가 선택된다는 보장이 없습니다.
- 초광각 카메라를 명시적으로 선택하는 표준 방식이 안정적으로 제공되지 않습니다.
- 줌 값을 제어하는 `zoom` capability는 일부 브라우저와 기기에서만 지원됩니다.
- 세로 비율을 요청해도 실제 스트림은 가로 비율로 들어올 수 있습니다.
- 기기와 브라우저 조합에 따라 같은 코드도 다르게 동작합니다.

따라서 모바일에서는 “세로로 꽉 찬 화면”, “확대 없음”, “얼굴 잘림 없음”을 동시에 항상 만족시키기 어렵습니다. 이 프로젝트에서는 얼굴 분석 정확도가 중요하기 때문에 화면을 꽉 채우는 것보다 얼굴이 잘리지 않는 것을 우선했습니다.

### 3.2 기존 얼굴 인식 루프가 모바일에 너무 무거움

기존 구현은 컴퓨터 웹 환경을 기준으로 `requestAnimationFrame` 루프를 사용했습니다. 이 방식은 브라우저 화면 갱신 주기에 맞춰 매우 자주 얼굴 인식을 시도합니다.

컴퓨터에서는 CPU/GPU 성능이 충분하고 브라우저 WebGL 환경도 안정적이기 때문에 문제가 적었습니다. 하지만 모바일에서는 다음 문제가 발생할 수 있습니다.

- MediaPipe 얼굴 랜드마크 추론이 모바일 GPU에서 지연됨
- 한 번의 추론이 오래 걸리는 동안 다음 프레임 처리가 밀림
- 브라우저가 카메라, WebGL, JavaScript 실행을 동시에 처리하면서 성능이 떨어짐
- 특정 순간 `detectInFlightRef`가 `true`로 고정되면 다음 감지가 전부 막힘

기존 코드는 `detectInFlightRef`를 사용해 동시에 여러 얼굴 인식 요청이 겹치지 않게 막고 있었습니다. 원래는 좋은 보호 장치지만, 모바일에서 한 번의 감지 요청이 오래 걸리거나 멈추면 이 값이 계속 `true`로 남아 다음 감지가 진행되지 않는 문제가 생길 수 있었습니다.

### 3.3 MediaPipe GPU delegate가 모바일에서 불안정할 수 있음

기존 `faceLandmarker.ts`는 MediaPipe Face Landmarker를 만들 때 GPU delegate를 먼저 사용하고, 실패하면 CPU로 fallback하는 구조였습니다.

컴퓨터에서는 이 방식이 성능상 유리합니다. 그러나 모바일 Chrome에서는 GPU delegate가 처음 몇 프레임은 동작하다가 이후 지연되거나 불안정해지는 경우가 있습니다. 특히 카메라 스트림, WebGL, 화면 렌더링이 동시에 동작할 때 모바일 GPU 자원이 부족해질 수 있습니다.

이 때문에 모바일에서는 최고 프레임률보다 안정성이 중요하다고 판단했고, 모바일 환경에서는 CPU delegate를 우선 사용하도록 변경했습니다.

### 3.4 자동 촬영 카운트다운이 감지 실패에 너무 민감함

기존 자동 촬영 구조는 얼굴이 감지되면 3초 카운트다운을 시작하고, 얼굴이 감지되지 않으면 즉시 타이머를 취소하는 방식에 가까웠습니다.

컴퓨터에서는 얼굴 감지가 안정적이므로 문제가 적습니다. 하지만 모바일에서는 한두 프레임 정도 얼굴 감지가 흔들릴 수 있습니다. 이때 카운트다운을 바로 취소하면 사용자는 “인식이 되다가 멈췄다”고 느끼게 됩니다.

따라서 모바일에서는 한 번의 감지 실패로 즉시 카운트다운을 끊지 않고, 최근 일정 시간 안에 얼굴이 감지되었다면 얼굴이 유지되고 있다고 판단하는 완충 로직이 필요했습니다.

## 4. 적용한 해결책

### 4.1 모바일 카메라 요청 방식 개선

모바일에서는 전면 카메라와 세로 비율을 우선 요청하도록 변경했습니다.

적용 방향은 다음과 같습니다.

- `facingMode: { ideal: 'user' }`로 전면 카메라 요청
- `width: 720`, `height: 1280`, `aspectRatio: 9 / 16`을 우선 요청
- 권한 획득 후 `enumerateDevices()`로 카메라 목록 확인
- 전면 카메라 후보를 우선 선택
- 가능한 경우 `front`, `selfie`, `전면`, `wide`, `ultra`, `0.5`, `초광각` 같은 라벨을 기준으로 더 적합한 카메라 후보를 선택
- 기기가 `zoom` capability를 지원하면 최소 줌으로 설정

단, 모바일 웹에서는 카메라 라벨이 항상 제공되지 않거나, 초광각 렌즈가 브라우저에 노출되지 않을 수 있습니다. 따라서 이 처리는 “가능한 기기에서는 더 나은 렌즈를 선택하는 보강”이며, 모든 기기에서 초광각을 강제하는 기능은 아닙니다.

### 4.2 모바일 프리뷰 표시 방식 조정

모바일에서 비디오를 세로 프레임에 억지로 꽉 채우면 얼굴이 잘릴 수 있으므로, 모바일 프리뷰는 얼굴이 잘리는 것을 줄이기 위해 `contain` 기반으로 조정했습니다.

이전에는 검은 배경이 보였지만, 모바일에서는 카메라 프레임과 비디오 배경을 흰색으로 맞춰 검은 여백이 튀지 않도록 변경했습니다.

즉, “무조건 꽉 채우는 화면”보다 “얼굴이 덜 잘리고 분석에 필요한 정보가 보이는 화면”을 우선한 것입니다.

### 4.3 MediaPipe Face Landmarker 인스턴스 분리

`src/services/faceLandmarker.ts`를 변경해 Face Landmarker 인스턴스를 GPU와 CPU용으로 분리했습니다.

기존 구조는 하나의 전역 promise만 사용했습니다. 이 경우 한 번 GPU 기반 인스턴스가 만들어지면 모바일에서도 같은 인스턴스를 계속 사용할 수 있었습니다.

변경 후에는 다음처럼 동작합니다.

- PC 환경: GPU delegate 우선 생성
- GPU 생성 실패 시 CPU fallback
- 모바일 환경: CPU delegate 우선 생성
- GPU/CPU 인스턴스 promise를 별도로 보관

이렇게 해서 모바일에서는 GPU/WebGL 불안정성을 피하고, PC에서는 기존 성능 이점을 유지할 수 있게 했습니다.

### 4.4 모바일 전용 얼굴 인식 루프 추가

`src/components/PhotoAnalyzer.tsx`의 얼굴 인식 루프를 PC와 모바일로 분리했습니다.

PC에서는 기존처럼 `requestAnimationFrame` 기반 루프를 유지합니다. 컴퓨터에서는 빠른 갱신과 부드러운 landmark 표시가 장점이기 때문입니다.

모바일에서는 `setTimeout` 기반 저빈도 루프를 사용합니다.

- 모바일 감지 간격: 약 320ms
- 매 프레임 추론하지 않고 일정 간격으로만 얼굴 인식 수행
- 모바일 CPU/GPU 부담 감소
- 브라우저가 카메라와 추론을 동시에 처리하다가 멈추는 가능성 감소

이 방식은 PC보다 landmark 갱신 속도는 낮을 수 있지만, 모바일에서 인식이 끊기지 않고 지속되는 것이 더 중요하기 때문에 안정성을 우선했습니다.

### 4.5 `detectInFlightRef` 고착 방지

모바일에서 얼굴 인식 요청이 멈추면 `detectInFlightRef`가 계속 `true`로 남아 다음 감지가 진행되지 않을 수 있습니다.

이를 막기 위해 다음 값을 추가했습니다.

- `detectStartedAtRef`: 감지 요청 시작 시각 기록
- `MOBILE_DETECT_STUCK_MS`: 모바일 감지 stuck 기준 시간

모바일에서 감지 요청이 1.5초 이상 끝나지 않으면 stuck 상태로 보고 `detectInFlightRef`를 강제로 해제합니다. 이렇게 하면 다음 감지 루프가 다시 실행될 수 있습니다.

### 4.6 얼굴 감지 유지 완충 로직 추가

한 프레임에서 얼굴이 감지되지 않았다고 바로 실패로 처리하지 않도록 `lastFaceSeenAtRef`를 추가했습니다.

동작 방식은 다음과 같습니다.

- 얼굴이 감지되면 현재 시간을 `lastFaceSeenAtRef`에 저장
- 이후 한 번 얼굴이 감지되지 않아도 최근 약 1.1초 안에 얼굴이 감지된 기록이 있으면 `얼굴 유지 중`으로 판단
- 이 시간 안에는 카운트다운을 유지
- 일정 시간 이상 얼굴이 계속 감지되지 않을 때만 카운트다운 취소

이 변경으로 모바일에서 흔히 발생하는 순간적인 추론 누락 때문에 카운트다운이 중간에 끊기는 문제가 줄어듭니다.

### 4.7 모바일 인식 상태 문구 개선

사용자가 현재 무슨 상태인지 알 수 있도록 상태 문구를 분리했습니다.

현재 사용되는 상태는 다음과 같습니다.

- `얼굴을 가이드 안에 맞춰주세요`
- `얼굴 인식 중`
- `얼굴 유지 중`

## 7. 남아 있는 한계

### 7.1 모바일 웹은 네이티브 카메라 앱과 동일하게 제어할 수 없음

모바일 웹에서는 초광각 렌즈를 항상 선택하거나, 전면 카메라의 실제 스트림 비율을 완전히 통제할 수 없습니다. 브라우저와 기기가 허용하는 범위 안에서만 제어할 수 있습니다.

따라서 특정 기기에서는 여전히 기대한 것보다 좁게 보이거나, 완벽한 세로 스트림이 나오지 않을 수 있습니다.

### 7.2 CPU delegate는 안정적이지만 GPU보다 느릴 수 있음

모바일에서 CPU delegate를 우선 사용하도록 바꾼 이유는 안정성입니다. 다만 CPU 방식은 GPU보다 얼굴 랜드마크 갱신 속도가 느릴 수 있습니다.

이 프로젝트에서는 모바일에서 빠른 프레임률보다 “촬영까지 안정적으로 완료되는 것”이 더 중요하기 때문에 CPU 우선 방식을 선택했습니다.

### 7.3 실제 모바일 기기별 추가 테스트 필요

모바일 브라우저 카메라와 MediaPipe 동작은 기기별 차이가 큽니다. Android Chrome, Samsung Internet, iOS Safari 등에서 다르게 동작할 수 있습니다.

현재 수정은 가장 문제가 컸던 모바일 Chrome 계열을 기준으로 안정성을 높인 것입니다. 이후 실제 사용 기기에서 다음 항목을 확인해야 합니다.

- 얼굴 가이드가 항상 보이는지
- 얼굴 박스와 샘플 영역이 계속 갱신되는지
- 카운트다운이 중간에 멈추지 않는지
- 촬영 후 분석 결과가 정상적으로 저장되는지
- 카메라 비율이 사용자가 얼굴을 맞추기 적절한지

## 8. 결론

이번 문제는 단순히 CSS 비율이나 버튼 위치 문제가 아니라, 모바일 웹 환경에서 카메라 스트림과 MediaPipe 얼굴 인식 추론이 컴퓨터 웹과 다르게 동작하면서 발생한 복합 문제였습니다.

컴퓨터에서는 고성능 환경과 안정적인 GPU/WebGL 덕분에 기존 `requestAnimationFrame + GPU delegate` 구조가 잘 동작했습니다. 하지만 모바일에서는 같은 구조가 너무 무겁고, 한 번 감지가 지연되면 다음 감지가 막히는 문제가 발생했습니다.

해결 방향은 모바일에서 네이티브 카메라 앱과 동일한 동작을 억지로 재현하는 것이 아니라, 모바일 웹의 제약을 인정하고 다음처럼 안정성을 높이는 것이었습니다.

- 모바일은 CPU delegate 우선
- 모바일은 저빈도 감지 루프 사용
- 감지 stuck 복구 처리
- 순간적인 얼굴 미감지에 대한 완충 처리
- 카운트다운 유지 안정화
- 얼굴이 잘리지 않는 프리뷰 우선
- 상태 문구와 가이드 표시 개선

이 변경으로 모바일 웹에서도 얼굴 인식이 `3 → 2`에서 멈추는 현상이 줄어들고, 자동 촬영까지 더 안정적으로 이어지도록 개선했습니다.
