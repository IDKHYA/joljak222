# 단일 옷 추천 기능 — 「이 옷으로 코디 찾기」 상세 보고서

작성일: 2026-05-26
대상 코드: `src/features/recommendation/AnchorOutfitFinder.tsx`, `src/services/recommendationEngine.ts`(`buildAnchoredRecommendations`, `scoreOutfit`), `src/App.tsx`, `src/features/wardrobe/WardrobeSection.tsx`, `src/index.css`

---

## 1. 한 줄 요약

옷 한 벌을 **기준(anchor)**으로 고정하면, 사용자의 퍼스널컬러와 색상 조화를 기준으로 어울리는 나머지 옷을 골라 **완성된 코디 조합**을 점수순으로 보여 주는 기능이다.

기존 추천이 "내 옷장 전체에서 좋은 코디 N개"를 뽑는 **전수 탐색**이라면, 이 기능은 "**이 옷**에 뭘 같이 입을까?"라는 **국소 탐색**이다.

---

## 2. 왜 만들었나 (문제 정의)

기존 추천 대시보드는 옷장 전체를 훑어 상위 코디를 보여 준다. 하지만 실제 사용 맥락은 종종 다음과 같다.

- "오늘 이 셔츠를 입고 싶은데, 뭘 매치하지?"
- "이 아우터에 어울리는 상·하의 조합이 궁금하다."
- "이 바지를 샀는데, 내 옷장 옷이랑 어울리나? 아니면 뭘 더 사야 하나?"

즉 **고정하고 싶은 한 벌이 이미 정해져 있는** 상황이다. 전수 추천은 이 한 벌을 보장하지 못하므로, 기준 옷을 강제로 포함하는 별도 탐색이 필요했다.

---

## 3. 사용자 흐름 (UX Flow)

1. **진입.** 옷장 상세 화면에서 보유한 옷(상의·하의·아우터)의 카드에 있는 `✨ 이 옷으로 코디` 버튼을 누른다.
2. **오버레이.** 화면 위에 코디 찾기 모달이 뜬다. 상단에 기준 옷의 썸네일·종류·역할("이 상의에 어울리는 코디를 찾았어요")이 표시된다.
3. **출처 선택.** `옷장 + 카탈로그 / 내 옷장 / 카탈로그` 세 가지 토글로 상대 옷을 어디서 가져올지 고른다. 기본값은 `옷장 + 카탈로그`다.
4. **결과.** 기준 옷을 반드시 포함한 코디들이 점수순으로 그리드에 깔린다. 각 코디 카드의 아이템에는 출처 뱃지(`보유`/`카탈로그`)가 붙어, 내가 가진 옷인지 사야 할 옷인지 구분된다.
5. **저장.** 마음에 드는 코디는 카드의 저장 버튼으로 보관함에 담는다.
6. **닫기.** 우상단 X로 오버레이를 닫으면 원래 화면으로 돌아온다.

> 진입점은 **옷장 상세의 보유 아이템에만** 있다. "옷장 만들기(카탈로그 선택)" 흐름에는 일부러 두지 않았다. 자세한 이유는 8.2절 참고.

## 5. 핵심 알고리즘 — `buildAnchoredRecommendations`

```
buildAnchoredRecommendations(anchor, pool, result, mode = '데일리')
```

### 5.1 날씨 축 고정

```ts
const band: RecommendationWeatherBand = '상관없음';
```

이 기능의 질문은 "이 옷에 **색이** 어울리는 옷"이다. 날씨를 섞으면 "이 반팔에 어울리는 패딩"처럼 색은 맞지만 계절이 어긋난 조합에 점수가 깎여 의도가 흐려진다. 그래서 날씨를 `상관없음`으로 고정해 **퍼스널컬러 + 색상 조화**에 집중한다.

### 5.2 후보 풀 정리

```ts
const usable = dedupeRecommendationItems(
  pool.filter((item) =>
    item.availabilityStatus !== '추천제외' &&
    item.availabilityStatus !== '세탁중' &&
    itemUniqueKey(item) !== anchorKey,   // 기준 옷 자신은 상대 후보에서 제외
  ),
);
```

- `추천제외`·`세탁중` 상태 아이템은 후보에서 뺀다.
- **기준 옷 자기 자신**을 상대 풀에서 제외한다. 안 그러면 "이 옷 + 이 옷"이라는 자가 조합이 나온다.
- `dedupeRecommendationItems`로 같은 카탈로그 상품 중복을 제거한다.

### 5.3 기준 카테고리별 조합 규칙

| 기준 옷 | 조합 방식 |
| --- | --- |
| 상의 | `하의 × (아우터 또는 없음)` |
| 하의 | `상의 × (아우터 또는 없음)` |
| 아우터 | `상의 × 하의` (아우터는 기준 옷이 담당) |

```ts
const outerOptions = [undefined, ...usable.filter((item) => item.category === '아우터')];

if (anchor.category === '상의') {
  bottoms.forEach((bottom) => outerOptions.forEach((outer) => pushOutfit([outer, anchor, bottom])));
} else if (anchor.category === '하의') {
  tops.forEach((top) => outerOptions.forEach((outer) => pushOutfit([outer, top, anchor])));
} else if (anchor.category === '아우터') {
  tops.forEach((top) => bottoms.forEach((bottom) => pushOutfit([anchor, top, bottom])));
}
```

`outerOptions` 맨 앞의 `undefined`는 "아우터 없는 2피스 코디"를 의미한다. 즉 상의·하의 기준일 때는 아우터를 한 겹 더 얹는 3피스와, 얹지 않는 2피스를 **둘 다** 생성한다.

`pushOutfit`은 아이템 2벌 미만이거나 이미 본 조합이면 건너뛴다(`outfitUniqueKey`로 중복 차단).

### 5.4 신발·액세서리 기준 — v1 미지원

기준 옷이 신발·액세서리면 위 `if` 분기에 걸리지 않아 `outfits`가 비고, 빈 배열이 반환된다. 그래서 **진입 버튼 자체를 상의·하의·아우터에만** 노출한다(`ANCHORABLE_CATEGORIES`). v1 범위 결정이다(11절 참고).

## 6. 점수 산식 — `scoreOutfit` (재사용 엔진)

코디 한 벌의 최종 점수는 4축 가중 합이다.

```
최종 점수 = 퍼스널컬러 38% + 날씨 22% + 색상 조화 28% + 착용 안정성 12%
```

| 축 | 계산 | 비중 |
| --- | --- | --- |
| 퍼스널컬러 | 아이템별 `personalFitScore` 평균. 의류 대표색과 시즌 팔레트의 CIEDE2000 Lab 거리 기반. | 0.38 |
| 날씨 | `상관없음` 고정이라 상태값(보유/세탁/보관) 위주로 평가. | 0.22 |
| 색상 조화 | `calculateHarmonyScore`. Itten 색상환 hue 각도 + 명도 균형 + 채도 균형 + 시즌 대비 선호. | 0.28 |
| 착용 안정성 | 모든 아이템이 `보유중`이면 92, 아니면 68. | 0.12 |

색상 조화 점수 내부는 다시 `hue 50% + 명도 30% + 채도 20%`로 나뉘며, 모든 계수는 이번 작업에서 **이름 있는 상수로 근거화**돼 있다(예: `IDEAL_LIGHTNESS_GAP = 0.22`, `HARMONY_COMPONENT_WEIGHTS`). 이 부분은 별도 문서 `개선사항.md`의 Tier 2 항목 참고.

또한 `scoreOutfit`은 코디마다 **설명 문장**도 합성한다.

- `reason`(헤드라인): 예 `"트루 윈터에 어울리는 선명한 대비 (적합 88)"`.
- `explanationBullets`(상세): 퍼스널핏·조화·날씨 각각의 근거 문장.
- 아이템별 `fitReason`: 예 `"트루 윈터 팔레트의 파랑 계열과 잘 맞음 (적합 87)"`.

이 문장들은 LLM이 아니라 **이미 계산된 값에서 규칙적으로 생성**된다. 졸업작품의 "엔진이 직접 계산한다"는 논지를 지키기 위한 결정이다.

## 7. 다양성 제어 — 가장 중요한 버그와 수정

### 7.1 증상

초기 구현은 출처에 옷이 많아도 결과가 **항상 3개 안팎**으로 고정됐다. 조합이 정적이고 변별력이 없었다.

### 7.2 원인

표준 다양성 함수 `diversifyRecommendations(outfits, maxPerItem = 3)`는 "같은 아이템이 3번 넘게 등장하면 자른다"로 동작한다.

그런데 **기준 옷은 정의상 모든 코디에 들어간다.** 따라서 기준 옷이 3번 등장한 시점(=코디 3개)에서 한도에 걸려, 4번째 코디부터 전부 잘려 나갔다. 결과가 3개로 고정된 이유다.

### 7.3 수정 — 상대 옷만 카운트

기준 옷을 등장 횟수 제한에서 **제외**하고, 기준 옷을 뺀 **상대 옷(counterpart)**만 최대 2개 코디까지 등장하도록 바꿨다.

```ts
const MAX_APPEARANCE_PER_COUNTERPART = 2;
const ANCHORED_RESULT_LIMIT = 24;

const counterpartCount = new Map<string, number>();
return outfits
  .sort((a, b) => b.score - a.score)
  .filter((outfit) => {
    const counterparts = outfit.items.filter((item) => itemUniqueKey(item) !== anchorKey);
    if (counterparts.some((item) => (counterpartCount.get(itemUniqueKey(item)) ?? 0) >= MAX_APPEARANCE_PER_COUNTERPART)) return false;
    counterparts.forEach((item) => counterpartCount.set(itemUniqueKey(item), (counterpartCount.get(itemUniqueKey(item)) ?? 0) + 1));
    return true;
  })
  .slice(0, ANCHORED_RESULT_LIMIT);
```

효과는 두 가지다.

- **같은 하의·아우터가 반복되지 않는다.** 한 상대 옷은 최대 2개 코디에만 등장하므로, 비슷한 조합이 화면을 도배하지 않는다.
- **다양한 조합이 골고루 나온다.** 점수순 상위에서 서로 다른 상대 옷을 우선 채우다가 24개에서 멈춘다.

검증: 하의 12벌만 있는 상황에서 상의 기준으로 돌리면 **12개 결과**가 모두 나온다(회귀 테스트로 고정). 실제 브라우저에서는 옷장+카탈로그 기준 **24개**가 정상 출력됐다.

## 8. 설계 결정 및 트레이드오프

### 8.1 출처를 토글로 — "옷장/카탈로그/둘 다"

상대 옷을 어디서 가져올지 사용자가 직접 고른다(기본 `둘 다`).

- **내 옷장**: 지금 가진 옷만으로 코디를 짠다. "오늘 뭐 입지" 맥락.
- **카탈로그**: 살 옷까지 포함해 본다. "이 옷에 뭘 더 사면 좋을까" 맥락.
- **둘 다**: 가진 옷 + 살 옷을 섞는다.

`App.tsx`는 카탈로그 전체(약 860벌)를 미리 퍼스널컬러 점수까지 매겨 `anchorCatalogPool`(memo)로 보관한다. 퍼스널컬러 결과가 바뀔 때만 재계산되므로, 토글을 눌러도 즉시 반응한다.

### 8.2 진입점을 옷장 상세로 한정 — 피드백 반영

초기엔 "옷장 만들기(카탈로그 선택)" 화면의 아이템에도 진입 칩을 달았다. 하지만 **옷을 고르는 중에 추천이 끼어드는 게 어색**하다는 피드백을 받아 제거했다.

현재 진입은 **내가 가진 옷**에만 둔다. 즉 "내가 가진 이 옷에 뭘 입을까"라는 맥락에서만 시작한다. 카탈로그는 진입점에서는 빠졌지만 **상대 옷 출처(토글)로 계속 참여**하므로, "가진 옷에 살 옷을 더해 보는" 흐름은 그대로 가능하다.

### 8.3 오버레이로 마운트 — 라우팅 변경 회피

별도 페이지/라우트를 만들지 않고 조건부 오버레이(`{anchorItem && <AnchorOutfitFinder .../>}`)로 띄운다. 기존 라우팅·뒤로가기 동작을 건드리지 않고, 어느 화면에서든 기준 옷만 정하면 뜨도록 한 surgical한 선택이다.

### 8.4 `OutfitCard` 재사용 + 출처 뱃지

결과 카드는 추천 대시보드의 `OutfitCard`를 그대로 쓰되, 선택적 `catalogIds` prop을 추가했다. 이 Set에 포함된 아이템은 `카탈로그`, 아니면 `보유` 뱃지를 단다. 컴포넌트 중복 없이 출처만 구분한다.

## 10. 테스트 / 검증

`recommendationEngine.test.ts`에 기준 옷 관련 테스트를 추가했다(전체 23개 통과).

- 기준 옷이 모든 결과 코디에 **항상 포함**되는가.
- **자기중복**(기준 옷 + 기준 옷)이 생기지 않는가.
- 빈 풀이면 빈 배열을 반환하는가.
- 신발 기준이면 빈 배열인가(v1 미지원 확인).
- 아우터 기준이면 상의·하의를 채워 3피스를 만드는가.
- 하의 12벌 → **12개 결과**가 나오는가(다양성 회귀 고정).

추가로 `tsc --noEmit` 린트 통과, `vite build` 성공, 브라우저 실사용 검증(결과 24개, 출처 뱃지 정상, 출처 토글 전환 시 결과 수 변동 확인)을 마쳤다.

## 11. 한계 및 향후 과제

- **신발·액세서리 기준 미지원.** v1은 상의·하의·아우터만 기준이 된다. 신발을 기준으로 "이 신발에 맞는 옷"을 찾으려면 조합 규칙과 조화 산식(현재 상의×하의 중심)을 확장해야 한다.
- **카탈로그 아이템을 기준으로** 삼는 흐름은 아직 없다. 지금은 카탈로그가 상대 풀로만 들어간다. 카탈로그 브라우징 화면에서 직접 기준을 잡는 별도 진입점은 추후 과제다.
- **색 버킷 단위 다양성**은 아직 없다. 현재 다양성은 "같은 상대 옷 2회 제한"까지다. "비슷한 색 계열 조합이 몰리지 않게" 하는 버킷 단위 분산은 다음 단계다.
- **패턴·소재 정밀도.** 색 중심 평가라 패턴·소재 조합의 미묘함은 약하다. Tier 3에서 LLM 비전으로 패턴·소재 메타데이터를 보강하면 입력 품질이 올라간다(데이터 입력 용도 한정, 점수 계산은 엔진 유지).

## 12. 한눈 요약

- **무엇**: 옷 한 벌을 기준으로 어울리는 코디를 찾아 주는 기능.
- **어떻게**: 기준 옷을 모든 조합에 강제 포함 → 기존 4축 엔진(`scoreOutfit`)으로 점수화 → 상대 옷만 카운트하는 다양성 제어로 최대 24개 노출.
- **어디서**: 옷장 상세의 보유 아이템(상의·하의·아우터) `이 옷으로 코디` 버튼.
- **상대 옷 출처**: 옷장 / 카탈로그 / 둘 다 (토글, 기본 둘 다).
- **핵심 결정**: 엔진 재사용, 날씨 고정(상관없음), 진입점 한정, 다양성 버그 수정.
