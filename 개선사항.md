# 개선사항.

작성일. 2026-05-24.
대상. Fitly 퍼스널컬러 기반 AI 옷장 및 코디 추천 시스템.
범위. 중간 발표 이후 추천 알고리즘, 추천 근거 UI, 코드 구조, 테스트 기반을 정리한 최신 개선 내역.

---

## 1. 개선 방향 요약.

중간 발표 시점의 핵심 기능은 퍼스널컬러 진단, 옷장 등록, 날씨 연동, 코디 추천이 각각 동작하는 것이었다. 이후 개선의 방향은 기능을 무작정 추가하는 것이 아니라, 추천 결과가 왜 나왔는지 설명 가능하게 만들고, 개별 의류 점수 중심의 추천을 코디 단위 평가로 확장하는 데 두었다.

현재 추천 엔진은 다음 네 축으로 최종 점수를 만든다.

| 축 | 비중 | 의미 |
| --- | ---: | --- |
| 퍼스널컬러 적합도 | 38% | 의류 색상이 사용자의 12계절 팔레트와 얼마나 가까운지 평가한다. |
| 날씨 적합도 | 22% | 현재 기온 구간과 의류 계절 태그, 착용 상태를 반영한다. |
| 색상 조화도 | 28% | 상의와 하의의 색상 관계, 명도 대비, 채도 균형을 평가한다. |
| 착용 안정성 | 12% | 보유중 여부와 추천 가능 상태를 반영한다. |

발표용 핵심 문장은 다음과 같다.

> 기존 추천은 의류별 적합도를 합산하는 방식에 가까웠지만, 개선 후에는 색차 거리의 의미를 구간화하고 상하의의 색상, 명도, 채도 관계를 함께 반영해 코디 단위 완성도를 평가하도록 확장했다.

## 2. 코드 구조 개선.

### 2.1 App.tsx 모듈 분리.

기존 `App.tsx`는 화면 전환, 타입 정의, 추천 알고리즘, 옷장 상태, 데일리룩 편집 로직이 한 파일에 섞여 있었다. 발표와 유지보수에서 특정 기능의 위치를 설명하기 어려웠고, 추천 로직을 테스트하기도 어려웠다.

현재는 다음과 같이 역할을 분리했다.

| 파일 | 역할 |
| --- | --- |
| `src/wardrobeTypes.ts` | 옷장, 의류, 추천 결과, 저장 코디 타입 정의. |
| `src/wardrobeConstants.ts` | 의류 카테고리, 날씨 구간, 추천 모드, 표시 상수. |
| `src/services/recommendationEngine.ts` | 추천 점수 계산, 색상 조화 계산, 코디 조합 생성. |
| `src/services/dailyLook.ts` | 데일리룩 캔버스 배치 상태 생성. |
| `src/services/clothingDisplay.ts` | 의류 이미지와 색상 표시 보조 함수. |
| `src/components/common.tsx` | 공통 표시 컴포넌트. |

이 개선으로 추천 알고리즘을 발표할 때 `recommendationEngine.ts` 하나를 중심으로 설명할 수 있게 되었다.

### 2.2 추천 엔진 테스트 추가.

`vitest` 기반 단위 테스트를 추가해 추천 엔진의 핵심 계산을 확인한다.

검증 대상은 다음과 같다.

- 점수 등급 경계값.
- Itten 색상환 기반 조화 유형 분류.
- 패턴 충돌 감점.
- 퍼스널컬러 미진단 상태 처리.
- 추천 후보 생성과 상태 필터링.
- 추천 근거 데이터 생성.
- 색상 조합 필터 분류.
- Delta E 구간형 점수 함수.
- 명도 대비와 고채도 포인트 충돌 보정.

## 3. 추천 근거 UI 추가.

추천 결과가 단순히 점수와 의류 이미지만 보여주면 사용자는 왜 이 조합이 나왔는지 알기 어렵다. 이를 보완하기 위해 추천 카드 안에 작은 근거 패널을 추가했다.

추천 카드에는 다음 정보가 표시된다.

- 퍼스널컬러, 날씨, 조화, 안정성 점수의 기여도.
- 퍼스널컬러 평균 점수와 가장 적합도가 높은 의류.
- 회피색 근접 감점 여부.
- 상하의 색상 조화 유형과 색상 각도 차이.
- 날씨 구간에서의 착용 가능성.

이 UI의 목적은 과장된 시각화가 아니라, 사용자가 추천 결과를 보고 바로 납득할 수 있게 하는 것이다.

예시 설명 방향은 다음과 같다.

> 트루 오텀 기준 평균 78점입니다. 셔츠의 색상 적합도가 가장 높습니다. 회피색에 가까운 색이 1개 있어 감점이 반영됐습니다.

이 설명은 새 모델이 생성한 문장이 아니라, 추천 엔진이 실제로 계산한 값을 바탕으로 만들어진다.

## 4. 색상 조합 필터 개선.

추천 결과 상단의 색상 조합 필터는 처음에는 `무채색 × 파랑`처럼 hue 버킷을 기준으로 묶었다. 이 방식은 단순하지만 카키, 올리브, 차콜, 네이비처럼 실제 의류에서 자주 나오는 중간색을 어색하게 분류하는 문제가 있었다.

현재까지 적용한 개선은 다음과 같다.

- 무채색 판정 기준을 완화했다.
- 채도가 매우 낮은 색만 무채색으로 보낸다.
- 카키나 올리브처럼 색상 방향이 있는 저채도 색은 hue 기준 색상군으로 분류한다.
- 필터 pill의 색상 점은 실제 대표색을 보여준다.

이후 개선 방향은 고정 색상군 분류가 아니라, Lab 색공간에서 가까운 색 조합끼리 묶는 방식이다.

예상 구조는 다음과 같다.

1. 각 추천 코디에서 상의 대표색과 하의 대표색을 가져온다.
2. 두 색을 Lab 좌표로 변환한다.
3. 기존 그룹 대표 조합과 Delta E 거리를 비교한다.
4. 상의와 하의가 모두 기준 거리 안에 있으면 같은 그룹으로 묶는다.
5. 가까운 그룹이 없으면 새 색상 조합 그룹을 만든다.

이 방식은 발표에서 “색상환 구간 분류에서 지각 색차 기반 그룹화로 확장할 수 있다”고 설명할 수 있다.

## 5. 퍼스널컬러 점수 공식 근거화.

### 5.1 기존 문제.

기존 의류 색상 점수는 다음과 같이 선형 감점으로 계산했다.

```text
paletteScore = 100 - DeltaE * 4.5
```

이 방식은 구현은 단순하지만 두 가지 문제가 있었다.

- `4.5`라는 상수의 근거를 설명하기 어렵다.
- 색차가 커질수록 점수가 너무 급격하게 떨어져 실제 의류 색상 평가와 어긋날 수 있다.

### 5.2 개선 방식.

현재는 Delta E 거리 구간을 기준으로 점수를 계산한다.

| Delta E 거리 | 해석 | 점수 범위 |
| ---: | --- | ---: |
| 0~5 | 거의 같은 색 또는 매우 가까운 색. | 96~100점 |
| 5~12 | 팔레트와 잘 맞는 색. | 86~96점 |
| 12~22 | 사용할 수 있지만 강한 적합색은 아님. | 70~86점 |
| 22~35 | 팔레트와 거리가 있음. | 45~70점 |
| 35 이상 | 추천 우선순위가 낮음. | 45점 이하 |

코드에서는 `scorePaletteDistance(distance)` 함수가 이 구간형 점수 곡선을 담당한다.

발표에서 방어할 수 있는 포인트는 다음과 같다.

> 색상 적합도는 단순한 선형 감점이 아니라, 색차 거리의 의미를 구간으로 해석하도록 바꿨다. 그래서 거의 같은 색, 가까운 색, 거리가 있는 색의 점수 의미가 더 명확해졌다.

### 5.3 중립색과 데님 보너스 조정.

기존에는 중립색이나 데님이면 퍼스널컬러 점수에 `+8`점을 무조건 더했다.

```text
neutral 또는 denim이면 +8.
```

이 방식은 중립색과 데님을 “항상 퍼스널컬러에 잘 맞는 색”처럼 처리한다는 문제가 있었다.

현재는 퍼스널컬러 점수의 무조건 보너스를 `+3`으로 줄였다. 중립색과 데님의 역할은 개별 색상 적합도보다 코디 전체 안정성에서 더 크게 반영한다.

해석은 다음과 같다.

> 중립색과 데님은 퍼스널컬러 자체가 무조건 좋은 색이라기보다, 강한 색을 받쳐 주고 코디를 안정시키는 요소로 재정의했다.

## 6. 색상 조화 점수 고도화.

### 6.1 기존 문제.

기존 조화 점수는 상의와 하의의 hue 각도 차이를 중심으로 계산했다.

예시는 다음과 같다.

- 0~15도. 단색 조화.
- 16~45도. 유사색 조화.
- 46~90도. 긴장 배색.
- 91~135도. 삼각 배색.
- 136~180도. 보색 대비.

이 방식은 색상 관계를 빠르게 설명하기에는 좋지만, 실제 옷 조합에서는 부족했다.

예를 들어 같은 파랑 계열이어도 연청과 네이비는 느낌이 다르고, 같은 카키라도 밝기와 탁함에 따라 완성도가 달라진다.

### 6.2 개선 방식.

현재 조화 점수는 세 가지 요소를 함께 본다.

```text
조화 점수 = 색상 관계 50% + 명도 균형 30% + 채도 균형 20% + 안정성 보정 - 패턴 감점
```

| 요소 | 반영 이유 |
| --- | --- |
| 색상 관계 | 기존 Itten 색상환 기반 조화 유형을 유지한다. |
| 명도 균형 | 상의와 하의의 밝기 차이가 너무 작으면 실루엣이 흐려질 수 있다. |
| 채도 균형 | 상의와 하의가 둘 다 고채도 포인트 색이면 과하게 튈 수 있다. |
| 안정성 보정 | 중립색이나 데님이 강한 색을 받쳐 주면 안정성 보너스를 준다. |
| 패턴 감점 | 그래픽, 체크, 스트라이프가 겹치면 시각적으로 복잡해질 수 있다. |

### 6.3 코디 단위 평가로 확장.

이 개선의 의미는 개별 옷 점수에서 코디 전체 평가로 판단 범위를 넓힌 것이다.

기존에는 좋은 상의와 좋은 하의를 평균 내는 방식에 가까웠다. 현재는 상의와 하의가 함께 놓였을 때의 관계를 본다.

반영된 기준은 다음과 같다.

- 상의와 하의의 명도 대비가 너무 약하면 감점한다.
- 상의와 하의가 둘 다 강한 포인트 색이면 감점한다.
- 중립색이나 데님은 퍼스널컬러 점수보다 조합 안정성에서 보정한다.
- 패턴이 과하게 겹치면 조화 점수에서 감점한다.
- 같은 아이템이 상위 추천에 반복되는 것은 `diversifyRecommendations`로 제한한다.

발표 문장은 다음과 같이 정리할 수 있다.

> 추천 기준을 개별 의류 적합도에서 코디 단위 완성도로 확장했다. 상의와 하의의 색상 관계뿐 아니라 명도 대비, 채도 균형, 중립색 안정화까지 함께 반영한다.

## 8. 발표에서 말할 수 있는 개선 흐름.

중간 발표 이후의 개선 흐름은 다음 순서로 설명하는 것이 좋다.

1. 추천 로직을 `recommendationEngine.ts`로 분리해 알고리즘 단위로 설명할 수 있게 했다.
2. 추천 결과 카드에 점수 분해와 근거 설명을 추가해 블랙박스 추천처럼 보이지 않게 했다.
3. 색상 필터의 단순 hue 분류 문제를 확인하고, 저채도 색상 분류 기준을 보정했다.
4. 퍼스널컬러 점수를 Delta E 구간형 점수로 바꿔 점수의 의미를 설명 가능하게 만들었다.
5. 중립색과 데님은 무조건 좋은 색이 아니라 코디 안정성을 높이는 요소로 재정의했다.
6. 색상 조화 점수에 명도와 채도 균형을 추가해 코디 단위 평가로 확장했다.
7. 색상 필터를 고정 대표색 이름이 아니라 Lab 거리상 가까운 상하의 조합으로 묶도록 바꿨다.
8. 아우터가 있는 코디는 상의와 하의뿐 아니라 아우터와 안쪽 옷의 관계까지 조화 점수에 반영했다.
9. 추천 결과에 기존 축 평균과 코디 보정 후 평균을 비교하는 진단 패널을 추가했다.

최종 발표 문장은 다음과 같이 가져갈 수 있다.

> 기존 시스템은 진단 결과, 옷장, 날씨를 연결해 추천을 제공하는 수준이었다. 개선 후에는 추천 점수의 산식을 구간화하고, 색상 조화에 명도와 채도 균형을 추가해 추천 결과가 왜 나왔는지 설명 가능한 코디 평가 구조로 발전시켰다.

## 9. 2026-05-24 추가 개선.

이번 추가 작업은 앞선 2순위와 3순위 개선안을 실제 화면과 엔진에 연결한 것이다.

### 9.1 가까운 색끼리 묶는 색상 필터.

이전 필터는 상의 대표색과 하의 대표색을 hue 버킷으로 나눠 `무채색 × 파랑`처럼 표시했다.

문제는 실제 카드 이미지의 색감과 텍스트 라벨이 어긋날 수 있다는 점이었다. 저채도 카키, 중청 데님, 어두운 네이비처럼 경계에 있는 색은 사람이 보기에는 비슷한데 라벨은 다르게 나올 수 있었다.

개선 후에는 상의와 하의의 대표색을 각각 Lab 색공간으로 변환하고, Delta E 거리가 가까운 조합끼리 같은 그룹으로 묶는다.

```text
같은 그룹 조건 = 상의 대표색 거리 <= 22 그리고 하의 대표색 거리 <= 22
```

화면에는 `비슷한 색 조합 1`처럼 중립적인 이름을 쓰고, 실제 대표색 스와치를 함께 보여준다. 그래서 라벨이 색을 단정하기보다, 사용자가 보이는 색으로 판단할 수 있게 했다.

발표 문장은 다음과 같이 정리할 수 있다.

> 색상 필터를 고정 색상명 분류에서 실제 색상 거리 기반 그룹으로 바꿔, 화면의 옷 색감과 필터 정보가 어긋나는 문제를 줄였다.

### 9.2 아우터 포함 조화 점수.

기존 조화 점수는 상의와 하의 관계를 중심으로 계산했다.

하지만 실제 코디에서는 아우터가 들어가면 바깥 레이어가 전체 인상을 크게 바꾼다. 예를 들어 상의와 하의가 안정적이어도 아우터가 너무 강한 포인트 색이면 전체 코디는 어색해질 수 있다.

개선 후에는 아우터가 있을 때 다음 관계를 함께 본다.

| 관계 | 반영 비중 |
| --- | --- |
| 상의와 하의 | 50% |
| 아우터와 상의 | 30% |
| 아우터와 하의 | 20% |

각 관계는 색상 각도, 명도 균형, 채도 균형을 함께 계산한다.

발표 문장은 다음과 같이 정리할 수 있다.

> 아우터가 포함된 추천에서는 안쪽 상하의만 보지 않고, 바깥 레이어와의 색상 관계까지 계산해 코디 전체 완성도를 평가하도록 확장했다.

# 2026-05-24 추가 — 추천 설명 문장 동적화 (Tier 1).

## 2. 작업 1 — `fitReason` 내용 강화.

```text
트루 윈터 팔레트 기준 87점
```

이 문장은 어떤 색이 왜 맞는지를 말하지 못했다. 개선 후에는 매칭된 대표 색계열과 회피색 근접 여부까지 합성한다.

```text
트루 윈터 팔레트의 파랑 계열과 잘 맞음 (적합 87)
트루 윈터 팔레트에서 빨강 계열은 회피색에 가까워 감점 (적합 61)
```

- 색계열 라벨은 기존 `getHueBucket` + `HUE_BUCKET_KO`를 재사용한다. 새 매핑을 만들지 않았다.


```text
트루 윈터에 어울리는 선명한 대비 (적합 88) · 23~27도
```

- 조화 유형 라벨은 기존 `HARMONY_TITLE_KO`를 재사용한다.
- 측정 결과가 없으면 시즌 문구를 빼고 조화 유형 위주로 표시한다.

# 2026-05-25 추가 — 조화 점수 상수 근거화 (Tier 2).

`추천알고리듬 고도화 작전.md`의 Tier 2(수치화·유기화) 중 색상 조화 점수 부분을 진행한 작업이다. 목표 #4 "모든 점수가 근거로 추적되게"와 작전 문서가 꼽은 약점 #2 "헤드라인 숫자가 임의 상수로 굴러간다"를 정조준했다.

## 1. 진행 전 확인한 데이터 사실 (방향이 바뀐 지점).

원래 Tier 2 후보 중 하나는 "조화 점수도 퍼컬 축(38%)처럼 `dominantColors`를 쓰게 해 일관성을 맞춘다"였다. 그런데 코드를 확인하니 전제가 깨졌다.

`src/services/clothingMeta.ts`에서 대표색은 이렇게 만들어진다.

```text
representativeHex = dominantColors[0].hex   (clothingMeta.ts:125, 130)
```

즉 대표색은 이미 1순위 주요색과 같은 값이다. 그래서 조화 점수가 "대표색 대신 `dominantColors[0]`을 쓰게" 바꿔도 결과는 똑같다(no-op). 실제로 효과를 내려면 보조색(`[1]`, `[2]`)까지 가중 평균해야 하는데, 보조색은 대부분 같은 색의 그림자·하이라이트라 솔리드 옷에선 의미가 거의 없고 패턴 옷에서만 신호가 생긴다. 게다가 패턴 판정(`patternType`)은 아직 텍스트 정규식 추측이라(작전 문서 Tier 3 대상) 그 위에 점수 로직을 더 쌓는 것은 시기상조라고 판단했다.

결론은 다음과 같다.

> 조화-보조색 확장은 작전 문서가 가정한 것보다 가치가 작고(패턴 옷에 한정) 토대가 흔들린다. 패턴 실측 데이터(Tier 3) 확보 후가 더 맞다.

그래서 Tier 2의 방향을 "조화의 `dominantColors` 적용"에서 "조화 점수 안의 임의 상수 근거화"로 바꿨다.

## 2. 선택지 비교와 결정.

세 가지를 두고 골랐다.

| 선택지 | 내용 | 위험 | 발표 가치 |
| --- | --- | --- | --- |
| 상수 근거화 (선택) | 매직 넘버를 이름과 근거로 분리, 값은 유지 | 낮음 (점수 불변) | 높음. "그 숫자 왜 그래요?"를 정면 방어 |
| 패턴 보조색 조화 | 조화가 보조색까지 가중 평균 | 중간 (점수 분포 이동) | 낮음. 효과가 패턴 옷에만 몰리고 토대가 추측 |
| 스킵 | Tier 3 이후로 미룸 | 없음 | 없음 (이번 회차) |

발표에서 가장 큰 리스크는 점수 산식의 임의 상수를 추궁당하는 것이라, **값을 바꾸지 않고 모든 계수를 추적 가능하게 만드는 쪽**을 골랐다. 점수 분포를 흔들지 않으니 시연 결과도 그대로다.

`src/services/recommendationEngine.ts`의 색상 조화 계산부에 흩어져 있던 숫자를 이름과 근거 주석을 가진 상수로 끌어냈다. **값은 전부 기존과 동일하다.** 바뀐 것은 가독성과 근거 추적성뿐이다.

### 3.1 hue 관계 기본 점수 (`HARMONY_BASE_SCORES`).

Itten 색상 이론의 배색 선호 서열을 0~100 척도에 배치한 값이라는 근거를 주석으로 명시했다. 50을 "겨우 쓸 만함" 기준선으로 본다.

| 조화 유형 | 각도 구간 | 점수 | 근거 |
| --- | --- | ---: | --- |
| complementary | 136~180° | 88 | 보색. 가장 강하고 세련된 고전적 대비라 최고점 |
| analogous | 16~45° | 82 | 인접색. 통일감 있고 자연스러워 높게 평가 |
| monochromatic | 0~15° | 80 | 톤온톤. 안전하지만 다소 밋밋 |
| triadic | 91~135° | 76 | 삼각 배색. 균형은 잡히나 다소 번잡해 중상위 |
| tension | 46~90° | 55 | Itten이 부조화로 본 어중간한 거리. 가장 낮은 기준선 |

무채색이 끼면 hue 각도가 의미를 잃으므로 별도 중립 기준 점수 `NEUTRAL_PAIR_HUE_SCORE = 85`를 적용한다. 이 값은 공유 맵에 넣지 않았다. `calculateBaseScore`의 레거시 경로(`?? 75`)와 동작이 달라지면 안 되기 때문이다.

### 3.2 조화 합산 가중치 (`HARMONY_COMPONENT_WEIGHTS`).

```text
쌍 조화 점수 = hue 관계 × 0.5 + 명도 균형 × 0.3 + 채도 균형 × 0.2
```

| 요소 | 비중 | 근거 |
| --- | ---: | --- |
| hue 관계 | 0.5 | 색상 관계가 조화의 1차 결정 요인 (Itten 색상환 기반) |
| 명도 균형 | 0.3 | 상하의 분리감(가독성)을 좌우 |
| 채도 균형 | 0.2 | 과한 충돌 억제용 미세 조정 |

### 3.3 명도 균형 계수.

| 상수 | 값 | 근거 |
| --- | ---: | --- |
| `IDEAL_LIGHTNESS_GAP` | 0.22 | 약 22%p 명도차가 상하의를 구분하면서도 과하지 않은 대비 |
| `LIGHTNESS_GAP_PENALTY_SLOPE` | 120 | 이상값에서 멀어질수록 깎는 기울기 |
| `LOW_LIGHTNESS_CONTRAST` | 0.08 / 28 | 명도차 8%p 미만은 한 덩어리로 뭉개져 보여 강한 감점 |
| `HIGH_LIGHTNESS_CONTRAST` | 0.55 / 10 | 명도차 55%p 초과는 지나치게 갈려 보여 약한 감점 |

### 3.4 채도 균형 계수.

| 상수 | 값 | 근거 |
| --- | ---: | --- |
| `SATURATION_DIFF_SLOPE` | 45 | 채도 차이가 클수록 통일감 하락 (차이 1.0당 45점) |
| `DOUBLE_POINT_SATURATION` | 0.65 / 12 | 둘 다 고채도면 포인트가 둘이라 과하게 충돌 |
| `FLAT_SATURATION` | 0.12 / 6 | 둘 다 거의 무채면 밋밋해 소폭 감점 |
| `BALANCE_SCORE_FLOOR` | 45 | 색이 안 맞아도 다른 축이 살릴 수 있어 0이 아닌 45를 바닥 |

### 3.5 시즌 대비 연결 계수.

퍼컬 시즌의 대비 선호(`contrast` 특질)와 코디 배색을 잇는 부분이다. 퍼컬 축과 조화 축이 대화하는 유일한 지점이라 근거를 분명히 남겼다.

| 상수 | 값 | 근거 |
| --- | ---: | --- |
| `SEASON_CONTRAST.highPreference` | 0.5 | 이 이상이면 고대비 선호 시즌(겨울 계열) |
| `SEASON_CONTRAST.lowPreference` | -0.2 | 이 이하면 저대비 선호 시즌(여름 계열) |
| `COMPLEMENTARY_BONUS_FOR_HIGH_CONTRAST` | +6 | 고대비 시즌은 보색 조합에 가산 |
| `COMPLEMENTARY_PENALTY_FOR_LOW_CONTRAST` | -10 | 저대비 시즌은 보색 조합에 감점 |
| `ANALOGOUS_BONUS_FOR_LOW_CONTRAST` | +6 | 저대비 시즌은 유사색 조합에 가산 |

### 3.6 코디 안정성 가산 계수.

| 상수 | 값 | 근거 |
| --- | ---: | --- |
| `POINT_COLOR_SATURATION` | 0.45 | 이 이상이면 강한 포인트 색으로 간주 |
| `STABILITY_BONUS_WITH_POINT` | 6 | 포인트 색 + 받침색(무채/데님) 공존 시 가장 안정적 |
| `STABILITY_BONUS_BASE` | 3 | 받침색만 있을 시 |

## 4. 점수 불변을 어떻게 증명했나 (회귀 방법).

값을 안 바꿨다는 주장을 말로만 두지 않고 테스트로 박제했다.

1. 리팩토링 **전에** 대표 색 조합 3개의 절대 조화 점수를 `toMatchInlineSnapshot`으로 캡처했다. 현재 코드 기준 값은 네이비×레드 = 89, 화이트×데님 = 85, 그레이 모노 = 84로 자동 기록됐다.
2. 그 다음 상수 분리 리팩토링을 했다.
3. 스냅샷 갱신 없이(`-u` 미사용) 다시 돌렸고, 세 값이 그대로 89/85/84로 통과했다. 즉 리팩토링이 점수를 1점도 바꾸지 않았음이 증명됐다.

이 절대값 스냅샷은 앞으로 누가 계수를 건드리면 바로 실패로 잡아 주는 회귀 방지선이 된다.

## 6. 의미와 다음.

이번 작업으로 색상 조화 점수의 모든 계수가 이름과 근거를 갖게 됐다. 발표에서 "조화 점수 88은 어떻게 나왔나요?"라는 질문에 "보색 기본 88점 × 0.5 + 명도 균형 × 0.3 + 채도 균형 × 0.2, 각 계수의 근거는 이것"이라고 끝까지 추적해 답할 수 있다.

미뤄 둔 것은 조화 축의 보조색(`dominantColors`) 반영이다. 이건 패턴 옷에서만 효과가 있고 패턴 데이터의 신뢰도가 받쳐 줘야 하므로, 무료 비전 LLM으로 패턴·소재를 실측하는 Tier 3 이후로 둔다.

# 2026-05-25 추가 — 이 옷으로 코디 찾기 기능.

옷 한 벌을 기준으로 잡고, 옷장과 카탈로그에서 잘 어울리는 코디를 찾아 주는 기능이다. 기존 추천이 옷장 전체를 전수 조합해 보여 주는 흐름이었다면, 이건 "이 옷에 뭘 같이 입지"라는 실제 옷 입기 질문에 답한다.

## 1. 동작 방식.

- 옷장 상세에서 보유한 옷(상의·하의·아우터)의 `이 옷으로 코디` 버튼을 누르면 기준 옷이 정해진다.
- 그 옷을 반드시 포함하는 코디만 만들어 점수순으로 보여 준다. 기준이 상의면 하의(+아우터)를, 하의면 상의(+아우터)를, 아우터면 상의×하의를 채운다.
- 상대 옷을 어디서 가져올지는 화면 토글로 고른다. 옷장 / 카탈로그 / 둘 다, 기본은 둘 다.
- 결과 카드의 각 아이템에는 보유/카탈로그 출처 뱃지가 붙어, 가진 옷으로 입을지 살 옷을 더할지 구분된다.
- 날씨 축은 '상관없음'으로 고정해 퍼스널컬러와 색상 조화에 집중한다.

## 2. 엔진 — 점수 산식 공유.

기존 추천 엔진을 새로 만들지 않고 재사용했다. `buildRecommendations`의 점수 합성부를 `scoreOutfit` 헬퍼로 추출해, 전수 추천과 기준 옷 추천이 같은 산식(퍼컬 38 + 날씨 22 + 조화 28 + 안정 12, reason·fitReason 포함)을 쓰게 했다.

추출이 점수를 바꾸지 않았음은 조화 회귀 스냅샷(89/85/84)이 그대로 통과하는 것으로 확인했다.

`buildAnchoredRecommendations(anchor, pool, result)`가 기준 옷을 반드시 포함하는 조합만 생성한다. 신발·액세서리 기준은 v1에서 빈 배열을 반환하고, 진입 버튼도 상의·하의·아우터에만 노출한다.

## 3. 1차 구현의 두 문제와 수정.

처음 구현을 화면에서 돌려 보고 두 가지 문제를 발견해 고쳤다.

### 3.1 옷장 만들기 흐름에 추천이 끼는 문제.

처음엔 카탈로그 선택 화면(옷장 만들기)의 카드마다 코디 칩을 달았다. 그런데 이 화면은 여러 벌을 골라 옷장에 담는 곳이라, 옷장을 구성하는 중에 추천 기능이 끼면 의도가 섞여 어색했다.

그래서 카탈로그 선택 화면의 칩을 제거했다. 기준 옷 진입은 옷장 상세의 보유 아이템에만 둔다. 즉 "내가 가진 이 옷에 뭘 입을까"라는 맥락에서만 시작한다. 카탈로그는 사라지지 않고, 상대 옷 출처(토글)로 계속 참여한다. 그래서 "가진 옷에 살 옷을 더해 보는" 흐름은 그대로 가능하다.

### 3.2 조합이 정적인 문제 (핵심 버그).

결과 코디가 3개 정도로 고정돼 다양하지 않았다. 원인은 `diversifyRecommendations`였다.

```text
diversifyRecommendations: 한 아이템이 최대 3개 코디까지만 등장하도록 제한.
```

이 함수는 같은 옷이 추천을 독점하지 않게 막는 장치인데, 기준 옷은 정의상 모든 코디에 들어간다. 그래서 기준 옷이 3번 등장한 시점(=코디 3개)에서 한도에 걸려, 4번째 코디부터 전부 잘려 나갔다. 결과가 항상 3개로 고정된 이유다.

수정은 기준 옷을 등장 횟수 제한에서 제외하는 것이다. 기준 옷은 모든 코디에 들어가는 게 정상이므로 세지 않고, 상대 옷(하의·아우터 등)만 최대 2개 코디까지 등장하도록 제한했다. 그러면 같은 하의가 반복되지 않으면서 다양한 조합이 골고루 나온다. 최종 노출은 점수순 상위 24개로 둔다.

데이터도 확인했다. 카탈로그는 상의 582, 하의 75, 아우터 30개라, 상의 하나를 기준으로 잡아도 하의 75벌 × 아우터 조합으로 다양성 재료는 충분하다. 즉 정적함은 데이터 부족이 아니라 이 diversify 버그가 원인이었다.

회귀 테스트로 박제했다. 하의 12개를 주면 결과가 12개 나오는지 확인한다(수정 전이라면 3개로 잘렸다).

## 4. 화면 검증.
